## 2. Overview

This TDD operationalizes the PRD's requirement to close the loop from
autonomous-dev-filed (or human-labeled) issues on **enrolled repos** to
auto-submitted fix requests, without introducing runaway cost, duplicate
work, or auto-merge of self-fixes. The design is:

1. **A new pure module tree at `plugins/autonomous-dev/intake/triggers/self_improve/`** owning three concerns:
   - **Detection** — a data-driven registry of "actionable-issue classes"
     (A1/A2/A3 in PRD §5) with per-class detector + confirmed-real-evidence
     checks. New classes are one row + one detector; the outer loop never
     changes.
   - **Guard chain** — a single ordered pipeline evaluating GD1..GD12 and
     NA1..NA7 (PRD §7); each guard is a pure function of `(env, ownership,
     ledger, issue, now)`; the first to trip short-circuits with a typed
     `SkipReason`.
   - **Ledger** — a small persistent file at
     `~/.autonomous-dev/state/self-improve/ledger.json` (schema v1),
     atomic-write via the same `.tmp + rename` convention used by
     `trigger_store.ts`. Idempotency keys are `(repoId, issueNumber)`;
     the ledger survives restarts and reconciles on tick start.
2. **A submission adapter (`submitFromIssue`)** that composes the
   `request submit` payload per PRD §10 (description + acceptance criteria
   extraction) and routes through the existing `cli_adapter.ts`
   submission path (TC-03) — never re-implements submission. It also
   posts the linking GitHub comment with the
   `<!-- autodev-self-improve: REQ-NNNNNN -->` marker (FR-SUB-06).
3. **A watch-tick integration point.** `runWatchTick` in
   `intake/triggers/watch_tick.ts` gains one optional dependency
   `selfImprove?: SelfImproveDeps`; when injected, the tick invokes
   `scanEnrolledRepos()` **after** the existing completion detection and
   watch advance. When `GD1` (`AUTONOMOUS_DEV_SELF_IMPROVE=1`) is off,
   the tick short-circuits before any GitHub call — bit-for-bit
   backward-compatible (NFR-COMPAT-02).
4. **Human-merge invariant enforcement** in two places: at request
   creation (`source: "self-improve"` + label `autodev:self-fix` on the
   PR) and at the `integration`/`deploy` phase (any auto-merge helper
   MUST hard-check `state.self_improve?.sourceIssue` before proceeding).
5. **CLI surface** on `bin/triggers-cli.ts`: three new verbs under a
   `self-improve` sub-command — `tick` (a peer of `watch-tick`),
   `status`, and `reset` — for operator introspection/recovery.
6. **Events + schemas.** Five new typed event types written to the
   existing audit sink already threaded through `runWatchTick` (TC-05),
   each with a JSON schema entry under `docs/schemas/events/`.

The loop is deliberately layered on top of — never inside — the existing
watch machinery: `runWatchTick`'s current two-phase body is preserved
verbatim; the self-improve scan is a third, opt-in phase invoked after
them, with its own event summary. This ensures the existing 11 jest
tests in `watch_tick.test.ts` continue to pass unchanged.

The loop is **default OFF**. `AUTONOMOUS_DEV_SELF_IMPROVE=1` is the
compound activation gate together with the per-repo enrollment flag
(reusing `isEnrolled()` — no new ownership model).

### 3.3 Data Flow (single tick)

1. `runWatchTick` finishes phases 1+2 unchanged.
2. If `deps.selfImprove` is `undefined`, return (backward-compatible).
3. `selfImprove.scan()` reads env config; if `GD1` off → emit
   `self_improve_disabled` (once), return.
4. Read `ownership` via `readOwnership()` and filter by `isEnrolled`.
5. For each enrolled repo (bounded by GD11 tick cap across all repos):
   a. `listOpenIssues(repoId)` — `gh api /repos/<repoId>/issues?state=open&per_page=100`
      with label filter `autodev:pipeline-failed,autodev:reviewer-finding,autodev/auto-fix`.
   b. For each issue: `classify(issue) → { class: A1|A2|A3|NA?, evidenceInputs }`.
   c. `guardChain.evaluate({ class, issue, ledger, env, now })` in the
      order from §7; first `Trip` short-circuits with a typed
      `self_improve_issue_skipped` event.
   d. If class ∈ {A1,A2,A3} survives: `submitFromIssue(...)` → writes
      ledger entry → emits `self_improve_request_submitted` → posts
      linking comment.
6. Emit `self_improve_tick_summary` with per-guard counts.

### 3.4 Threading, Concurrency, Locking

- The self-improve scan is `async` and runs sequentially per issue to
  bound concurrent `gh` calls (also honoring NFR-PERF-02 streaming
  requirement). Between repos, iteration is sequential; there is no
  worker pool in v1.
- Ledger writes use the same `.tmp.<pid>.<rand> + rename` pattern as
  `trigger_store.ts` (mode `0o600`). Cross-process concurrency
  (NFR-RELIABILITY-02) is defended by:
  1. A per-scan **advisory file lock** on
     `~/.autonomous-dev/state/self-improve/ledger.lock` using `fs.open`
     with `O_CREAT|O_EXCL`. If lock acquisition fails within 500 ms, the
     tick emits `self_improve_error { code: "LOCK_BUSY" }` and returns.
  2. On every write, the writer captures the ledger file's `mtime`
     before load, and on save re-reads and verifies no mtime drift
     since capture; drift → CAS retry (once) → give up + emit error.
- Time reads are always injected via `deps.now(): number` (already
  present in `WatchTickDeps`).

### 4.3 `self_improve/evidence.ts` (confirmed-real-evidence — FR-DET-05, FR-DET-06)

```ts
export interface EvidenceCheck {
  ok: boolean;
  reason?: string;   // human-readable reason string for skipped event
  detail?: unknown;  // optional structured evidence (state, verdict)
}

export interface EvidenceDeps {
  /** Resolve a request state.json for a given repo path + requestId (500 ms budget). */
  readState: (repoPath: string, requestId: string) => Promise<{ status?: unknown } | null>;
  /** Fetch issue events (for A3 human-labeler check). */
  fetchIssueEvents: (repoId: string, issueNumber: number) => Promise<IssueEventsSnapshot>;
  /** Time budget in ms. */
  timeoutMs: number; // default 500
  /** Bot login for "not-bot" check. */
  botLogin: string;
}

export async function checkEvidence(
  klass: ActionableClassId,
  issue: IssueSnapshot,
  ownership: Ownership,
  deps: EvidenceDeps,
): Promise<EvidenceCheck>;
```

- **A1**: parse the fingerprint marker; iterate ownership.repos to find
  the local `repoPath` (fallback: skip with NA7 if repo path unknown);
  attempt to read `<repoPath>/.autonomous-dev/requests/<requestId>/state.json`
  by scanning references in the issue body (`REQ-NNNNNN` regex) — the
  **first matching** request whose `state.status === "failed"` satisfies
  the check. If no request id is referenced (which is atypical — the
  filer includes it in the body), fall back to accepting the marker
  alone (documented behavior: `evidence.detail.mode = "marker-only"`).
- **A2**: parse `<!-- autodev-reviewer: <fp> -->`; verify verdict is
  `REQUEST_CHANGES`; verdicts of `ERROR` map to NA7 (falls under
  REQ-000056 F4).
- **A3**: fetch issue events via
  `gh api /repos/<repoId>/issues/<n>/events`; find the most recent
  `labeled` event with `label.name === LABEL_AUTO_FIX`; the event's
  actor.login must NOT equal `botLogin`.

Timeouts use `Promise.race` with `setTimeout` → 500 ms default, override
via `AUTONOMOUS_DEV_SELF_IMPROVE_EVIDENCE_TIMEOUT_MS`.

### 4.4 `self_improve/guards.ts` (ordered guard pipeline)

```ts
export type GuardId =
  | 'GD1'|'GD2'|'GD3'|'GD4'|'GD5'|'GD6'|'GD7'|'GD8'|'GD9'|'GD10'|'GD11'|'GD12'
  | 'NA1'|'NA2'|'NA3'|'NA4'|'NA5'|'NA6'|'NA7';

export interface GuardTrip {
  guard: GuardId;
  evidence: Record<string, unknown>;
}

export interface GuardCtx {
  env: SelfImproveConfig;
  ownership: Ownership;
  ledger: LedgerReader;
  issue: IssueSnapshot;
  klass: ActionableClassId | null;
  evidence: EvidenceCheck | null;
  now: number;
  fnRegistry: Set<string>;               // GD9 fingerprints
  tickSubmittedSoFar: number;            // for GD11
  concurrencyView: ConcurrencyView;      // active in-flight per repo/global
  costWindow: CostWindowView;            // daily + weekly costs
}

export interface GuardResult { ok: true } | { ok: false; trip: GuardTrip };

/** The ordered guard pipeline. First trip short-circuits. */
export function evaluateGuards(ctx: GuardCtx): GuardResult;
```

**Ordering (PRD §7):**

`GD1 → GD2 → GD10 → GD9 → NA1 → GD4 → GD5 → GD6 → GD7 → NA3 → NA4 → NA2 → GD11 → (evidence-based NA7 for A1/A2)`

Notes:
- **GD1**: `env.enabled === false` → `NA6` synonym (GD1 trip is exactly
  the "loop off" case; see FR-GUARD-02).
- **GD8** (back-off math): consumed indirectly via NA4 — `ledger
  .entries[key].backoffUntil > now` → NA4. Back-off scheduling itself
  is set by `submit.ts` when a failure outcome is observed by the
  reconciler.
- **NA1**: `fnRegistry.has(issue.fingerprint)`.
- **NA2**: `ledger.getInFlightAutoFixRequest(key) !== undefined`.
- **NA3**: `ledger.entries[key].attempts >= env.maxAttemptsPerIssue`.
- **GD12** (human-merge invariant): not a scan-time guard — enforced
  entirely in `merge_gate.ts`.

**Backoff math (FR-GUARD-04):**

```ts
export function computeBackoffUntil(
  attempts: number,
  lastAttemptAt: number,
  baseMinutes: number,
): number {
  const growth = baseMinutes * Math.pow(2, Math.max(0, attempts - 1));
  const capped = Math.min(growth, 24 * 60);
  return lastAttemptAt + capped * 60_000;
}
```

### 4.5 `self_improve/ledger.ts`

```ts
export type LedgerStatus = 'idle' | 'backoff' | 'in_flight' | 'capped';

export interface LedgerEntry {
  repoId: string;
  issueNumber: number;
  issueFingerprint: string | null;
  requestIds: string[];
  attempts: number;
  lastAttemptAt: string;      // ISO-8601
  lastOutcome: 'success' | 'failed' | 'cancelled' | 'unknown';
  backoffUntil: string | null;
  status: LedgerStatus;
}

export interface WindowCost { totalUsd: number; requestCount: number }

export interface LedgerFile {
  version: 1;
  entries: Record<string, LedgerEntry>;      // key: `${repoId}#${issueNumber}`
  windowCosts: Record<string, WindowCost>;   // key: ISO date (YYYY-MM-DD, UTC)
}

export interface LedgerIO {
  homedir(): string;
  readFile(p: string): string | undefined;
  writeFile(p: string, data: string): void;
  now(): number;
}

export function ledgerPath(io: LedgerIO): string;   // ~/.autonomous-dev/state/self-improve/ledger.json
export function loadLedger(io: LedgerIO): LedgerFile;
export function saveLedger(f: LedgerFile, io: LedgerIO): void;    // atomic + mode 0600

export interface LedgerReader {
  getEntry(key: string): LedgerEntry | undefined;
  getInFlightAutoFixRequest(key: string): string | undefined;
  countActiveGlobal(): number;
  countActivePerRepo(repoId: string): number;
  costLast24h(): number;
  costLast7d(): number;
}
export function makeReader(f: LedgerFile, cfg: SelfImproveConfig, now: number): LedgerReader;

export interface LedgerMutator {
  recordSubmission(key: string, entry: LedgerEntry): void;
  recordOutcome(key: string, outcome: 'success'|'failed'|'cancelled', costUsd: number): void;
  reset(key: string): void;
  reconcile(openIssueKeys: Set<string>, closedResolver: (k: string) => void): void;
}
```

**Concurrency (NFR-RELIABILITY-02).**
- The `saveLedger` implementation performs:
  1. Acquire lock file (`ledger.lock`) via `openSync(path, 'wx')` with
     retry loop bounded at 500 ms.
  2. Re-read ledger; compute new state (caller-provided mutator applied
     in-memory on a fresh reader copy).
  3. Write to `ledger.json.tmp.<pid>.<rand>`; `rename`.
  4. Release lock (unlink).
- Read-only accesses (`LedgerReader`) do not lock but pin their loaded
  `LedgerFile` snapshot for the duration of one scan.

**In-flight determination.**
- `getInFlightAutoFixRequest` returns `requestIds[requestIds.length-1]`
  iff `status === 'in_flight'` and (optionally) confirmed via
  cross-check against the intake DB — the intake-DB cross-check is
  injected via `deps.readRequestStatus?: (id) => 'active'|'terminal'|null`;
  when absent, the ledger's own `status` field is authoritative.

### 4.8 `self_improve/submit.ts`

```ts
export interface SubmitDeps {
  /** In-process routing to cli_adapter (TC-03) — see §4.11. */
  requestSubmit: (input: RequestSubmitInput) => Promise<RequestSubmitResult>;
  postGithubComment: (repoId: string, issueNumber: number, body: string) => Promise<void>;
  ledger: LedgerMutator;
  emit: EventEmitter;
  now: () => number;
}

export interface RequestSubmitInput {
  description: string;
  repo: string;                    // target repo (path if enrolled repo has .path; else slug)
  priority: 'high'|'normal'|'low';
  type: 'bug' | 'refactor';
  source: 'self-improve';          // literal
  selfImproveContext: {
    sourceIssue: SourceIssueMeta;  // wired into state.json.self_improve.sourceIssue
  };
}

export async function submitFromIssue(
  issue: IssueSnapshot,
  klass: ActionableClassId,
  cfg: SelfImproveConfig,
  deps: SubmitDeps,
): Promise<SubmitOutcome>;
```

**Sequence (FR-SUB-03 ordering — ledger before comment):**

1. `payload = buildSubmitPayload(issue, klass, cfg)`.
2. `result = await deps.requestSubmit(payload)`. On failure → emit
   `self_improve_error` and return without ledger write.
3. `deps.ledger.recordSubmission(key, { requestIds: [...prev, result.requestId], attempts: prev+1, status: 'in_flight', ... })`
   — atomic write.
4. `deps.postGithubComment(repoId, issueNumber, buildLinkComment(result.requestId, result.prUrl))`.
   Comment body includes marker `<!-- autodev-self-improve: <REQ> -->`.
   Failures here are logged but do NOT roll back the ledger (which is
   the source of truth for idempotency — a duplicate comment is a
   cosmetic issue, not a duplicate request).
5. Emit `self_improve_request_submitted`.

### 4.11 In-process request submission wiring (TC-03)

The PRD requires routing through `cli_adapter.ts::submit`. Two
implementation options:

**Option A (chosen)** — invoke the CLI adapter's `SubmitHandler` router
in-process. `cli_adapter.ts` line 615 delegates to `dispatch('submit',
{...})` which builds a command via `buildCommand` and routes through
`routerProvider().route(cmd)`. We reuse the same
`routerProvider` via a thin wrapper exposed from
`intake/adapters/cli_adapter_entry.ts` (existing) and pass a synthetic
command object. This avoids `child_process` overhead per issue and
keeps errors as typed rejections.

**Option B (rejected)** — shell `autonomous-dev request submit ...`.
Rejected: heavier, less testable, doesn't propagate structured errors,
and creates a new subprocess per issue.

The wrapper (`self_improve/submit.ts`) accepts the routerProvider as an
injected dependency (TC-07 pure-function requirement), so tests pass a
fake router.

## 6. Data Design

### 6.1 Ledger file

Path: `~/.autonomous-dev/state/self-improve/ledger.json` (file mode
`0o600`; parent dir `0o700`).

Schema (version 1) — see `LedgerFile` in §4.5.

Key: `${repoId}#${issueNumber}` (e.g., `owner/name#42`).
`repoId` is validated by the same `SAFE_REPO_SLUG` regex used in
`issue_filer.ts` (`/^[A-Za-z0-9_.-]+\/[A-Za-z0-9_.-]+$/`). `issueNumber`
must be a positive integer.

**Migration.** No migration required — this is a NEW file. First write
creates the file lazily (mkdir -p; mode 0600). Version bumps are
guarded by a `version: 1` field; future migrations will add a loader
switch.

### 6.2 False-negative registry (GD9)

Path: `AUTONOMOUS_DEV_SELF_IMPROVE_FN_REGISTRY_PATH` (default
`~/.autonomous-dev/state/self-improve/false-negatives.json`).

Format (JSON, per OQ-4 recommendation):

```json
{
  "version": 1,
  "entries": [
    { "fingerprint": "abc123def456", "addedBy": "operator@example.com", "reason": "known flaky test", "addedAt": "2026-07-01T00:00:00Z" }
  ]
}
```

Absent file → empty set (FR-CFG-03). Loader is fault-tolerant; malformed
file → empty set + emit `self_improve_config_invalid`.

### 6.3 GitHub state (side-effects on source issue)

- One additive comment per submission (with marker
  `<!-- autodev-self-improve: REQ-NNNNNN -->`).
- Optionally the label `autodev:in-progress` (v1 keeps this OFF by
  default; controlled by env
  `AUTONOMOUS_DEV_SELF_IMPROVE_ADD_INPROGRESS_LABEL=1`).
- NEVER edits title / body; NEVER closes the issue (FR-DET-04, N7,
  S-BOT-1).

## 7. Error Handling

### 7.1 Error taxonomy

| Code | Where | Handling |
|---|---|---|
| `LOCK_BUSY` | ledger lock file busy > 500 ms | Skip tick; emit `self_improve_error`; next tick retries |
| `GH_LIST_FAILED` | `gh api /repos/…/issues` returns non-zero | Skip that repo; emit; continue next repo |
| `GH_EVENTS_FAILED` | issue events lookup fails | Fall through to NA7 for that issue |
| `GH_COMMENT_FAILED` | linking comment fails after successful submit | Log-only; ledger stays valid; emit |
| `EVIDENCE_TIMEOUT` | evidence check > 500 ms | Downgrade to NA7 skip |
| `SUBMIT_FAILED` | `requestSubmit` throws | Do NOT write ledger; emit error; next tick may retry (attempt counter unchanged) |
| `SUBMIT_PARTIAL` | request created but ledger write failed | Reconciler marks the orphan `cancelled` on next tick (FR-SUB-02) |
| `LEDGER_CORRUPT` | JSON parse fails on load | Preserve corrupt file in sidecar (`ledger.json.corrupt-<ts>`); reset in-memory to empty; emit |
| `CONFIG_INVALID` | env var parse fails | Fall back to default; emit `self_improve_config_invalid` |
| `FN_REGISTRY_MISSING` | path set but file absent | Empty set; NOT an error (documented FR-CFG-03) |

### 7.2 Reconciler

On tick start, `scan.ts` runs a reconciliation pass:

1. For each ledger entry with `status === 'in_flight'`, verify the last
   `requestId` still exists in intake DB (via injected
   `readRequestStatus`). If terminal (`done`|`failed`|`cancelled`),
   mark ledger `status` accordingly and update
   `windowCosts[<today>].totalUsd += <cost_accrued_usd>`.
2. For any tracked issue key whose GitHub issue is no longer OPEN
   (learned from the `listOpen` pass — a key with no matching open
   issue), set ledger entry to `status: 'idle'` and note
   `lastOutcome: 'unknown'` if not set. Do NOT decrement `attempts`.
3. For orphaned `SUBMIT_PARTIAL` requests (request exists but not in
   ledger), the code path invokes `requestCancel(id, reason:
   'self-improve-orphan')` — FR-SUB-02.

### 7.3 User-facing error messages

- CLI `self-improve reset`:
  `no ledger entry for owner/name#42; nothing to reset` (exit 1).
- CLI `self-improve status` with corrupt ledger: prints
  `error: ledger unreadable; corrupt file preserved at <path>` (exit 2).
- Submit failure surfaces the underlying router error string; no new
  taxonomy invented.

## 9. Security Considerations

- **NFR-SECURITY-01.** Only OPEN issues on repos returning `true` from
  `isEnrolled()` are read. `gh_issues.listOpen` refuses to run for a
  repo not in the enrollment set (assertion in `scan.ts`).
- **NFR-SECURITY-02.** Ledger file created with `mode 0o600`; parent
  dir with `0o700`. Same for false-negative registry.
- **NFR-SECURITY-03.** No new tokens; `gh` CLI is the only auth path
  (TC-04). `ExecFn` seam is the same used elsewhere.
- **Input sanitation.**
  - `repoId` validated by `SAFE_REPO_SLUG`.
  - `issueNumber` validated as `Number.isInteger(n) && n > 0`.
  - `requestId` (parsed from issue body) validated by
    `/^REQ-\d{6}$/` before being used as a path segment.
  - Issue body is truncated at `bodyTruncateBytes` (default 32 KB) and
    embedded inside a fenced code block in `description` — no
    interpolation, no shell exposure.
- **Comment body composition.** `postGithubComment` shells `gh issue
  comment` with `--body` as an argv element (not through a shell), so
  the marker cannot be injected via issue content.
- **No secrets in events.** Event payloads never include token
  material; the audit sink emits JSON via the existing pipeline.
- **False-positive robustness.** GD9 (`fnRegistry`) + NA7
  (evidence-not-confirmed) are defense-in-depth against auto-fixing
  false-negative failures (Risk in PRD §14).

## 10. Performance Considerations

- **Per-tick budget.** ≤5 s wall (NFR-PERF-01, excluding gh network
  time). Reached by:
  - Bounded page size (100/repo) with a page-2 cap.
  - Sequential per-repo iteration (no worker pool needed at expected
    enrolled-repo counts <10).
  - Evidence timeout of 500 ms per A1/A2 issue.
- **Memory.** `IssueSnapshot` averages ≤2 KB; 200 issues × 10 repos =
  ≤4 MB peak — safely under NFR-PERF-02.
- **Ledger size.** Even at 10 000 entries (one per issue lifetime), the
  file remains ≤5 MB; JSON parse cost is ~50 ms on modern hardware.
  If the ledger grows unbounded in operation, we defer compaction
  (dropping `idle`+`status` entries with `attempts=0`) to a future v2.
- **Cost accounting.** `costLast24h` / `costLast7d` iterate
  `windowCosts` keyed by ISO date; O(≤8 keys) — negligible.
- **Watch-tick regression.** When `AUTONOMOUS_DEV_SELF_IMPROVE=0`,
  `buildDefaultSelfImproveDeps()` returns `null` and `runWatchTick`
  skips the new block entirely — bit-for-bit identical to today.

## 11. Architecture Decision Records

### ADR-001 — Separate ledger file (not `triggers.json`)

- **Context.** REQ-000057 introduces new persistent state per-issue.
- **Options.**
  1. Extend `triggers.json` with a `self_improve` sub-object.
  2. New file `~/.autonomous-dev/state/self-improve/ledger.json`.
- **Decision.** Option 2.
- **Consequences.** Isolates schema evolution blast radius; matches
  PRD §6 ("deliberately separate"); requires new atomic-write helper
  (modeled after `trigger_store.ts`).
- **Traces to.** PRD §6, NFR-COMPAT-01.

### ADR-002 — In-process router wrapping vs `child_process`

- **Context.** How does the loop submit requests?
- **Options.** In-process via `cli_adapter.ts` router (A) vs shell out
  to `bin/autonomous-dev request submit` (B).
- **Decision.** Option A.
- **Consequences.** Faster, structured errors, testable. Requires
  small exposure change (export a `routerProvider` accessor from
  `cli_adapter_entry.ts`). Documented on `TC-03`.
- **Traces to.** TC-03, TC-07, NFR-PERF-01.

### ADR-003 — Data-driven actionable-class catalog

- **Context.** Future actionable classes (A4, A5…) should be low-friction.
- **Options.** Switch statement in `scan.ts` vs data table + detectors.
- **Decision.** Data table (`ACTIONABLE_CATALOG` in `actionable.ts`).
- **Consequences.** New class = 1 catalog row + 1 detector fn + 1 test.
- **Traces to.** NFR-MAINT-01, S-SM-1.

### ADR-004 — Guard chain is ordered pipeline, not policy engine

- **Context.** How are 12 guards + 7 skip classes composed?
- **Options.** Rule engine (Drools-style) vs an explicit ordered
  function that returns on the first trip.
- **Decision.** Ordered function.
- **Consequences.** Trivially testable, no new dependencies, matches
  the PRD's explicit §7 ordering.

### ADR-005 — Sequence: ledger-write before GitHub comment

- **Context.** A crash between submit and side-effect must not leave
  the loop in an inconsistent state.
- **Options.** Comment-first (fewer duplicate submissions, some
  duplicate comments) vs Ledger-first (rare duplicate comments, never
  duplicate submissions).
- **Decision.** Ledger-first. Duplicate submissions are the critical
  failure mode; a duplicate comment is cosmetic.
- **Traces to.** FR-SUB-03.

### ADR-006 — Human-merge invariant is a runtime check, not a label check

- **Context.** Labels can be edited by anyone with write access; the
  invariant that self-improve PRs never auto-merge is safety-critical.
- **Options.** Label-only check (simpler) vs
  `source === "self-improve"` check backed by immutable state.json.
- **Decision.** state-based check via `merge_gate.ts::checkAutoMergeAllowed`.
- **Traces to.** FR-MERGE-01/03.

### ADR-007 — Advisory file lock (not fcntl) for ledger writes

- **Context.** Cross-process concurrency safety without new deps.
- **Options.** `flock`-based, `proper-lockfile` npm dep, or manual
  lock file via `openSync('wx')`.
- **Decision.** Manual `wx` lock file with 500 ms retry.
- **Consequences.** Zero new deps; well-established pattern; unlink
  on release. Stale-lock recovery: if lock file `mtime` > 60 s, treat
  as stale and forcibly unlink.

### B. Relationship to REQ-000056 (self-heal)

- Self-heal (REQ-000056) runs **inside** a pipeline execution to
  auto-remediate transient failures before the request is
  marked `failed`.
- REQ-000057 (this design) runs **after** self-heal has given up and
  a `failed` state.json + a `autodev:pipeline-failed` issue exists.
- The two loops are serially composed; no shared mutable state.
- Both share the "confirmed-real-evidence" principle: neither loop
  should act on known false-negative fingerprints (registry file is
  shared; T1 config supports the same
  `AUTONOMOUS_DEV_SELF_IMPROVE_FN_REGISTRY_PATH`).

### C. Alignment with repo principles

- **"Human is the final escalation, not the first."** The loop
  auto-generates the *fix request*; the human still owns the merge
  (FR-MERGE-01/02/03, ADR-006).
- **"Least-destructive continue rather than hard fail."** Every guard
  prefers skipping over submitting; NFR-RELIABILITY-01 catches all
  errors and downgrades to a logged skip.
- **"Additive, backwards-compatible schema evolution."** New file for
  the ledger; new optional field on `state.json`; new optional dep on
  `WatchTickDeps`; no rename or removal.
- **"Default-off, one-flag opt-in."** `AUTONOMOUS_DEV_SELF_IMPROVE=1`
  + per-repo enrollment is the compound gate; either off → loop is
  inert.

### D. Non-goals (reaffirmed from PRD)

- No auto-merge of self-fix PRs in v1.
- No cross-repo triage.
- No new poller/scheduler process — runs inside `watch-tick`.
- No UI / portal integration.
- No auto-close of source issues.
- No new labels beyond the additive `autodev:in-progress` and PR
  `autodev:self-fix`.
