## 1. Overview

Two co-located defects in `bin/supervisor-loop.sh` jointly prevent the autonomous merge/deploy path from ever firing on this repo. This TDD designs the minimal fix for both, behind feature-flagged config knobs so the behavior is opt-in / overrideable per deployment:

- **#629 ŌĆö merge gate is over-strict (THE key unblock).** `maybe_merge_integration_pr` hard-gates on `mergeStateStatus == CLEAN`. This repo has *permanently-red, pre-existing* CI gates (markdown/lychee, visual-regression #361, scope-enforcement/kind #575) that NEVER go green on any branch, so no PR ever reaches CLEAN and auto-merge never fires. **Fix:** introduce a non-blocking checks allowlist (configurable; default covers the three known-red gates) AND an optional "this check was already failing on the base branch" heuristic. When `mergeStateStatus` is not CLEAN, recompute a *synthetic* mergeability by partitioning `gh pr checks` into (a) required real checks and (b) allowlisted/baseline-red checks. If every check in (a) is success/skipped and every failure in (b) is allowlisted-or-baseline-red, allow the merge. Otherwise preserve today's "skip_not_mergeable" behavior.

- **#628 ŌĆö `pr_comment_reentry` over-counts.** `read_pr_comment_payload` calls `gh api repos/{owner}/{repo}/pulls/comments --paginate`, which returns **every review-thread comment in the entire repository** rather than just this PR. That, combined with the absence of bot/CI/self filtering and a "delta since last-seen" mechanism, is why a PR with 0 issue-comments + 1 line-comment logged "66 new comment(s)" and looped the build back to code 3├Ś. **Fix:** scope the API call to the specific PR number (`/pulls/{n}/comments`), filter out bot/CI/self authors from the actionable set, count only the delta since `addressed_ids`, and *guard re-entry on the actionable-delta count instead of the raw comment list*.

Both fixes are confined to `supervisor-loop.sh`, are testable end-to-end via the existing bats harness (`tests/bats/pr_comment_loop.bats`, `tests/bats/trust_gated_merge.bats`, `tests/bats/merge_gate_order_aware.bats`), and require no schema or wire-protocol changes.

### Key design choices

1. **Config-driven, not policy-baked.** Both knobs live in `config/defaults.json` so operators can extend the allowlist or tune the actionable-comment author filter without code changes.
2. **Conservative defaults.** The non-blocking-checks default covers the *three named* gates from the description (markdown, visual-regression, scope-enforcement). The baseline-red heuristic is opt-in (off by default) because it adds an extra `gh` call per merge tick and depends on a stable base branch.
3. **No relaxation of the real safety gate.** A PR whose `typecheck`/`lint`/`test`/`shell` checks are failing is *still* blocked. We only ignore checks the operator has explicitly told us to ignore.
4. **No new external dependencies.** Both fixes use `gh` + `jq` already on the supervisor's `$PATH`.

### 3.2 #629 ŌĆö synthetic merge-readiness

#### 3.2.1 New helper: `_evaluate_merge_checks`

Add a private function (sibling to `_record_merge_decision`, ~line 3395). Pseudocode:

```bash
# _evaluate_merge_checks(project, pr_url) -> echoes "ok|not_ok|unreadable<TAB>ignored_csv<TAB>blocking_csv"
# Returns "ok" if every failing/pending check is either:
#   (a) listed in MERGE_GATE_NON_BLOCKING_CHECKS, or
#   (b) MERGE_GATE_SKIP_BASELINE_RED=true AND failing on base branch HEAD.
# Returns "unreadable" if gh pr checks did not yield parseable JSON ŌĆö caller
# treats this exactly like today's CLEAN-failure path (skip, leave for human).
_evaluate_merge_checks() {
    local project="$1" pr_url="$2"

    local checks_json
    checks_json=$( (cd "${project}" 2>/dev/null && \
        gh pr checks "${pr_url}" --json name,state,bucket \
            < /dev/null 2>/dev/null) || echo "")
    if [[ -z "${checks_json}" ]] || ! echo "${checks_json}" | jq empty 2>/dev/null; then
        printf 'unreadable\t\t\n'
        return 0
    fi

    # Compute baseline-red set (opt-in). On any failure or when disabled, the
    # set is empty (so it never widens the allowlist by accident).
    local baseline_red='[]'
    if [[ "${MERGE_GATE_SKIP_BASELINE_RED}" == "true" ]]; then
        baseline_red=$(_baseline_red_checks "${project}" "${pr_url}")
        [[ -n "${baseline_red}" ]] || baseline_red='[]'
    fi

    # Partition checks via jq. A check is "failing" if its bucket is "fail" or
    # state Ōłł FAIL/FAILURE/ERROR/CANCELLED/TIMED_OUT. Pending checks (IN_PROGRESS
    # / QUEUED) are treated as blocking ŌĆö we never short-circuit them.
    local result
    result=$(echo "${checks_json}" | jq -c \
        --argjson allow "${MERGE_GATE_NON_BLOCKING_CHECKS}" \
        --argjson baseline "${baseline_red}" \
        '
        def failing: (.bucket // "") == "fail"
                     or ((.state // "") | IN("FAILURE","FAIL","ERROR","CANCELLED","TIMED_OUT"));
        def pending: ((.state // "") | IN("IN_PROGRESS","QUEUED","PENDING"));
        def ignorable($name): ($allow | index($name)) != null
                              or ($baseline | index($name)) != null;
        . as $rows
        | ($rows | map(select(failing) | select(ignorable(.name) | not))) as $hard_failures
        | ($rows | map(select(pending))) as $pendings
        | ($rows | map(select(failing) | select(ignorable(.name))) | map(.name)) as $ignored
        | { ok: (($hard_failures | length) == 0 and ($pendings | length) == 0),
            ignored: $ignored,
            blocking: ($hard_failures | map(.name)) }
        ')

    local ok ignored_csv blocking_csv
    ok=$(echo "${result}" | jq -r '.ok')
    ignored_csv=$(echo "${result}" | jq -r '.ignored | join(",")')
    blocking_csv=$(echo "${result}" | jq -r '.blocking | join(",")')

    if [[ "${ok}" == "true" ]]; then
        printf 'ok\t%s\t\n' "${ignored_csv}"
    else
        printf 'not_ok\t%s\t%s\n' "${ignored_csv}" "${blocking_csv}"
    fi
}
```

#### 3.2.2 Optional baseline-red helper

```bash
# _baseline_red_checks(project, pr_url) -> echoes JSON array of check names
# Best-effort: returns "[]" on any failure. Resolves the base branch from gh,
# fetches that branch HEAD, then reads its commit-status check rollup.
_baseline_red_checks() {
    local project="$1" pr_url="$2"
    local base
    base=$( (cd "${project}" 2>/dev/null && gh pr view "${pr_url}" \
                --json baseRefName --jq '.baseRefName' < /dev/null 2>/dev/null) || echo "")
    [[ -n "${base}" ]] || { echo "[]"; return 0; }

    local base_checks
    base_checks=$( (cd "${project}" 2>/dev/null && \
        gh api "repos/{owner}/{repo}/commits/${base}/check-runs" \
            --jq '[ .check_runs[] | select(.conclusion=="failure") | .name ]' \
            < /dev/null 2>/dev/null) || echo "[]")
    echo "${base_checks}" | jq empty 2>/dev/null || base_checks="[]"
    echo "${base_checks}"
}
```

### 3.3 #628 ŌĆö comment count fix

#### 3.3.1 Root cause

`read_pr_comment_payload` at lines 3760ŌĆō3762:

```bash
api_json=$( (cd "${project}" 2>/dev/null && gh api \
                "repos/{owner}/{repo}/pulls/comments" --paginate \
                < /dev/null 2>/dev/null) || echo "")
```

The endpoint `/pulls/comments` (no PR number) returns **every review-thread comment for every PR in the repo**. On a repo with 60+ existing review comments, the seen-marker file is empty on first run, so all 66 fall into the "new" set and trigger a re-entry. Then because `_pr_comment_mark_addressed` records *every visible id* as addressed, the first re-entry burns through `MAX_PR_COMMENT_REENTRIES` exactly when new repo-wide comments arrive elsewhere.

#### 3.3.2 Fix 1 ŌĆö PR-scope the API call

Replace lines 3760ŌĆō3762:

```bash
# NEW: derive the PR number from the URL, then call the PR-scoped endpoint.
local pr_number
pr_number=$(echo "${pr_url}" | sed -nE 's#.*/pull/([0-9]+).*#\1#p')
if [[ -z "${pr_number}" ]]; then
    api_json="[]"
else
    api_json=$( (cd "${project}" 2>/dev/null && gh api \
                    "repos/{owner}/{repo}/pulls/${pr_number}/comments" --paginate \
                    < /dev/null 2>/dev/null) || echo "")
fi
echo "${api_json}" | jq empty 2>/dev/null || api_json="[]"
```

Rationale: GitHub REST documents both forms; `/pulls/{pull_number}/comments` is the PR-scoped review-thread comments endpoint and returns at most the comments on this PR. This single-line fix removes ~95% of the over-count by itself.

#### 3.3.3 Fix 2 ŌĆö Filter non-actionable authors

Extend `pr_comment_new_ids` (line 3803) to take the author filter into account. Refactor to read the comment objects, not just IDs:

```bash
pr_comment_new_ids() {
    local payload_json="$1" seen_file="$2"
    local seen_ids="[]"
    if [[ -f "${seen_file}" ]]; then
        seen_ids=$(jq -c '.addressed_ids // []' "${seen_file}" 2>/dev/null || echo "[]")
    fi

    # NOTE: PR author login is passed in via env when the caller has it.
    # Empty PR_AUTHOR_LOGIN disables self-filtering (safe default for tests).
    echo "${payload_json}" | jq -r \
        --argjson seen "${seen_ids}" \
        --argjson nonact "${PR_COMMENT_NON_ACTIONABLE_AUTHORS:-[]}" \
        --arg author "${PR_AUTHOR_LOGIN:-}" \
        '
        def is_non_actionable($login):
            ($login == $author)
            or any($nonact[]; (. | ascii_downcase) as $needle
                              | ($login // "" | ascii_downcase) | contains($needle));
        [ (.comments // [])[]
          | select((.id as $i | ($seen | index($i)) | not))
          | select(is_non_actionable(.author) | not)
          | .id
        ] | .[]
        ' 2>/dev/null || true
}
```

## 4. API Specification

### 4.1 New shell function signatures

```bash
# Returns "<verdict>\t<ignored_csv>\t<blocking_csv>" on stdout.
# verdict Ōłł {ok, not_ok, unreadable}
# Always exits 0 (callers branch on verdict).
_evaluate_merge_checks(project: str, pr_url: str) -> str

# Returns JSON array of check names that are failing on the PR's base branch.
# Returns "[]" on any failure. Always exits 0.
_baseline_red_checks(project: str, pr_url: str) -> str
```

### 4.2 Modified function contracts

| Function                          | Change                                                                                  | Backward compatibility |
| --------------------------------- | --------------------------------------------------------------------------------------- | --------------------- |
| `load_config`                     | Sets 3 new globals (`MERGE_GATE_NON_BLOCKING_CHECKS`, `MERGE_GATE_SKIP_BASELINE_RED`, `PR_COMMENT_NON_ACTIONABLE_AUTHORS`) | Adds new outputs only |

## 6. Error Handling

| Failure mode                                       | Behavior                                                                                           |
| -------------------------------------------------- | -------------------------------------------------------------------------------------------------- |
| `gh pr checks` fails / returns non-JSON           | `_evaluate_merge_checks` returns `unreadable` ŌåÆ caller records `skip_status_unreadable`, leaves PR for human (matches today's behavior for unreadable PR status). |
| `gh api .../check-runs` fails (baseline-red lookup) | `_baseline_red_checks` returns `[]` ŌåÆ no widening of allowlist; PR may not merge this tick. Safe. |
| `MERGE_GATE_NON_BLOCKING_CHECKS` is empty AND `MERGE_GATE_SKIP_BASELINE_RED` is false | Synthetic-readiness path is **disabled** entirely ŌåÆ behavior bit-identical to pre-fix. |
| PR url cannot be parsed for `pr_number` in `read_pr_comment_payload` | `api_json="[]"` ŌåÆ only `gh pr view` comments are considered. Logged at info via existing fall-through. |
| `gh pr view --json author` fails                  | `PR_AUTHOR_LOGIN=""` ŌåÆ self-filter disabled; bot/etc filter still applies. Re-entry guard still correct. |
| jq partition produces unexpected output            | `verdict` is empty / not in {ok, not_ok, unreadable} ŌåÆ handled by the `!= "ok"` branch as "not_ok"; PR left for human (fail-closed). |

No new user-facing error messages; all reasons land in `merge_decision.reason` and `events.jsonl`.

## 8. Security Considerations

- **No new privileges.** The new code path uses the same `gh` token already on the daemon's environment and the same `gh pr merge --squash` (no `--admin`, no `--force`) as today.
- **Config-driven allowlist is the only way to relax the gate.** A misconfigured allowlist could in principle allow merges on a PR with a failing genuine check ŌĆö but only if the operator names that check explicitly. This is identical in risk to the operator setting `auto_merge` in GitHub UI; explicitly auditable through `merge_decision.reason`.
- **Audit trail is preserved.** Every synthetic-readiness merge stamps the ignored-check list in `merge_decision.reason`, so the postmortem of any wrongful merge is immediate.
- **Comment-source spoofing.** A malicious bot author cannot exploit the filter to suppress operator feedback ŌĆö the filter is *additive* (operator comments are NOT in the bot list), and the worst case is over-counting (re-entry triggered when it shouldn't be), which the loop bound caps at `MAX_PR_COMMENT_REENTRIES`.
- **Path injection.** PR url is parsed only via `sed -nE 's#.*/pull/([0-9]+).*#\1#p'` which yields digits-only; not shell-interpolated.

## 11. Open Questions

1. **Should `merge_gate_non_blocking_checks` support regex / glob?** The description explicitly names three checks and they may produce variant rows like `scope-enforcement/kind` vs `scope-enforcement`. Current design uses exact-match only and asks the operator to list every variant. *Proposal:* keep exact-match for v1 (simpler, fully auditable); revisit if operators hit friction.

2. **Should `gh pr view` be batched?** ┬¦9 notes we now make `gh pr view` twice (once for `state,mergeable,mergeStateStatus`, once for `author`). We can fold `author` into the first call. *Proposal:* do this as a micro-refactor inside task 3, given the negligible blast radius.

3. **Should `addressed_ids[]` purge bot IDs we never acted on?** Today `_pr_comment_mark_addressed` records every visible ID. Recording bot IDs is harmless (they don't cause re-entry anyway) but the file grows. *Proposal:* leave as-is; file size is bounded by repo PR comment count.

4. **Should we record `merge_decision` for the `ok` synthetic-readiness verdict prior to the merge attempt?** Currently we only stamp the `merged` event after `gh pr merge` returns. This is consistent with existing behavior. *Proposal:* no change.

5. **Telemetry / metric exposure.** Should we add a Prometheus / structured-log counter for synthetic-readiness merges? Not in the description; out of scope.

## 12. ADRs (Architecture Decision Records)

### ADR-001: Configurable allowlist over hard-coded check names

- **Context:** Three named CI gates (`markdown`/`lychee`, `visual-regression`, `scope-enforcement/kind`) are permanently red on this repo. The description suggests an allowlist AND/OR a baseline-red heuristic.
- **Options considered:**
  - (A) Hard-code the three names in `supervisor-loop.sh`.
  - (B) Configurable allowlist in `defaults.json` (chosen).
  - (C) Baseline-red detection only.
- **Decision:** B (primary) + C (opt-in secondary). Hard-coding would not generalize to other repos and would invite drift; baseline-red alone depends on the base branch having check history, which can race during base-branch CI rotation.
- **Consequence:** Operators can extend the list without code changes. Net code addition is small.

### ADR-002: Synthetic readiness only when `mergeable == MERGEABLE`

- **Context:** A `BLOCKED` `mergeStateStatus` can mean "failing checks" OR "branch protection requires reviews". A `CONFLICTING` `mergeable` always means merge conflicts ŌĆö no allowlist should rescue it.
- **Decision:** Synthetic readiness runs only when `mergeable == MERGEABLE` AND `mergeStateStatus != CLEAN`. `mergeable != MERGEABLE` short-circuits to the existing skip path.
- **Consequence:** Drafts (`mergeable=MERGEABLE` is false for drafts on most repos), conflicting PRs, and "behind base" PRs are NOT silently merged. The existing G1 rebase gate still handles the "behind base" recovery.

### ADR-003: PR-scope the comments API instead of post-filtering

- **Context:** `read_pr_comment_payload` could either (a) call the PR-scoped endpoint, or (b) keep calling the repo-wide endpoint and filter by `pull_request_url`.
- **Decision:** Use the PR-scoped endpoint.
- **Consequence:** Fewer paginated calls; smaller blast radius if the post-filter logic ever has a bug. The (a)/(b) distinction also matches GitHub REST best practices.

### ADR-004: Author filter is a list of substrings, case-insensitive

- **Context:** Bot logins vary: `github-actions[bot]`, `dependabot[bot]`, `renovate[bot]`, `Codecov-Bot`, plus repo-specific bots. An exact-match list would be brittle.
- **Decision:** Substring match, case-insensitive, on `author.login`. The default list includes `[bot]` to catch most GitHub apps automatically.
- **Consequence:** A human user whose login literally contains `[bot]` would be filtered. Operators can shorten / customize the list per repo.

## 13. Backward Compatibility & Migration

- **Default config preserves behavior** for the unaffected `#629` case: when an operator has not opted into the allowlist (i.e. `merge_gate_non_blocking_checks=[]`), `maybe_merge_integration_pr` follows the exact pre-fix path. Shipping with a non-empty default list IS a behavior change for *this* repo (and is the entire point). Operators who do NOT want that should set the key to `[]` in their `config.json`.
- **`#628` fix is purely a bug fix** ŌĆö there is no scenario in which the previous over-count was desired. No opt-out needed.
- **`pr-comment-seen.json` files in flight at upgrade time** continue to work: `addressed_ids` semantics are unchanged. Past false-positive `addressed_ids[]` entries from repo-wide bleed are harmless going forward.
- **No state-schema migration. No on-disk format change. No event-shape change.**
