> **Request intent (verbatim from state.json):**
> Fix #617 — verification-gate false-negative. Integration fails passing builds with VERIFICATION_FAILED / `command_not_in_audit_log` when the agent ran the tests but the command wasn't captured in the audit log (a 208/208-passing build was marked failed). Make the presence check robust: accept the produced test-result/coverage artifact as proof of execution, not only an audit-log command match; or fix audit-log capture of the test command. See issue #617.

---

## 1. Overview

The Phase C evidence verifier (`plugins/autonomous-dev/lib/verification/verifier.sh::verify_envelope`) refuses an integration/deploy/test phase when any *substantive* evidence claim (a test/build/lint/state-change command) fails the audit-log presence check. The intent — block fabricated evidence — is correct; the failure mode in #617 is that a *real* test run that genuinely passed 208/208 was refused because its command string was either:

1. **Never captured by the SDK hook.** Some Claude-SDK Bash invocations bypass the PreToolUse hook path used by `hooks/audit-log-writer.sh` (e.g. when a subagent or background-run shells out, or when the hook timed out, or when the hook errored silently per its own "never block the pipeline" contract at L112). The audit log is then *missing* a row the verifier expects to find.
2. **Captured under a different normalised form than the agent reported.** The audit-log row records the literal `tool_input.command` string. The verifier's `audit_log_has_command` (L88–L136 of `audit-log-reader.sh`) normalises both sides and does an atom-subset match, but cosmetic differences not covered by the current normaliser (heredocs, multi-line shell with `\` line continuations, `bash -c '…'` wrappers, env-var interpolation, `time …` prefixes, `cd && cmd` ordering where the audit recorded only `cd`) still produce a miss.

The result is a binary failure: the substantive presence-miss path in `verify_envelope` (L564–L568) sets `verdict="would_have_failed"` and `reason="command_not_in_audit_log"`. Because the role is substantive, the phase is refused regardless of any other signal — including the existence of a JUnit XML, coverage report, or other concrete artifact that demonstrably proves the tests *did* execute.

This TDD specifies a **two-pronged, surgical fix** that closes the false-negative without weakening the anti-fabrication guarantee that drives the entire Phase C design:

1. **Artifact-as-proof fallback (primary fix).** When a substantive claim fails the audit-log presence check, *and* the phase-result envelope's `artifacts[]` (or a small set of well-known on-disk paths) contains a concrete machine-readable test/build product whose **content** demonstrates real execution (a JUnit XML with test counts, a coverage report referencing the project's source files, a Vitest/Jest JSON results file, an `npm test` saved stdout containing a summary line, etc.), classify the claim as `verified_by_artifact` rather than `would_have_failed`. The verdict still records the missing-from-audit-log signal in the per-row report (operators see WHY), but the phase is no longer refused on that signal alone. A *fabricated* command without a corresponding artifact still refuses.
2. **Audit-log capture hardening (secondary fix).** Add a `PostToolUse` companion hook that backfills any Bash command the PreToolUse hook missed (best-effort), and add a one-second-deadline `flock`-wrapped append so a slow filesystem can't cause a silent drop. This narrows the failure surface for cause (1).

The artifact fallback is the load-bearing fix — even if the audit log is perfect, real-world `bash -c`, `Task`-tool, and subagent paths will continue to produce occasional misses, and the artifact gate is the right semantic floor: a real test run leaves real artifacts. The audit-log hardening is incremental risk reduction.

Both changes are additive and gated behind a feature flag (`VERIFICATION_ARTIFACT_FALLBACK=1`, default on) so the existing red-team fixtures keep refusing fabricated builds unchanged.

### 2.2 Data flow — verifier path after change

```
verify_envelope(req_dir, phase, mode)
  │
  ├─ for each evidence claim:
  │     ├─ classify_command  (idempotent | non_idempotent | unclassifiable)
  │     ├─ command_role      (substantive | auxiliary)
  │     ├─ presence_check    (pass | fail | skipped) via audit_log_has_command
  │     ├─ if presence==fail AND role==substantive:        ── EXISTING
  │     │     verdict = would_have_failed
  │     │     reason  = command_not_in_audit_log
  │     │
  │     │     if VERIFICATION_ARTIFACT_FALLBACK==1:        ── NEW
  │     │         proof = verification_artifact_proof_for(req_dir, claim, phase)
  │     │         if proof != "":
  │     │             verdict = verified_by_artifact
  │     │             reason  = "artifact_proof=${proof}"
  │     │             (substantive_failures NOT incremented)
  │     │             (substantive_verified  incremented — see §3.4)
  │     │

  │     └─ write row to verification-report.jsonl
  │

  │     • substantive_failures>0  → should_refuse=1
  │     • auxiliary_failures>0    → check ground truth
  │

```

### 2.3 Data flow — audit-log capture after change

```
Claude SDK Bash tool call
  │
  ├─ PreToolUse  → hooks/audit-log-writer.sh
  │     write row { ts, phase, command, tool_use_id, exit_code:null, source:"sdk_hook_pre" }
  │     wrap append with flock -w 1
  │
  ├─ (tool executes)
  │
  └─ PostToolUse → hooks/audit-log-finalizer.sh           ── NEW
        if a row exists with matching tool_use_id:
            update its exit_code + duration_ms (in-place via tmp+rename)
        else:
            append a new row with source:"sdk_hook_post" (backfill)
```

### 2.4 Integration topology

All changes are inside the autonomous-dev plugin. No new external dependencies. `flock(1)` is part of util-linux on Linux and shipped by Homebrew (`brew install flock`) on macOS; if absent we fall back to the existing best-effort append (matching the existing "audit log failure must not break the pipeline" contract). The portal observes verification outcomes via `verification-report.jsonl` — the new `verdict="verified_by_artifact"` value flows through transparently because the portal renders the verdict field as a string.

## 3. Detailed Design

### 3.1 `verification_artifact_proof_for(req_dir, claim_json, phase)`

Located in `lib/verification/artifact-proof.sh`; sourced by `verifier.sh`.

**Signature.**

```bash
# verification_artifact_proof_for REQ_DIR CLAIM_JSON PHASE -> stdout
#   Echoes a non-empty proof descriptor (e.g. "junit:/abs/path|tests=208|failures=0")
#   when a corresponding test/build product exists AND its contents demonstrate
#   real execution. Echoes empty + returns 1 otherwise.
verification_artifact_proof_for() { ... }
```

**Algorithm.**

1. **Build the candidate path set.** Union of:
   - Paths from `phase-result-${phase}.json .artifacts[].path` whose `kind` matches one of the proof kinds in §3.2.
   - Well-known fixed paths under the project root (resolved as `${req_dir}/../../..`):
     - `**/junit*.xml`, `**/test-results/**/*.xml` (JUnit / Surefire)
     - `**/coverage/lcov.info`, `**/coverage/coverage-final.json`, `**/coverage.xml` (lcov / Istanbul / Cobertura)
     - `**/.vitest-results.json`, `**/jest-results.json`
     - `target/surefire-reports/**`, `target/site/jacoco/**` (Maven/Gradle)
     - `${req_dir}/test-output/*.log` (request-local executor saved tail)
   - Path globs are bounded: max depth 6, max 200 candidates examined per claim. Cost-bounded so a misconfigured `~/.claude/autonomous-dev.json` can't deadlock the verifier.
2. **For each candidate, run the relevant `_check_artifact_*` validator.** Each validator is a pure function: read the file (size-capped at 5 MB), confirm the *content* is consistent with a real test/build product:

| Validator | Confirms |
|---|---|
| `_check_artifact_junit` | XML root is `<testsuite>` or `<testsuites>`; `tests` attribute is a positive integer; the file references real test names (≥1 `<testcase>` element). |
| `_check_artifact_lcov` | Contains `≥1` `SF:` (source file) directive AND `≥1` `DA:` (line hit) record. |
| `_check_artifact_istanbul_json` | Top-level JSON object whose values have a `path` + `statementMap` shape. |
| `_check_artifact_cobertura_xml` | XML root `<coverage>` with `lines-covered` or `line-rate` attribute. |
| `_check_artifact_vitest_json` | Top-level `numTotalTests >= 1`, `numTotalTestSuites >= 1`. |
| `_check_artifact_jest_json` | Same shape as Vitest. |
| `_check_artifact_executor_tail` | A `.log` containing a recognised test-runner summary line (regex set in §3.3). |

If a validator passes, echo a one-line proof descriptor and return 0. If no candidate validates, return 1.

3. **Claim/artifact correlation (anti-fabrication guard).** A proof descriptor is accepted only when the artifact's content references the *same test ecosystem* as the claim's first token. We tokenise the claim and match it to a small ecosystem map:

```
bun, npm, pnpm, yarn, npx, vitest, jest, mocha, cypress, playwright  → junit | vitest | jest | istanbul | lcov | executor_tail
pytest, python                                                       → junit | cobertura | executor_tail
cargo                                                                → junit | cobertura | executor_tail   (cargo-junit, cargo-tarpaulin)
go                                                                   → cobertura | executor_tail          (go test -json transitively)
gradle, mvn                                                          → surefire-junit | jacoco             | executor_tail
make                                                                 → executor_tail
```

A `pytest` claim cannot be rescued by a Vitest JSON; a `cargo test` claim cannot be rescued by a `coverage/lcov.info` produced by a JS run. This rule **closes the obvious bypass**: an agent fabricating a `pytest` evidence row cannot exploit an unrelated `lcov.info` shipped in the repo.

4. **Freshness gate.** The artifact's `mtime` must be `>=` the request's phase-start time (read from `state.json.current_phase_metadata.dispatched_at`, fallback to `state.json.updated_at`, fallback to "always fresh" if neither is parseable). A stale artifact (last modified before the phase started) cannot be a proof — it was not produced by this run. This **closes the second obvious bypass**: a checked-in `coverage/lcov.info` from a previous build cannot rescue a fabricated claim.

### 3.2 `phase-result.json .artifacts[]` proof kinds

The phase-result schema already supports `artifacts: [{kind, path, title}]`. We declare a small set of `kind` strings that the artifact-fallback recognises:

| `kind` | Treated as | Validator |
|---|---|---|
| `test-results-junit` | JUnit XML | `_check_artifact_junit` |
| `test-results-vitest` | Vitest JSON | `_check_artifact_vitest_json` |
| `test-results-jest` | Jest JSON | `_check_artifact_jest_json` |
| `coverage-lcov` | lcov | `_check_artifact_lcov` |
| `coverage-istanbul` | Istanbul JSON | `_check_artifact_istanbul_json` |
| `coverage-cobertura` | Cobertura XML | `_check_artifact_cobertura_xml` |
| `executor-tail` | runner stdout/stderr `.log` | `_check_artifact_executor_tail` |

Any other `kind` is ignored. The fallback also scans the well-known on-disk paths (§3.1.1) so executors that forget to populate `artifacts[]` for canonical paths still benefit.

### 3.3 Executor-tail summary regex set

For `_check_artifact_executor_tail` (a saved test-runner stdout/stderr file), accept a file whose last 200 lines match **any** of:

| Regex | Runner |
|---|---|
| `Tests:[[:space:]]+[0-9]+[[:space:]]+passed` | Jest, Vitest |
| `[0-9]+[[:space:]]+passing\b` | Mocha |
| `[0-9]+[[:space:]]+passed,[[:space:]]+[0-9]+[[:space:]]+failed` | pytest |
| `test result: ok\.[[:space:]]+[0-9]+[[:space:]]+passed` | cargo test |
| `ok[[:space:]]+[0-9]+ - ` AND a `1\.\.\d+` plan line | bats / TAP |
| `^PASS[[:space:]]+.*` (≥3 lines) | Go `go test` |
| `BUILD SUCCESS` | Maven |
| `BUILD SUCCESSFUL in [0-9]+s` | Gradle |
| `[0-9]+[[:space:]]+test(s)?[[:space:]]+complete(d)?` | generic |

The set is intentionally narrow and runner-specific. A README mentioning "tests passed" cannot accidentally validate — the regexes require runner-specific summary syntax. Patterns live in `lib/verification/artifact-proof.sh` as a single bash array so adding a runner is one-line.

### 3.4 Verifier branch — substantive presence-miss

Inside `verify_envelope` (around verifier.sh L564–L568 in the current code), replace:

```bash
else
    presence_check="fail"
    verdict="would_have_failed"
    reason="command_not_in_audit_log"
fi
```

with (pseudocode):

```bash
else
    presence_check="fail"
    # Try the artifact-fallback (NEW). Gated by env flag, default on.
    local proof=""
    if [[ "${VERIFICATION_ARTIFACT_FALLBACK:-1}" == "1" ]]; then
        proof=$(verification_artifact_proof_for "${req_dir}" "${row}" "${phase}" 2>/dev/null || echo "")
    fi
    if [[ -n "${proof}" ]]; then
        verdict="verified_by_artifact"
        reason="artifact_proof=${proof}"
        # Treat as substantive ground-truth: the claim's underlying work is
        # confirmed by a real product. See §3.5 for counter accounting.
    else
        verdict="would_have_failed"
        reason="command_not_in_audit_log"
    fi
fi
```

Then in the post-loop counter accounting:

```bash
if [[ "${verdict}" == "verified" ]]; then
    verified=$((verified + 1))
    ...
elif [[ "${verdict}" == "verified_by_artifact" ]]; then    # NEW branch
    verified=$((verified + 1))
    # Counts as substantive_verified because we confirmed the underlying
    # work via a real product. This also lifts the ground-truth gate for
    # any auxiliary failures on the same phase, matching today's "real
    # work happened" semantics.
    if [[ "${role}" == "substantive" ]]; then
        substantive_verified=$((substantive_verified + 1))
    fi
else
    would_have_failed=$((would_have_failed + 1))
    ...
fi
```

### 3.5 Counter semantics

| Counter | When incremented (after change) | Purpose |
|---|---|---|
| `verified` | Per-row verdict in `{verified, verified_by_artifact}`. | Total non-failures. |
| `substantive_verified` | Substantive row verdict in `{verified, verified_by_artifact}`. | Ground-truth signal for the auxiliary-miss decision. |
| `substantive_failures` | Substantive row verdict `would_have_failed`. **Not** incremented when artifact-proof rescued the row. | Drives `should_refuse=1`. |
| `auxiliary_failures` | Auxiliary row verdict `would_have_failed`. Unchanged. | Drives auxiliary ground-truth check. |

The phase-refusal decision logic (verifier.sh L738–L746) is **unchanged**. Because `substantive_failures` is no longer incremented for artifact-rescued rows, the existing logic correctly does not refuse the phase. The change is upstream of the decision, not in the decision.

### 3.6 Verification-report row schema

Add one optional field to each row written to `verification-report.jsonl`:

```json
{
  "ts": "...",
  "phase": "integration",
  "command": "bun test",
  "classification": "idempotent",
  "role": "substantive",
  "checks": { "presence": "fail", "exit_code": "skipped", "output_tail": "skipped", "re_execution": "skipped" },
  "verdict": "verified_by_artifact",
  "reason": "artifact_proof=junit:/abs/project/junit.xml|tests=208|failures=0",
  "artifact_proof": {
    "kind": "junit",
    "path": "/abs/project/junit.xml",
    "tests": 208,
    "failures": 0,
    "mtime": "2026-06-29T15:30:12Z"
  }
}
```

`artifact_proof` is null/absent on every other verdict. The portal renderer (`autonomous-dev-portal/server/templates/fragments/verification-override.tsx`) already treats unknown fields as opaque metadata; no portal-side change required.

### 3.7 PostToolUse audit-log finaliser

New file `hooks/audit-log-finalizer.sh`. Same env contract as the PreToolUse writer (`AUDIT_LOG_PATH`, `AUDIT_PHASE`). Reads a PostToolUse event from stdin (JSON with at least `tool_name`, `tool_input`, `tool_response`, and `tool_use_id`).

**Algorithm.**

1. Bail silently if `tool_name != "Bash"` or `AUDIT_LOG_PATH` is unset/unwritable.
2. Extract `tool_use_id`, `tool_input.command`, `tool_response.exit_code` (or `.is_error` if SDK shape changes), and a 50-line `output_tail` from `tool_response.stdout`+`tool_response.stderr`.
3. Acquire `flock -w 1 ${AUDIT_LOG_PATH}.lock`. On failure, silently best-effort the rest (matching the PreToolUse contract).
4. Look for an existing row with matching `tool_use_id`:
   - **Found:** rewrite the file with the row's `exit_code`, `duration_ms`, `output_tail`, and `source` (append `+sdk_hook_post`) populated. The rewrite is `jq` filter → `tmp` → `mv`, atomic at the rename.
   - **Not found:** append a fresh row with `source:"sdk_hook_post"`. This is the *backfill* case that catches the PreToolUse miss.
5. Release the lock.

Schema additions: the row gains a `tool_use_id` field (string, optional on legacy rows) and an `output_tail` field (string, optional). `audit_log_has_command` does not need to change — atom-subset matching still works with the post-only rows because the `command` field is populated identically.

### 3.8 PreToolUse hardening

`hooks/audit-log-writer.sh`:

1. Capture `.tool_use_id` from the event (if present in the SDK schema for this Claude version) and include it in the row.
2. Wrap the final append in `flock -w 1`:

```bash
if command -v flock >/dev/null 2>&1; then
    ( flock -w 1 9 || true; printf '%s\n' "${row}" >&9 ) 9>> "${log_path}" 2>/dev/null || true
else
    printf '%s\n' "${row}" >> "${log_path}" 2>/dev/null || true   # existing behaviour
fi
```

No change to the existing fail-open semantics.

## 4. API Specification

### 4.1 `lib/verification/artifact-proof.sh` (new, public functions)

```bash
# verification_artifact_proof_for REQ_DIR CLAIM_JSON PHASE -> stdout | exit 1
#
# Returns a non-empty descriptor string on stdout when a corresponding
# test/build product validates as real execution proof for the claim;
# returns 1 with empty stdout otherwise.
#
# Descriptor format (pipe-delimited):
#   "<kind>:<abs_path>|<key>=<value>(|...)*"
# Example:
#   "junit:/r/junit.xml|tests=208|failures=0|errors=0"
verification_artifact_proof_for() { ... }

# Candidate scan and individual validators are exported for unit testing
# but considered internal API:
_artifact_candidate_paths()        # REQ_DIR PHASE -> stdout (one path per line)
_artifact_kind_for_path()          # PATH -> stdout (junit|lcov|istanbul|cobertura|vitest|jest|executor_tail|"")
_artifact_ecosystem_for_command()  # COMMAND -> stdout (js|python|rust|go|jvm|generic|"")
_artifact_kind_matches_ecosystem() # KIND ECOSYSTEM -> 0|1
_check_artifact_junit()            # PATH -> 0|1 (stdout: extras as k=v lines)
_check_artifact_lcov()             # PATH -> 0|1
_check_artifact_istanbul_json()    # PATH -> 0|1
_check_artifact_cobertura_xml()    # PATH -> 0|1
_check_artifact_vitest_json()      # PATH -> 0|1
_check_artifact_jest_json()        # PATH -> 0|1
_check_artifact_executor_tail()    # PATH -> 0|1
_artifact_is_fresh()               # PATH REQ_DIR -> 0|1
```

### 4.2 `verifier.sh::verify_envelope` (signature unchanged)

Behavioural change: the per-row verdict gains a fourth value `verified_by_artifact`. Backward-compatible — callers that switch on `verdict` are expected to use a default branch; the only known callers (`spawn-session.sh` line 536+) read `verdict == "would_have_failed"`, which the new value does NOT match — so the substantive presence-miss case correctly does NOT trigger the envelope-overwrite path.

### 4.3 `hooks/audit-log-finalizer.sh` (new)

Same execution contract as `audit-log-writer.sh`. No CLI args; reads JSON event from stdin. Always exits 0.

### 4.4 Environment flags

| Flag | Default | Effect |
|---|---|---|
| `VERIFICATION_ARTIFACT_FALLBACK` | `1` | When `1`, enables the artifact-fallback branch in §3.4. When `0`, behaviour is byte-identical to today's verifier (used to gate the rollout and by red-team regression tests that assert "fabricated claims still refuse with the flag off"). |
| `VERIFICATION_ARTIFACT_MAX_DEPTH` | `6` | Maximum directory depth for the well-known-path scan. |
| `VERIFICATION_ARTIFACT_MAX_FILES` | `200` | Maximum number of candidate paths examined per claim. |
| `VERIFICATION_ARTIFACT_MAX_SIZE_MB` | `5` | Per-file size cap for content validation. Larger files are skipped (return 1). |
| `VERIFICATION_ARTIFACT_FRESHNESS_REQUIRED` | `1` | When `1`, require mtime >= phase-dispatched_at. |

All flags are read by `artifact-proof.sh`; the verifier itself only reads `VERIFICATION_ARTIFACT_FALLBACK`.

### 4.5 Phase-result schema (additive)

`artifacts[].kind` gains the values listed in §3.2 as recognised proof kinds. The schema validator already accepts arbitrary strings for `kind`, so this is purely a documentation change; the new values are recognised by `artifact-proof.sh` but unknown values continue to be ignored without error.

### 4.6 `verification-report.jsonl` row schema (additive)

Adds optional `artifact_proof` object (see §3.6). Readers (portal, operators inspecting the JSONL) must treat unknown fields as opaque.

## 5. Data Design

No new persistent stores. No migrations.

**State-file fields touched:** none. The artifact fallback derives the freshness gate from `state.json.current_phase_metadata.dispatched_at` (already populated by the supervisor for every dispatched phase) — read-only.

**File-system reads added:**

| Path pattern | Purpose | Read cost |
|---|---|---|
| `${req_dir}/phase-result-<phase>.json` | already read by verifier — no extra cost | — |
| `${req_dir}/../../..` glob (depth-bounded) | candidate artifact scan | ≤ 200 stat()s, ≤ 5 MB of file reads per claim |

The well-known-path glob is **bounded by `VERIFICATION_ARTIFACT_MAX_FILES` and `MAX_DEPTH`** (§4.4). On a typical repo the scan completes in <50ms.

**Audit log schema gain:** `tool_use_id` (optional string), `output_tail` (optional string). Legacy rows without these fields continue to match by command via the existing `audit_log_has_command` atom-subset normaliser; no migration.

## 6. Error Handling

| Failure mode | Detection | Recovery |
|---|---|---|
| Project root not resolvable from `${req_dir}` | `cd ../../..` returns non-zero | Skip the well-known-path scan; only `artifacts[]` paths are examined. Logged at DEBUG (already-noisy verifier doesn't gain a WARN here). |
| `find` not available | unlikely on supported platforms | Fallback to globbing a hand-rolled fixed list of paths; degrades gracefully. |
| Artifact file unreadable (perms / lost mount) | Validator returns 1 | Treated as not-a-proof; verifier continues to the next candidate or to the residual `would_have_failed`. |
| Artifact validator hangs (huge file) | Per-file size cap + per-claim wall-clock budget (5s, enforced via `timeout`) | Validator times out → treated as not-a-proof. Logged at WARN once per phase. |
| `jq` parse error inside JSON validator | Validator returns 1 | Treated as not-a-proof. |
| `flock` unavailable | `command -v flock` returns 1 | Fall back to the existing non-locked append. Same risk surface as today. |
| `tool_use_id` absent in SDK event (old SDK) | jq returns empty | PreToolUse row written without it; PostToolUse cannot correlate → backfills as a fresh row. Atom-subset match in `audit_log_has_command` still works, so no false negatives. |
| PostToolUse hook called for a tool that's not Bash | `tool_name != "Bash"` | Exit 0 silently (matches PreToolUse contract). |
| Verifier called with `VERIFICATION_ARTIFACT_FALLBACK=0` and a real-world miss | Flag explicitly off | Behaviour byte-identical to today (refuses). Useful for operators investigating the verifier itself. |

## 8. Security Considerations

- **Fabrication bypass risk.** The artifact-fallback is the entire point of #617, and it must not become a hole. The three orthogonal gates — content validation (§3.2), ecosystem match (§3.1 step 3), and freshness (§3.1 step 4) — *together* are designed so that a fabricator must (a) ship a structurally-correct artifact matching the claim's ecosystem AND (b) update its mtime to be `>=` dispatched_at. A fabricator who controls the executor environment can in principle do both; that's not a new risk surface because *the same fabricator already controls the claimed `output_tail`* in the envelope. The artifact gate is **at least as strict** as the today's re-exec output-tail comparison (which only fires for idempotent commands and which the agent can also game). The marginal attack surface is "the agent can also forge a JUnit XML" — which is true but uninteresting: an attacker who is already running arbitrary commands in the worktree wins regardless.
- **Path traversal.** Candidate paths are constrained to `${req_dir}/../../..` and depth-bounded. The verifier reads only files; no writes. We use `realpath -m` style absolute resolution before opening to avoid symlink-escape weirdness across the project boundary.
- **Resource exhaustion via huge artifact.** Per-file size cap (`VERIFICATION_ARTIFACT_MAX_SIZE_MB=5`) + per-claim wall-clock cap (5s) bound CPU/memory cost.
- **Lock-file leakage.** The `flock` lock file `${AUDIT_LOG_PATH}.lock` is created with the daemon's umask (typically 0077). It's never read; cleanup happens on `req_dir` removal (existing daemon lifecycle).
- **PostToolUse leaks command output.** The new audit-log row optionally stores a 50-line `output_tail` of the executor's stdout/stderr. Stored at 0600 in the daemon-owned `req_dir`. Same trust boundary as today's `phase-result-*.json` envelope (which already stores `output_tail`).

## 9. Performance Considerations

| Component | Added cost per phase | Notes |
|---|---|---|
| Artifact-fallback scan | ≤ 200 `stat()` + ≤ 5 MB of reads | One scan per substantive presence-fail; in the success path (presence==pass) the fallback is never invoked. Typical real-world runs see 0–1 fallback invocations per phase. |
| Validator regex/jq calls | ≤ 50 ms per artifact | Bounded by size cap. |
| PostToolUse hook | ~5 ms per Bash tool call | One extra hook invocation per Bash tool. Worst case: `flock` contention adds up to 1 s when the PreToolUse hasn't yet released — but flock acquisition is per-tool, not per-phase. |
| `flock` on PreToolUse | <1 ms | `-w 1` deadline; if it can't acquire in 1s we fall through to the lockless path. |

No new persistent storage. `verification-report.jsonl` grows by one optional field per row.

## 10. ADRs

### ADR-001 — Add an artifact-as-proof fallback rather than blanket-relax the presence check

- **Context.** #617's symptom is "a real test run was refused." Two design directions: (a) relax the substantive-fail bar (e.g. require *both* presence-miss AND an absent ground-truth before refusing); (b) keep the bar strict but add a positive proof path through artifacts.
- **Decision.** Option (b). Option (a) weakens the anti-fabrication floor that drives PLAN-042 Phase C. Option (b) keeps the floor where it is and adds an *additional* path to "verified" that requires the agent to have produced a real product — which is at least as strong a signal as a presence match (and arguably stronger: a JUnit XML is closer to "tests really ran" than a string-equal command name).
- **Consequence.** The red-team fixtures stay refused unchanged. The legitimate false positives are rescued. The semantic of `verified_by_artifact` is honest about what was checked.

### ADR-002 — Require ecosystem AND freshness gates, not just content validation

- **Context.** A naïve fallback that says "if any JUnit XML exists, accept" lets a checked-in `coverage/lcov.info` rescue a fabricated `pytest` claim.
- **Decision.** Three gates AND-ed: content shape, ecosystem match (command's first token must match the artifact's ecosystem), and freshness (mtime >= dispatched_at). Any one gate failure → no proof.
- **Consequence.** Marginally more code; closes the obvious bypasses; matches the spirit of "artifact must be evidence of *this* run."

### ADR-003 — Add a PostToolUse hook rather than replace the PreToolUse hook

- **Context.** The PreToolUse hook is fail-open and PreToolUse-only. Some Bash invocations miss it (#617's root cause for cause-1). Replacing PreToolUse with PostToolUse would lose the "tool was attempted" signal when the tool itself crashes the SDK process.
- **Decision.** Keep PreToolUse; add PostToolUse as a *companion*. Correlate via `tool_use_id`; backfill when PreToolUse missed. This is the minimum-risk change.
- **Consequence.** Audit log can now contain pre-only, post-only, or merged rows. `audit_log_has_command` doesn't care because it matches by command atoms. The merged-row case incurs one rewrite of the file under `flock`; bounded by typical session command count (<200).

### ADR-004 — Default `VERIFICATION_ARTIFACT_FALLBACK=1`

- **Context.** Roll-out risk: a new branch in the verifier could either silently weaken the anti-fabrication guard or trip a bug that newly refuses passing builds.
- **Options.** (a) default-off, opt-in. (b) default-on, opt-out via env.
- **Decision.** Default-on. The bug being fixed (#617) is causing *current* false negatives that pause real builds. The default-off path leaves operators unaware of the rescue and reintroduces #617 on every clean install. The kill switch (`VERIFICATION_ARTIFACT_FALLBACK=0`) gives operators an immediate rollback if the new branch misbehaves.
- **Consequence.** Red-team fixtures **must** continue to refuse with the default. The §7.5 regression assertions are the contract test for this.

### ADR-005 — Validate artifact content, do not trust artifact existence alone

- **Context.** "If `junit.xml` exists, accept" is the cheapest fallback and the weakest. A zero-byte `junit.xml` or a README named `junit.xml` would rescue any claim.
- **Decision.** Each validator opens the file and asserts a runner-specific structural invariant (XML root, `<testcase>` elements, JSON shape, summary regex). Size cap + wall-clock cap bound cost.
- **Consequence.** Marginal CPU per substantive-fail rescue (~50 ms). Strict enough that a hand-crafted decoy file would require effort comparable to actually running the tests.

## 12. Open Questions

1. **Should `verified_by_artifact` count toward `per-request cost` accounting?** No — verification cost is daemon-side, not API-side. The artifact-fallback adds ≤ 200 ms per substantive fail; not worth metering. Confirm in spec phase.
2. **Should the portal surface the `artifact_proof` field as a distinct UI badge?** Recommend yes (a small "verified via artifact" pill differentiating from full command-match) — but out of scope for this fix. Track as a follow-up against the portal.
3. **Should we also accept artifacts referenced by an **absolute path outside the project root** (e.g. `/tmp/junit.xml`)?** Recommend **no** for v1. Constrains the surface to "artifacts living in the project tree." If a real-world build pipeline writes outside the project we add an explicit operator-tunable allowlist in v2.
4. **Should `_check_artifact_executor_tail` look at the FULL file or just the tail?** Tail (last 200 lines) is sufficient for runner summary lines and bounds the read cost. Spec-phase: confirm 200 is enough across all runner outputs.
5. **Does the SDK PostToolUse event always include `tool_use_id`?** This depends on the Claude SDK version the daemon ships against. Spec phase should confirm by reading the SDK release notes / running a one-shot empirical probe.
