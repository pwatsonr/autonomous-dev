## 2. Overview

### 2.1 Root cause (from the request description)

Integration verification currently confirms the executor ran the claimed test
command by looking it up in `<req_dir>/command-audit.jsonl` — the PreToolUse
hook installed by `bin/spawn-session.sh`. That log is intermittently flaky:
when the test command is not captured, `lib/verification/verifier.sh` emits
`command_not_in_audit_log`, `verify_envelope` marks the row
`would_have_failed`, and `spawn-session.sh` synthesises a `trust=low` phase
result with `failed_checks=exit_code,output_tail,presence,re_execution`. A
passing build gets false-failed.

The intended safety net — F7 self-heal in `bin/lib/self-heal.sh`
(`detect_verification_false_negative` / `remediate_self_verify`) and the
#625 artifact-as-proof fallback in `lib/verification/artifact-proof.sh` —
both look for `<req_dir>/test-results.json` and both require that file to
exist and be fresh (mtime ≥ phase_started_at). **Nothing in the pipeline
produces that file today.** `grep test-results.json` confirms only
`self-heal.sh` reads the path; no writer exists. Result: F7 always emits
`self_verify_skipped_missing_artifact` and #617 recurs.

### 2.2 Design principle

Verification of a claimed test pass must be based on **independent
re-execution**, not on **audit-log observation of the agent**. The audit log
becomes a secondary/observability signal; a missing command entry is no
longer a fatal signal for substantive test commands, provided ground truth
from independent re-execution is present.

### 2.3 High-level approach (three coordinated changes)

1. **Producer (new).** Add a daemon-owned pre-verification step invoked by
   `supervisor-loop.sh::dispatch_phase` (for `phase in {integration}`, and
   `test` where present) that **detects the repo's canonical test command**
   from `package.json.scripts.test` (falling back to a runner-family probe)
   and **re-executes it with a JSON reporter**, writing
   `<req_dir>/test-results.json` (mode 0600, mtime ≥ phase_started_at). This
   is the single canonical writer of the artifact.
2. **Consumer #1 — verifier.** In `lib/verification/verifier.sh`, extend the
   presence-miss rescue path (currently only artifact-proof) so that a
   presence-only miss on a **substantive test command** is
   `verified_by_artifact` when `<req_dir>/test-results.json` exists, is
   fresh, and reports 0 failures. Also decouple the `re_execution` check
   from audit-log presence: `reexec_check` must run for classifiable
   idempotent commands regardless of `presence_check` outcome.
3. **Consumer #2 — self-heal.** F7's `detect_verification_false_negative` in
   `bin/lib/self-heal.sh` already reads `test-results.json` from that exact
   path (lines 514–541). With the producer in place F7 will now find the
   artifact and correct false-negatives that slip through the verifier
   (e.g. when the daemon-owned re-execution ran but the verifier gate
   already fired for an out-of-band reason).

### 2.4 Explicit non-goals

- Do not remove the audit-log presence check. It remains as an
  observability + red-team defence signal, unchanged for **non-test**
  substantive commands (git/gh state-changers).
- Do not re-implement or bypass artifact-proof (`artifact-proof.sh`); this
  design *feeds* it a real artifact.
- Do not modify the F7 detector logic — only its inputs are new.

## 4. Detailed Design

### 4.1 New component: `daemon-run-tests` (canonical test re-execution)

**Location.** New file: `plugins/autonomous-dev/lib/verification/daemon-run-tests.sh`.
Sourced by `bin/supervisor-loop.sh` alongside `verifier.sh` and
`artifact-proof.sh`.

**Public interface (bash).**

```bash
# daemon_run_tests <req_dir> <project_root> <phase>
#
# Detect the repo's canonical test command, re-execute it inside
# <project_root>, write a normalized <req_dir>/test-results.json artifact
# with mode 0600.
#
# Returns:
#   0 — artifact written; caller must read .failures to decide pass/fail.
#   1 — no detectable test command; artifact NOT written (skipped_no_runner).
#   2 — detected but execution failed with an *infrastructure* error
#       (bad exit before tests could run, e.g. runner not installed);
#       artifact written with status="infrastructure_error".
#
# Side effects:
#   - writes <req_dir>/test-results.json          (0600, daemon-owned)
#   - writes <req_dir>/test-output/daemon-rerun.log (0600, stdout+stderr tail)
#   - emits events to <req_dir>/events.jsonl:
#       daemon_test_rerun_started
#       daemon_test_rerun_completed
#       daemon_test_rerun_skipped_no_runner
daemon_run_tests() { ... }
```

**Invocation site.** In `bin/supervisor-loop.sh::dispatch_phase`, immediately
**after** the agent session exits (right after the `H4 — Self-heal:
dispatch session_outcome` block near line 1896) and **before**
`verify_envelope` is called for `integration` (and `deploy`/`test` when
they run test suites). Sequencing matters: verifier's artifact-proof rescue
and F7's detector both consume the file, so it must exist by then.

Guard on phase:

```bash
if [[ "${phase}" == "integration" || "${phase}" == "test" ]]; then
    if declare -F daemon_run_tests >/dev/null 2>&1 \
       && [[ "${VERIFICATION_DAEMON_RERUN:-1}" == "1" ]]; then
        daemon_run_tests "${req_dir}" "${project}" "${phase}" \
            >> "${req_dir}/events.log" 2>&1 || true
    fi
fi
```

`VERIFICATION_DAEMON_RERUN=0` is the kill-switch (mirrors
`VERIFICATION_ARTIFACT_FALLBACK`) so operators can disable if the daemon
must not run tests (e.g. for a repo whose test suite has side effects).

#### 4.1.1 Language / runner detection

Pseudocode:

```
detect_runner(project_root):
    if not exists(project_root + "/package.json"):
        return { runner: "none", reason: "no_package_json" }

    test_script := jq -r '.scripts.test // empty' package.json
    if test_script != "":
        # Prefer whatever the repo already declares.
        return { runner: "npm_script", cmd_prefix: "npm test --", raw: test_script }

    # Fallback probe.
    if exists node_modules/.bin/vitest:
        return { runner: "vitest", cmd_prefix: "npx vitest run" }
    if exists node_modules/.bin/jest:
        return { runner: "jest",   cmd_prefix: "npx jest --ci" }

    return { runner: "none", reason: "no_runner_detected" }
```

The detector must be **tolerant**: if `package.json` scripts.test is a
prose stub (e.g. `echo "no tests"`) we still respect it — the description
says "re-run the repo's standard test command." Whether that command
passes is decided at re-execution time.

#### 4.1.2 JSON reporter selection

The reporter selection layer wraps the detected command with the
appropriate JSON reporter flag and target path.

| Detected runner | Wrapped invocation |
|-----------------|--------------------|
| vitest | `<cmd> --reporter=json --outputFile=<req_dir>/test-results.json` |
| jest | `<cmd> --json --outputFile=<req_dir>/test-results.json` |
| npm_script (uses vitest) | `npm test -- --reporter=json --outputFile=<req_dir>/test-results.json` |
| npm_script (uses jest) | `npm test -- --json --outputFile=<req_dir>/test-results.json` |
| npm_script (unknown) | run as-is; capture output and post-process (see §4.1.3) |

For the "unknown npm script" case, wrap with the runner-detection heuristic
(peek `scripts.test` for `vitest`/`jest` substrings, in that order) before
falling back to output post-processing. Peek is a substring check only —
this is a heuristic, not an evaluator.

#### 4.1.3 Post-processing (when no native JSON reporter can be attached)

If the runner cannot be forced to emit JSON we still produce
`test-results.json` in the daemon's canonical schema (see §4.2) by parsing
the tail of stdout with the same regex family the executor-tail check
already recognises (see `artifact-proof.sh:_check_artifact_executor_tail`
lines 534–614; grep for the "Tests: N passed, M failed" pattern used by
bun/vitest and the "Ran N tests" pattern used by jest). Failures counted
this way MUST have `.source = "tail_parsed"` for observability so a
downstream operator can tell native-JSON parse from tail-parse.

#### 4.1.4 Timeouts and safety

- Re-execution respects `VERIFICATION_REEXEC_TIMEOUT_S`
  (default 300s, same as `reexecute_command` in verifier.sh).
- Uses the resolved `timeout` binary via `resolve_timeout_bin`
  (existing in supervisor-loop.sh).
- **Working directory**: `<project_root>` (same reasoning as
  `verifier.sh:611–617` — never `req_dir`, which is not a git repo).
- Env: pass through `PATH`, `HOME`, `CI=1`; scrub `NODE_OPTIONS` warnings.
- On kill/timeout (exit 124): write `status = "timeout"` into the artifact,
  emit `daemon_test_rerun_timeout` event, return 2. Verifier's rescue
  branch treats a timeout artifact as **not proof** (see §4.3.2).

### 4.2 Data design: `<req_dir>/test-results.json` schema

**Canonical schema (v1)** — daemon-owned, mode 0600, single JSON object:

```jsonc
{
  "schema_version": 1,
  "phase": "integration",
  "produced_by": "daemon_run_tests",
  "produced_at": "2026-07-01T21:15:33Z",     // ISO-8601 UTC
  "phase_started_at": "2026-07-01T21:14:10Z",
  "runner": "vitest" | "jest" | "npm_script" | "unknown",
  "detected_command": "npm test --",
  "invoked_command": "npm test -- --reporter=json --outputFile=/…/test-results.json",
  "cwd": "/Users/pwatson/codebase/autonomous-dev",
  "exit_code": 0,
  "duration_ms": 4207,
  "source": "native_json" | "tail_parsed" | "infrastructure_error",
  "tests": 208,           // total collected
  "passed": 208,
  "failures": 0,          // authoritative pass/fail counter (0 = pass)
  "errors": 0,
  "skipped": 0,
  "status": "pass" | "fail" | "timeout" | "infrastructure_error",
  "output_tail_path": ".autonomous-dev/requests/REQ-XXX/test-output/daemon-rerun.log"
}
```

**Compatibility with existing consumers.**

- `bin/lib/self-heal.sh:562–564` reads
  `(.failures // .stats.failures // ([.tests[]? | select(.status=="failed")] | length))`.
  Our top-level `.failures` satisfies the first branch.
- `bin/lib/self-heal.sh:517` reads `.test_results_path // .test_artifact`
  from the phase-result envelope — unchanged, we do not need to write that
  key because F7 falls back to the canonical
  `<project>/.autonomous-dev/requests/<request_id>/test-results.json`
  path (line 527) which is exactly where we write.
- `artifact-proof.sh` already knows how to match vitest/jest artifact
  kinds; we register the daemon artifact via the executor-tail branch
  (kind = `daemon_rerun`, treated like `executor_tail` for phase mapping).

### 4.3 Verifier changes (`lib/verification/verifier.sh`)

#### 4.3.1 Consume the new artifact in the presence-miss rescue path

Around lines 585–593, extend the artifact-proof rescue to check for
`<req_dir>/test-results.json` **first** (before delegating to
`verification_artifact_proof_for`):

```bash
if [[ "${presence_check}" == "fail" && "${role}" == "substantive" ]]; then
    if _verifier_daemon_rerun_confirms "${req_dir}" "${phase_started_at}"; then
        verdict="verified_by_artifact"
        reason="artifact_proof=daemon_rerun:${req_dir}/test-results.json"
    elif declare -F verification_artifact_proof_for >/dev/null 2>&1; then
        _art_proof="$(verification_artifact_proof_for "${req_dir}" "${row}" "${phase}" 2>/dev/null || true)"
        if [[ -n "${_art_proof}" ]]; then
            verdict="verified_by_artifact"
            reason="artifact_proof=${_art_proof}"
        fi
    fi
fi
```

Where `_verifier_daemon_rerun_confirms` is a small helper (co-located in
verifier.sh) that:

1. Confirms `<req_dir>/test-results.json` exists and is mode 0600.
2. Confirms `.status == "pass"` and `.failures == 0`.
3. Confirms `.produced_at >= phase_started_at` (mtime check as a
   fall-back; matches F7's freshness rule).
4. Returns 0 if all three hold; 1 otherwise.

**Rationale for putting the daemon-rerun check first.** It's the
strongest signal (independent, fresh, daemon-owned) and cheapest to
verify. Existing artifact-proof paths (executor-tail, vitest/jest,
JUnit, cobertura, lcov) remain in place unchanged as fallbacks for
executor-produced artifacts.

#### 4.3.2 Verdict rule (per description R5)

The row is `verified_by_artifact` iff (see §4.3.1) the daemon rerun
confirms `.status="pass"` AND `.failures==0`. Any other artifact state
(`timeout`, `infrastructure_error`, `status="fail"`, `.failures>0`) is
treated as **not** proof — the presence miss remains `would_have_failed`
and the phase refuses. This preserves the description's requirement (b):
a genuinely-failing build still fails.

#### 4.3.3 Decouple `re_execution` check from audit-log presence (R6/R7)

Current code (verifier.sh:597–662) gates the `reexec_check` on
`case "${classification}"` alone but the `presence_check=fail` path can
have already set `verdict="would_have_failed"` with
`reason="command_not_in_audit_log"`, and downstream synthesis code
(`spawn-session.sh:560–615`) rolls `presence,exit_code,output_tail,re_execution`
into a single `failed_checks` string when the row hits refuse. Refactor:

1. **Presence is now advisory for the substantive-test class.** In the
   `substantive` + `idempotent test-runner` case (e.g. `bun test`,
   `vitest`, `jest`, `pytest`), a presence miss does NOT itself set
   `verdict=would_have_failed`. Instead we set an advisory
   `presence_check=fail_observability_only` and continue into the
   re-execution branch (or the daemon-rerun rescue).
2. **Re-execution runs regardless.** For `classification=idempotent`,
   `VERIFICATION_REEXEC` continues to drive `reexecute_command`. Its
   outcome (`reexec_check`) is the ground-truth verdict for the row.
3. **Non-test substantive commands are unaffected.** State-changing
   git/gh actions retain the strict audit-log presence check — the
   red-team fixtures f01–f10 (see `tests/red-team/no-faked-evidence/`)
   still refuse. The presence check keeps its role for those.

A small helper `_is_test_runner_command "${cmd}"` (matches against
`bun test`, `npm test`, `pnpm test`, `yarn test`, `vitest`, `jest`,
`pytest`, `python -m pytest`, `cargo test`, `go test`) discriminates. This
helper lives alongside `classify_command` in `verifier.sh` for locality.

**Report row shape.** The JSONL row schema is unchanged. New verdict
value `verified_by_artifact` with `reason` prefix
`artifact_proof=daemon_rerun:` is additive.

### 4.4 Self-heal changes (`bin/lib/self-heal.sh`)

**None functional.** The F7 detector at lines 492–581 already:

- reads the canonical path `<project>/.autonomous-dev/requests/<request_id>/test-results.json`,
- validates mtime against `phase_started_at`,
- counts `.failures`,
- returns 0 (heal) when 0 failures, emitting a corrected phase result.

With the new producer in place F7 will simply *find* the artifact and
succeed; no code change required. **Verification we must do**: F7's
detector's `phase_started_at` parsing (lines 546–548) uses `date -j` /
`date -d`. Our producer writes ISO-8601 UTC (`Z`) into
`.produced_at`; F7 does not read that field — it uses the file mtime and
`ctx_json.phase_started_at`. Confirmed compatible.

**Small hardening (optional, not required by description).** Add a debug
log line when F7 emits `self_verify_skipped_missing_artifact` including
the resolved artifact path, so a future producer regression is instantly
diagnosable. Nice-to-have; document as an open question in §12.

## 5. API Specification

### 5.1 `daemon_run_tests` (bash function)

Signature and semantics as in §4.1. Additional notes:

- **Idempotent.** Re-invoking for the same `(req_dir, phase)` overwrites
  the artifact. Callers must not depend on preserving a prior artifact.
- **Concurrency.** Not designed for concurrent invocation within the same
  `req_dir`. Supervisor guarantees a single agent per `(request_id, phase)`.
- **Ownership.** Artifact written 0600, owned by the daemon uid.

### 5.2 `_verifier_daemon_rerun_confirms` (bash helper, internal to verifier.sh)

```
_verifier_daemon_rerun_confirms <req_dir> <phase_started_at_iso>
    return 0 iff:
        - <req_dir>/test-results.json exists AND is a regular file
        - jq '.schema_version == 1' matches
        - jq '.status == "pass"' matches
        - jq '.failures == 0' matches
        - file mtime >= parse_iso(phase_started_at)
    return 1 otherwise
```

### 5.3 New environment variables

| Var | Default | Effect |
|-----|---------|--------|
| `VERIFICATION_DAEMON_RERUN` | `1` | Kill-switch: `0` disables the new producer entirely. |
| `VERIFICATION_DAEMON_RERUN_TIMEOUT_S` | `${VERIFICATION_REEXEC_TIMEOUT_S:-300}` | Wall-clock cap on canonical re-execution. |
| `VERIFICATION_DAEMON_RERUN_RUNNER` | `auto` | Override auto-detection (values: `auto` / `vitest` / `jest` / `npm_script`). Used by tests. |

Document these in `plugins/autonomous-dev/docs/operator-notes/verification-artifact-fallback.md` alongside the existing switch.

## 6. Error Handling

| Error | Origin | Handling | User-visible signal |
|-------|--------|----------|---------------------|
| No `package.json` | detector | `return 1`, emit `daemon_test_rerun_skipped_no_runner`, **do not write artifact** | Verifier falls back to existing artifact-proof rescue (executor-tail etc.); no regression vs today |
| Runner not installed | re-execution | Write artifact with `status="infrastructure_error"`, `return 2` | Verifier does NOT treat as proof; phase refuses if presence also missed. |
| Runner exits nonzero with 0 failures reported (odd) | re-execution | Trust the reporter: `status="pass"`, `failures=0`, `exit_code=<nonzero>` | Verifier treats as proof (rationale: JSON reporter is the authoritative test count; the exit-code discrepancy is logged for observability). |
| Runner times out | re-execution | `status="timeout"`, `return 2`, emit `daemon_test_rerun_timeout` | Verifier does NOT treat as proof. |
| JSON reporter output malformed | post-process | Attempt tail-parse; if that also fails, write `status="infrastructure_error"` | Not proof; phase refuses on presence miss. |
| `test-results.json` cannot be written (perm / disk) | I/O | `return 2`, emit `daemon_test_rerun_write_failed`, log with path | Verifier falls back to existing rescue; F7 emits `self_verify_skipped_missing_artifact` (unchanged behaviour). |

**Error taxonomy stays additive.** No existing verifier reason codes change
value. New reason substrings: `artifact_proof=daemon_rerun:<path>`.

## 7. Security Considerations

- Daemon-owned re-execution runs the repo's `package.json.scripts.test`
  script. **This is the same risk surface the executor already exposes**
  (the agent can run npm scripts today). Guardrails:
  - Only fires when `phase in {integration, test}`.
  - `VERIFICATION_DAEMON_RERUN=0` kill-switch documented.
  - Timeout enforced via `resolve_timeout_bin`.
  - CWD forced to `project_root`; env scrubbed to a minimal allow-list.
- **No shell interpolation** of user-supplied fields. The detected
  command is invoked via an argv array (bash arrays), not `bash -c "$cmd"`,
  so a hostile `scripts.test` string can only invoke the runner it names —
  it cannot escape into surrounding daemon context.
- Artifact written 0600. Consumers (verifier, F7) already assume 0600 for
  daemon-owned artifacts (see TDD-041 §D-06 referenced in verifier.sh).
- **Not a red-team regression.** The red-team fixtures f01–f10 fabricate a
  claim that `bun test` ran when it did not. Our change does *not* weaken
  the fabrication catch: the daemon rerun would either succeed (real
  tests pass — no fabrication) or fail (real tests fail — refuse). A
  fabricator cannot make the daemon's own re-run pass.

## 8. Testing Strategy

### 8.1 New bats regression suite

## 9. Performance Considerations

- **Cost.** One additional test-suite run per integration phase. In this
  repo the vitest run is ~4 s (observed via `executor-tail` fixtures in
  `verifier_617_regression.bats`). Under the 300 s timeout budget this is
  well within envelope.
- **Serialised, not parallel** with the agent's own run: the daemon rerun
  runs after the agent exits (see §4.5), so there is no test interleaving
  or lock contention.
- **Caching.** No caching planned — the whole point is a *fresh* run per
  phase. `phase_started_at` freshness rule enforces this.
- **Disk.** Artifact + tail log are <256 KiB per phase in practice;
  daemon-owned req_dir already lives outside the repo tree.

## 10. Architecture Decision Records

### ADR-058-01: Re-run tests independently rather than fix the audit-log capture

- **Context.** The PreToolUse audit hook in `bin/spawn-session.sh:253–273`
  can miss commands under process-tree race conditions we have not been
  able to eliminate over three release cycles (see #617 discussion).
- **Options.**
  1. Fix the audit-log capture race (root-cause the hook).
  2. Independently re-execute the canonical test command (this proposal).
  3. Trust the executor's phase-result envelope unconditionally.
- **Decision.** Option 2. Traces to R1 (independent of flaky audit log)
  and R2 (daemon must re-run). Option 1 has proven intractable; Option 3
  contradicts the anti-fabrication invariants preserved in the red-team
  suite.
- **Consequences.** +1 test suite run per integration phase; new
  producer/consumer coupling documented in §4.

### ADR-058-02: Feed the existing consumers (verifier artifact-proof + F7) rather than build a parallel gate

- **Context.** Both `verifier.sh` and `bin/lib/self-heal.sh` already
  contain the *reader* logic and both look at
  `<req_dir>/test-results.json`.
- **Options.** (a) Add a third path that bypasses verifier; (b) feed
  existing readers with a canonical producer.
- **Decision.** (b). Traces to R4 ("matching what F7's detector already
  expects at that exact path"). Minimises new surface.
- **Consequences.** The producer must honour F7's expected schema
  (`.failures` numeric, mtime freshness). Documented in §4.2.

### ADR-058-03: Presence check remains for state-changing substantive commands

- **Context.** The description says "keep the audit-log as a
  secondary/observability signal only," but the red-team fixtures depend
  on presence-check strictness for state-changing git/gh actions.
- **Decision.** Downgrade presence to advisory ONLY for the closed set of
  known test-runner commands (`_is_test_runner_command`). State-changing
  substantive commands retain strict presence semantics.
- **Consequences.** Fabrication of a `git push` claim still refuses (f09
  stays green). The description's "observability only" language is
  honoured for the specific class the description is about (test runs).

### ADR-058-04: JSON reporter wrapping vs stdout parsing

- **Decision.** Prefer native JSON reporters (vitest `--reporter=json`,
  jest `--json`) with a documented tail-parse fallback for unknown npm
  scripts. Native JSON is more robust than regex; tail-parse is retained
  because the description's scope includes "detect from package.json test
  script" (which can be arbitrary).
- **Consequences.** Artifact `.source` field distinguishes the two.

## 12. Open Questions

1. **`scripts.test = "echo skip"` semantics.** With no actual test framework
   invoked, our artifact will report `tests=0, passed=0, failures=0,
   status=pass`. Is that acceptable "vacuous pass" behaviour, or should we
   refuse when `tests==0`? Recommendation: accept vacuous pass (matches
   what the executor would report), but log a
   `daemon_test_rerun_no_tests_collected` event so operators can spot
   accidentally-empty test scripts. **Decision needed from reviewer.**
2. **Pytest / cargo / go test.** The description names vitest and jest.
   Do we in-scope Python and Rust/Go now, or file a follow-up? The
   `_is_test_runner_command` matcher already lists them for R6/R7
   presence downgrade (so their `reexec_check` runs cleanly), but
   `daemon_run_tests` §4.1.1 does not detect them. Recommendation:
   scope this REQ to vitest/jest (per description) and file a follow-up
   for pytest/cargo/go, marked in operator-notes.
3. **Multi-workspace repos.** In a workspace with multiple `package.json`
   files (this repo has `plugins/autonomous-dev/`, `plugins/autonomous-dev-portal/`, root),
   the detector picks the root `package.json`. Confirm this matches the
   executor's behaviour. If not, add a `VERIFICATION_DAEMON_RERUN_CWD`
   env override for operators.
4. **Interaction with git worktrees (multi-track).** verifier.sh
   already handles `${req_dir}/worktree`. Should the daemon rerun mirror
   that logic and prefer the worktree if present? Recommendation: yes —
   copy the `worktree=` resolution block from verifier.sh:605–617 into
   daemon-run-tests.sh detection.
5. **Exit-code / failures discrepancy.** Handled defensively in §6 (trust
   the reporter). If we discover a runner where exit=0 and failures>0
   (unlikely), we may need to invert. Track via events.
