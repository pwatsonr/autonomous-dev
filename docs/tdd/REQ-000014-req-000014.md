## 2. Overview

REQ-000013 introduced `sync_intake_db_row()` and `map_state_status_to_intake()`
at `plugins/autonomous-dev/bin/supervisor-loop.sh:2237–2323` and wired them into
five daemon-driven `state.json` writers (intake→prd, advance, terminal-done,
review-fail reset, retry-exhausted-failed). One residual gap remains: when
`escalate_to_paused()` (lines 1657–1703) writes `status = "paused"` to
`state.json`, it does **not** mirror to `intake.db`. The ledger continues to
report the pre-pause status (typically `active`), recreating the original
drift bug class — `request resume` is rejected, dashboards misreport, and
orphan monitors misclassify the request.

**Technical approach:** This TDD does *not* introduce new infrastructure.
It (a) adds a single, well-placed call to the existing helper
`sync_intake_db_row` inside `escalate_to_paused`, (b) commits a maintainer-
facing audit table that classifies every state.json writer in
`supervisor-loop.sh` as `MIRROR`, `EXEMPT`, or `OUT_OF_SCOPE`, with an
in-source pointer comment above the helper definition, and (c) adds three
new bats tests (pause-path regression, pause-path resilience, non-status
writer negative assertion) and re-asserts the REQ-000013 intake→prd guard.

**Key design decisions (full ADRs in §11):**
- ADR-1: Reuse `sync_intake_db_row` verbatim. No new function, no new SQL
  pattern, no new logging convention. (NG1, FR-D1.)
- ADR-2: Call the helper **after** the `mv "${tmp}" "${state_file}"` in
  `escalate_to_paused`, with the same `$ts` already written to
  `state.json.updated_at`. (FR-A1.)
- ADR-3: Pass `current_phase` (the phase that exhausted retries) — read from
  `state.json` immediately before the write — to the helper, not a sentinel.
  Resolves PRD OQ-1.
- ADR-4: Pause-path ledger write is best-effort and non-fatal; `state.json`
  remains authoritative. (FR-A3.)
- ADR-5: Ship the writer audit as a Markdown table in
  `docs/triage/state-json-writers.md` with a back-pointer comment block above
  `sync_intake_db_row`. (FR-B1, FR-B2.)

No schema change, no new dependency, no new framework. The helper's existing
`paused → paused` mapping (line 2242) already supports the new call site
without any code change to the mapper.

### 3.3 Data flow & ordering (inside `escalate_to_paused`)

For each pause transition the sequence is invariant and post-fix:

1. Compute `ts` (already done at line 1666).
2. Build new state via `jq` to `${state_file}.tmp`, then `mv` (atomic) —
   unchanged at lines 1669–1678.
3. Read `current_phase` from the newly-written `state.json` (single `jq -r`
   call) — **NEW** preparatory line.
4. Append the `retry_exhaustion` event to `events.jsonl` — unchanged at
   lines 1681–1697.
5. **NEW:** call
   `sync_intake_db_row "$request_id" "$current_phase" "paused" "$ts"`.
6. Emit alert + ERROR log — unchanged at lines 1700–1702.

Step 5 occurs *after* the `mv` and *after* `events.jsonl` append so that even
if the ledger write fails, both `state.json` and the events log already
reflect the pause and remain authoritative (NFR-3, ADR-2 / ADR-4). Step 3 is
the lone preparatory addition: `current_phase` is not currently a local in
this function and must be read out before the helper call.

### 3.4 Integration topology

- **Internal:** `escalate_to_paused` ← already called from
  `handle_phase_failure` (no signature change). It now calls
  `sync_intake_db_row` (already a peer helper in the same file). Depends on

- **External:** the `sqlite3` CLI (already required by REQ-000013 callers
  and the orphan path) and the `intake.db` file. No network, no auth, no

## 4. Detailed Design

### 4.1 `escalate_to_paused` — single new helper invocation

Reference: current function at `plugins/autonomous-dev/bin/supervisor-loop.sh:1657–1703`.

Pseudocode (the only delta is the read of `current_phase` and the new
helper call; everything else is unchanged):

```bash
escalate_to_paused() {
    local request_id="$1"
    local project="$2"
    local phase="$3"            # phase that exhausted retries (caller-supplied)
    local retry_count="$4"
    local state_file="${project}/.autonomous-dev/requests/${request_id}/state.json"
    local events_file="${project}/.autonomous-dev/requests/${request_id}/events.jsonl"

    local ts
    ts=$(date -u +"%Y-%m-%dT%H:%M:%SZ")

    # ── unchanged: atomic write of paused state ──
    local tmp="${state_file}.tmp"
    jq --arg ts "${ts}" \
        --arg reason "Retry exhaustion in phase ${phase}" \
        '
        .status = "paused" |
        .current_phase_metadata.paused_reason = $reason |
        .current_phase_metadata.paused_at = $ts |
        .updated_at = $ts
        ' "${state_file}" > "${tmp}"
    mv "${tmp}" "${state_file}"

    # NEW (FR-A1): read current_phase from the authoritative state.json so the
    # ledger mirror reflects the phase that exhausted retries (ADR-3).
    local current_phase
    current_phase=$(jq -r '.current_phase // empty' "${state_file}" 2>/dev/null || echo "")
    # Fallback: if state.json is malformed or empty, use the caller-supplied
    # phase argument. Either yields a non-empty, accurate value.
    if [[ -z "${current_phase}" ]]; then
        current_phase="${phase}"
    fi

    # ── unchanged: events.jsonl append ──
    local event
    event=$(jq -n \
        --arg ts "${ts}" \
        --arg req "${request_id}" \
        --arg phase "${phase}" \
        --argjson retries "${retry_count}" \
        '{ timestamp: $ts, type: "retry_exhaustion", request_id: $req,
           details: { phase: $phase, retry_count: $retries, escalated_to: "paused" } }')
    echo "${event}" >> "${events_file}"

    # NEW (FR-A1, FR-A2, FR-A3): mirror state.json into intake.db.
    # Non-fatal by contract; sync_intake_db_row always returns 0 and logs
    # every failure mode (ERROR/WARN/INFO).
    sync_intake_db_row "${request_id}" "${current_phase}" "paused" "${ts}"

    # ── unchanged: alert + log ──
    emit_alert "retry_exhaustion" "Request ${request_id} paused after ${retry_count} retries in phase '${phase}'"
    log_error "Request ${request_id} escalated to PAUSED (retry exhaustion in ${phase})"
}
```

Design notes:

- **Why read `current_phase` from the file** instead of using the `phase`
  argument directly: in the rare case where the caller passes a phase that
  has already mutated (e.g. a follow-up review-fail reset), the file value
  is the *current* truth (TC-4). The argument is kept as a graceful
  fallback so the helper is never called with an empty phase. Empirically
  the two values are identical at every current call site.
- **No change to the `jq` filter** — pausing already writes `.status =
  "paused"` and `.updated_at = $ts`. The helper sends the same `$ts` to
  the ledger so `state.json.updated_at == intake.db.updated_at` after the
  call (FR-A1, parity with REQ-000013 call sites).
- **No change to `map_state_status_to_intake`** — the existing line
  `paused) echo "paused" ;;` (line 2242) is exactly the mapping we need
  (FR-A2).
- **Resilience** is inherited from the helper itself (FR-A3): missing
  `sqlite3`, missing DB file, zero-row UPDATE, or a non-zero `sqlite3`
  exit all log and return 0. The pause is never aborted by a ledger
  failure (ADR-4, NFR-3).

## 6. Data Design

- **No schema change** (NG1, NFR-4, TC-3). The `requests` table is
  untouched.
- **Columns written** (by the helper): `current_phase` (TEXT, free-form),
  `status` (CHECK-constrained; `paused` already permitted per TC-3),
  `updated_at` (TEXT ISO-8601; equals `state.json.updated_at`).
- **CHECK constraint compatibility:** `paused` is listed in
  `intake/db/schema.sql:23–58` `status CHECK` set
  (`queued|active|paused|cancelled|done|failed`). No constraint violation
  is possible from this change.
- **Selector:** `WHERE request_id = :rid` (PRIMARY KEY → at most one row).
- **No migration, no backfill** (NG5). Historical drifted rows are left to
  the orphan-reconciliation path.
- **Concurrency:** pause is a rare, daemon-serialized event per request.
  Reuses the same UPDATE pattern and journaling already used by 5 other
  call sites since REQ-000013.

## 7. Error Handling

The error taxonomy is **identical** to REQ-000013 because the same helper
is used. No new error paths are introduced.

| Failure | Detection | Action | Log level | PRD Req |
|---------|-----------|--------|-----------|---------|
| `sqlite3` not on PATH | `command -v sqlite3` | skip write, pause completes | ERROR | FR-A3 |
| `intake.db` missing | `[[ ! -f $INTAKE_DB ]]` | skip write, pause completes | WARN | FR-A3 |
| Unmapped status (won't occur for `paused`) | `map_*` returns 1 | skip write | ERROR | FR-A3 |
| `sqlite3` non-zero exit | `rc -ne 0` | log error text, pause completes | ERROR | FR-A3 / NFR-6 |
| 0 rows updated | `SELECT changes()` == 0 | log, pause completes | WARN | FR-A3 / NFR-6 |
| Success | `changes() > 0` | confirmation log | INFO | NFR-6 |

Principle (FR-A3, NFR-3, NFR-6, ADR-4): every path through the helper
emits exactly one terminal log line; the pause is never aborted by a
ledger failure. No user-facing error messages change.

## 9. Security Considerations

- **SQL injection:** unchanged. The helper binds via `sqlite3`
  `.parameter set` and double-quote-escapes `request_id`/`phase` (helper
  lines 2289–2304). No new string-interpolated SQL.
- **No new attack surface:** no new file paths beyond
  `docs/triage/state-json-writers.md` (a static Markdown doc), no network,
  no auth. The helper writes a row keyed by an already-validated
  `request_id` (the daemon validated it at intake).
- **Least privilege:** unchanged — helper opens DB for a single bounded
  UPDATE. All other reads remain read-only where REQ-000013 left them.
- **No secrets** are handled or logged; log lines contain only
  `request_id`, `phase`, `status`.

## 10. Performance Considerations

- **Load:** pause is rare (only on retry-budget exhaustion — typically
  ≤ 1 per request lifecycle, often 0). Negligible amortized cost.
- **Budget (NFR-2):** one `sqlite3` invocation per pause, PRIMARY-KEY
  lookup. Target `< 200 ms`; realistically single-digit ms. The audit doc
  has zero runtime cost.
- **No caching, no hot loop.** The helper adds one process spawn per
  pause, dwarfed by the surrounding alert/log overhead.
- **Lock contention:** unchanged from REQ-000013. WAL on `intake.db`
  remains an explicit non-goal here (NG2-adjacent).

## 11. Architecture Decision Records

**ADR-1 — Reuse `sync_intake_db_row` verbatim; no new helper.**
*Context:* PRD NG1 / FR-D1; REQ-000013's helper already handles
`paused → paused`. *Options:* (a) new pause-specific helper; (b) reuse
existing. *Decision:* (b). *Consequences:* one tested code path, no
maintenance fork, no risk of divergent logging/resilience. *Drives:*
FR-A2, FR-D1.

**ADR-2 — Mirror after `mv`, reuse the existing `$ts`.**
*Context:* `state.json` is authoritative; atomic write must remain intact;
FR-A1 requires equal timestamps. *Decision:* call helper post-`mv` with
the same `ts` already written to `state.json.updated_at`. *Consequences:*
`state.json` never left partial; ledger `updated_at` matches; same
ordering as the 5 REQ-000013 call sites — uniform mental model.
*Drives:* FR-A1, NFR-3.

**ADR-3 — Pass `current_phase` (the phase that exhausted retries), not a
sentinel.** *Context:* PRD OQ-1 explicitly defers this to the TDD; the
intake.db `current_phase` column is free-form TEXT (no constraint).
*Options:* (a) sentinel like `paused`; (b) caller-supplied `phase`
argument; (c) read from authoritative state.json post-`mv`. *Decision:*
(c) with (b) as fallback if the read fails. *Consequences:* dashboards
and orphan monitors still see where the request paused; aligns with the
REQ-000013 convention of mirroring exactly what state.json holds.
*Drives:* FR-A1.

**ADR-4 — Pause-path ledger write is best-effort & non-fatal.**
*Context:* `state.json` is authoritative; original bug was a *silent*
miss, not a crash. *Decision:* inherit the helper's "always returns 0"
contract; every failure logs (NFR-6). *Consequences:* pause never blocked
by ledger issues, but failures are always visible.
*Drives:* FR-A3, NFR-3, NFR-6.

**ADR-5 — Ship writer audit as Markdown in `docs/triage/` + in-source
back-pointer.** *Context:* PRD FR-B1 leaves the exact location to "owner-
designated" — `docs/triage/` is the existing home for similar maintainer
notes in this repo. *Decision:* (a) one Markdown file +
