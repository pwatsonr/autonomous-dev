## 2. Overview

The autonomous-dev daemon keeps each request's lifecycle state in two stores:
`state.json` (per-request file, the daemon's authoritative driver) and the
SQLite ledger `~/.autonomous-dev/intake.db` (`requests` table, queried by the
CLI/portal/monitors). The phase-advancement write path in
`plugins/autonomous-dev/bin/supervisor-loop.sh` updates `state.json` but never
issues a matching `UPDATE requests …`, so the two stores drift apart
(reproduced in REQ-000011: ledger stuck at `('intake','queued')` while
`state.json` walked all 14 phases).

**Technical approach:** introduce a single, well-tested helper
`sync_intake_db_row()` that translates a `state.json` transition into a bounded,
parameter-bound `UPDATE` against `${INTAKE_DB}`, reusing the exact `sqlite3` CLI
+ `.parameter set` pattern already proven in `reconcile_orphans()`
(`supervisor-loop.sh:2778`). The helper is then wired into every daemon-driven
transition site (`advance_phase()` and `intake_to_prd_if_needed()`). The helper
maps the `state.json` status vocabulary to the ledger's CHECK-constrained
vocabulary, and is resilient: any failure is logged (never silent — the original
bug class) and never aborts the authoritative `state.json` advance.

**Key design decisions** (detailed as ADRs in §11):
- ADR-1: One shared helper, called at all transition sites (no duplicated SQL).
- ADR-2: Reuse the `sqlite3` CLI + heredoc `.parameter set` pattern, not
  `node`/`better-sqlite3` (per TC-2 in-code rationale).
- ADR-3: Status mapping via a pure bash `case` translator; unmapped → skip+log.
- ADR-4: Ledger write is best-effort and non-fatal; `state.json` stays
  authoritative (G4 / FR-C1).
- ADR-5: Call the helper *after* the `state.json` `mv`, using the same `$ts`.

This design adds no schema change, no new dependency, and no new framework. It
strictly mirrors the daemon's existing writes (TC-6).

### 3.3 Data flow & ordering

For each transition the sequence is invariant:
1. Compute `ts` (already done once per `advance_phase` call, line 2270).
2. Build new state via `jq` to `${state_file}.tmp.$$`, then `mv` (atomic).
3. Append the lifecycle event to `events.jsonl`.
4. **NEW:** call `sync_intake_db_row "$request_id" "$next_phase" "$next_status" "$ts"`.
5. `write_portal_request_action` (already present).

Step 4 occurs *after* the `mv` so that even if the ledger write fails,
`state.json` is already correct and authoritative (NFR-3, ADR-5).

### 3.4 Integration topology

- **Internal:** `sync_intake_db_row` ← called by `advance_phase`,
  `intake_to_prd_if_needed`. Depends on `map_state_status_to_intake`,

- **External:** the `sqlite3` CLI (already a hard transitive dependency) and the
  `intake.db` file. No network, no auth, no third-party service.

## 4. Detailed Design

### 4.1 `map_state_status_to_intake()` — pure status translator

Maps the `state.json` status vocabulary to the ledger's CHECK-constrained
vocabulary (PRD Appendix C). Pure function: stdin/args in, single token out;
no side effects beyond a WARN log on an unmapped value.

```bash
# map_state_status_to_intake(state_status) -> echoes mapped intake status, or ""
#   running -> active | gate -> active | queued -> queued
#   paused -> paused | done -> done | failed -> failed
#   anything else -> echo "" and return 1 (caller must skip the write)
map_state_status_to_intake() {
    local s="$1"
    case "$s" in
        queued)            echo "queued" ;;
        running|gate)      echo "active" ;;   # FR-A2, OQ-1 resolved to active
        paused)            echo "paused" ;;
        done)              echo "done"   ;;
        failed)            echo "failed" ;;
        *)                 echo ""; return 1 ;;
    esac
    return 0
}
```

- **FR-A2 / TC-3:** every emitted value is a member of the CHECK set
  `(queued, active, paused, cancelled, done, failed)` → no constraint violation.
- `gate → active` resolves PRD **OQ-1** in line with the acceptance criterion
  (only `running→active` is pinned; `gate` follows the same "in-flight" semantic).
- `cancelled` is intentionally **not** producible here — cancellation is owned by
  the orphan/cancel paths, not phase advancement (TC-6, NG4).

### 4.2 `sync_intake_db_row()` — ledger mirror helper

```bash
# sync_intake_db_row(request_id, phase, state_status, ts) -> int (always 0)
#   Mirrors a state.json transition into intake.db's requests row.
#   Best-effort & non-fatal: logs on every failure mode, never aborts caller.
sync_intake_db_row() {
    local request_id="$1" phase="$2" state_status="$3" ts="$4"

    # 1) sqlite3 must be available (FR-C2) — mirror orphan-path behavior.
    if ! command -v sqlite3 >/dev/null 2>&1; then
        log_error "sync_intake_db_row: sqlite3 CLI not found; cannot mirror ${request_id} -> ${phase}"
        return 0
    fi

    # 2) DB must exist (parity with reconcile_orphans guard).
    if [[ ! -f "$INTAKE_DB" ]]; then
        log_warn "sync_intake_db_row: ${INTAKE_DB} not present; skipping ledger update for ${request_id}"
        return 0
    fi

    # 3) Map status; skip the write on an unmapped value (FR-A2).
    local mapped
    if ! mapped=$(map_state_status_to_intake "$state_status") || [[ -z "$mapped" ]]; then
        log_error "sync_intake_db_row: unmapped state status '${state_status}' for ${request_id}; not writing ledger"
        return 0
    fi

    # 4) Parameter-bound UPDATE (FR-A1/A3/A4). Quote single-quotes defensively.
    local rid="${request_id//\'/\'\'}"
    local update_out rc
    update_out=$(sqlite3 "$INTAKE_DB" <<SQL 2>&1
.parameter set :rid '${rid}'
.parameter set :phase '${phase//\'/\'\'}'
.parameter set :status '${mapped}'
.parameter set :ts '${ts}'
UPDATE requests
   SET current_phase = :phase,
       status        = :status,
       updated_at    = :ts
 WHERE request_id = :rid;
SELECT changes();
SQL
)
    rc=$?

    # 5) Resilience: non-zero exit -> ERROR, do NOT abort (FR-C1).
    if [[ $rc -ne 0 ]]; then
        log_error "sync_intake_db_row: sqlite3 UPDATE failed for ${request_id} -> (${phase}, ${mapped}); rc=${rc}; out=${update_out}"
        return 0
    fi

    # 6) Zero-row match -> WARN (missing ledger row / silent no-op) (FR-C3).
    local changed
    changed=$(printf '%s\n' "$update_out" | tail -n1)
    if [[ "$changed" == "0" ]]; then
        log_warn "sync_intake_db_row: 0 rows updated for ${request_id} (no ledger row?) -> (${phase}, ${mapped})"
        return 0
    fi

    log_info "sync_intake_db_row: ledger updated ${request_id} -> (${phase}, ${mapped})"
    return 0
}
```

Design notes:
- **`SELECT changes();`** appended to the same heredoc lets us detect the
  zero-row case (FR-C3 / US-6) without a second connection.
- The function **always returns 0** so callers never branch on it; this enforces
  "ledger write never aborts the advance" (FR-C1, ADR-4) at the type level.
- Single-quote escaping matches the orphan path (`supervisor-loop.sh:2779`,
  `:2817`) — defense-in-depth even though request IDs are alphanumeric.
- Capturing `2>&1` into `update_out` ensures the actual sqlite error text reaches
  the log (NFR-6, SM-6).

### 4.3 Wiring into `advance_phase()` (FR-B1, FR-B3)

`advance_phase` already computes a single `ts` at line 2270 — reuse it (FR-A4).
Insert helper calls immediately after the relevant `mv "$tmp" "$state_file"`:

| Branch | Location | New status arg | Resulting ledger row |
|--------|----------|----------------|----------------------|
| Terminal / done | after `mv` at ~2294 | `done` | `(<current_phase>, done)` |
| Advance, gate | after `mv` at ~2347 (gate branch) | `gate` | `(<next_phase>, active)` |
| Advance, running | after `mv` at ~2347 (running branch) | `running` | `(<next_phase>, active)` |
| Review-fail reset | after `mv` at ~2418 | `running` | `(<author_phase>, active)` |
| Failure exhausted | inside `handle_phase_failure` write path | `failed` | `(<current_phase>, failed)` |

Pseudocode (advance-to-next branch):

```bash
mv "$tmp" "$state_file"
# … append phase_advance event …
sync_intake_db_row "$request_id" "$next_phase" "$next_status" "$ts"   # NEW
```

Pseudocode (terminal/done branch): the phase written to the ledger is the phase
that just completed (`current_phase`), since there is no next phase:

```bash
mv "$tmp" "$state_file"
# … append completed event …
sync_intake_db_row "$request_id" "$current_phase" "done" "$ts"        # NEW
```

For the **review-failure reset** branch (lines 2411–2421) the status remains
in-flight, so pass `running` → ledger `active`, phase = `author_phase`.

For the **failure/exhausted** path: `handle_phase_failure()` (the function that
sets `state.json status=failed` on retry exhaustion, ~line 2680–2704) must also
mirror to the ledger. Add `sync_intake_db_row "$request_id" "$current_phase"
"failed" "$ts"` after that write. If `handle_phase_failure` does not currently
take `ts`/phase, pass the values already in scope at the call site (see §10
Open Question OQ-A).

### 4.4 Wiring into `intake_to_prd_if_needed()` (FR-B2)

After the existing `mv "$tmp" "$state_file"` (line 2469) and event append,
insert:

```bash
sync_intake_db_row "$request_id" "prd" "running" "$ts"   # -> ('prd','active')
```

This directly satisfies the headline acceptance criterion (US-1, SM-2): a
request at `(intake, queued)` becomes `('prd','active')` in the ledger.

## 5. API Specification

Two new internal bash functions (no external/public API; no CLI surface change).

### 5.1 `map_state_status_to_intake`

| Aspect | Spec |
|--------|------|
| Args | `$1` = state.json status string |
| stdout | mapped intake status token, or empty string if unmapped |
| Return | `0` on mapped, `1` on unmapped |
| Side effects | none |
| Valid inputs | `queued, running, gate, paused, done, failed` |

Example:
```bash
$ map_state_status_to_intake running   # -> "active", rc=0
$ map_state_status_to_intake bogus      # -> "",      rc=1
```

### 5.2 `sync_intake_db_row`

| Aspect | Spec |
|--------|------|
| Args | `$1`=request_id, `$2`=phase, `$3`=state_status, `$4`=ts (ISO-8601 `Z`) |
| stdout | none meaningful (helper logs via `log_*`) |
| Return | always `0` (non-fatal contract, FR-C1) |
| Preconditions | `${INTAKE_DB}` global set; `log_*` available |
| Failure modes | no sqlite3 (ERROR), no DB file (WARN), unmapped status (ERROR), sqlite error (ERROR), 0 rows (WARN) — all non-fatal |
| Validation | `request_id` single-quote-escaped before binding; values bound via `.parameter set` (TC-5) |

Example:
```bash
$ sync_intake_db_row "REQ-000013" "prd" "running" "2026-05-20T00:00:00Z"
# intake.db row -> current_phase='prd', status='active', updated_at='2026-05-20T00:00:00Z'
```

## 6. Data Design

- **No schema change** (NFR-4, NG3, TC-3/TC-4). The `requests` table
  (`intake/db/schema.sql:23–58`) is untouched.
- **Columns written:** `current_phase` (TEXT, free-form — phase names written
  directly per TC-4), `status` (CHECK-constrained — mapped per §4.1),
  `updated_at` (TEXT ISO-8601; equals the `state.json` `ts`, FR-A4).
- **Selector:** `WHERE request_id = :rid` (PRIMARY KEY → at most one row).
- **No migration, no backfill** (NG1). Historical drift (e.g. REQ-000011) is left
  to the orphan-reconciliation path unless trivial; see OQ-2.
- **Concurrency:** phase boundaries are infrequent and serialized per request by
  the daemon poll loop. The single bounded `UPDATE` reuses SQLite's default
  journaling already used by the orphan writer; no new transaction wrapper is
  introduced (R2 — acceptable given infrequency; see §13 for WAL note).

## 7. Error Handling

| Failure | Detection | Action | Log level | Req |
|---------|-----------|--------|-----------|-----|
| `sqlite3` not on PATH | `command -v sqlite3` | skip write, continue advance | ERROR | FR-C2 |
| `intake.db` missing | `[[ ! -f $INTAKE_DB ]]` | skip write, continue | WARN | (parity) |
| Unmapped status | `map_*` returns 1 / empty | skip write, continue | ERROR | FR-A2 |
| `sqlite3` non-zero exit | `rc -ne 0` | log error text, continue | ERROR | FR-C1 |
| 0 rows updated | `SELECT changes()` == 0 | log, continue (no INSERT) | WARN | FR-C3 |
| Success | `changes() > 0` | confirmation log | INFO | NFR-6 |

Error taxonomy principle (NFR-6 / SM-6): **every** path through
`sync_intake_db_row` emits exactly one terminal log line — there is no silent
return. The advance is never aborted by a ledger failure (ADR-4 / G4).

No user-facing error messages change; failures are operator-facing daemon log
lines keyed by `request_id` and target `(phase, status)`.

## 9. Security Considerations

- **SQL injection:** all dynamic values bound via `sqlite3` `.parameter set`
  (TC-5), plus single-quote escaping of `request_id`/`phase` — same hardening as
  the orphan path (`supervisor-loop.sh:2779`). No string-interpolated SQL values.
- **No new attack surface:** no network, no auth, no new file paths; the helper
  only writes a row keyed by an already-validated `request_id`
  (`validate_request_id` runs at the `advance_phase` entry, line 2236).
- **Least privilege:** writer opens the DB read-write only for the bounded
  UPDATE; read-only queries elsewhere keep `-readonly`.
- **No secrets** handled or logged; log lines contain only request_id/phase/status.

## 10. Performance Considerations

- **Load:** phase boundaries are infrequent (≤ a few per request lifecycle).
- **Budget (NFR-2):** one bounded `sqlite3` invocation per transition,
  PRIMARY-KEY lookup, target < 200 ms; realistically single-digit ms.
- **No caching needed**; no hot loop. The helper adds one process spawn per
  boundary — negligible vs. the agent-session cost dominating each phase.
- **Lock contention (R2):** concurrent intake writers are rare; the UPDATE is
  short. If contention is observed, enabling WAL on `intake.db` is a low-risk
  follow-up (§13), out of scope here.

## 11. Architecture Decision Records

**ADR-1 — Single shared helper at all transition sites.**
*Context:* PRD root cause = no ledger write; orphan path is the only writer.
*Options:* (a) inline UPDATE at each site; (b) one shared helper.
*Decision:* (b) `sync_intake_db_row`. *Consequences:* one place to test/fix; no
duplicated SQL drift. *Drives:* FR-A1, G-(maintainers).

**ADR-2 — `sqlite3` CLI + `.parameter set`, not `better-sqlite3`.**
*Context:* TC-2 / in-code note (lines 2718–2724): native binding broke under
newer Node. *Decision:* reuse the orphan path's CLI + heredoc pattern.
*Consequences:* no native-binding runtime failures; consistent ops story.
*Drives:* FR-A3, TC-2, TC-5.

**ADR-3 — Pure `case`-based status translator.**
*Context:* divergent vocabularies (TC-3). *Decision:* dedicated
`map_state_status_to_intake`; unmapped → skip + log. *Consequences:* no CHECK
violations; testable in isolation. *Drives:* FR-A2, FR-D3.

**ADR-4 — Ledger write is best-effort & non-fatal.**
*Context:* `state.json` is authoritative; original bug was a *silent* miss, not a
crash. *Decision:* helper always returns 0; every failure logs.
*Consequences:* advances never blocked by ledger issues, but failures are always
visible. *Drives:* G4, FR-C1/C2/C3, NFR-3/NFR-6.

**ADR-5 — Mirror after the `state.json` `mv`, reuse the same `ts`.**
*Context:* atomic write-then-mv must remain intact; FR-A4 requires equal
timestamps. *Decision:* call helper post-`mv` with the existing `$ts`.
*Consequences:* state.json never left partial; ledger `updated_at` matches.
*Drives:* FR-A4, NFR-3.

**ADR-6 — `gate` maps to `active` (resolves OQ-1).**
*Context:* acceptance only pins `running→active`. *Decision:* treat `gate` as an
in-flight status → `active`. *Consequences:* consistent with "request is being
worked"; revisit only if DB owners want a distinct ledger value.
