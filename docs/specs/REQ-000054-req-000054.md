## 2. API Contracts

### 2.1 `_evaluate_merge_checks` (NEW)

```bash
# _evaluate_merge_checks(project: str, pr_url: str) -> stdout: "<verdict>\t<ignored_csv>\t<blocking_csv>"
#
# Pre:
#   - $MERGE_GATE_NON_BLOCKING_CHECKS is compact-JSON array of strings (may be "[]").
#   - $MERGE_GATE_SKIP_BASELINE_RED is the literal string "true" or anything else (false).
#   - `gh` and `jq` are on $PATH.
#   - $project is an absolute path to a checkout (cd-target for gh).
#   - $pr_url is a non-empty PR URL acceptable to `gh pr checks`.
#
# Post (always exits 0; callers branch on verdict):
#   - Writes exactly one TAB-separated line to stdout.
#   - verdict ∈ {"ok", "not_ok", "unreadable"}.
#   - ignored_csv = comma-separated names of failing checks that were ignored
#     (allowlisted OR baseline-red). Empty string when none.
#   - blocking_csv = comma-separated names of failing/pending checks that block
#     the merge. Empty string when verdict ∈ {"ok", "unreadable"}.
#
# Semantics:
#   - A check is "failing" iff (.bucket // "") == "fail"
#     OR (.state // "") ∈ {"FAILURE","FAIL","ERROR","CANCELLED","TIMED_OUT"}.
#   - A check is "pending" iff (.state // "") ∈ {"IN_PROGRESS","QUEUED","PENDING"}.
#   - A check is "ignorable" iff its `.name` is in $MERGE_GATE_NON_BLOCKING_CHECKS
#     OR (when $MERGE_GATE_SKIP_BASELINE_RED == "true") its `.name` is in the
#     output of _baseline_red_checks.
#   - Verdict "ok" iff zero non-ignorable failures AND zero pendings.
#   - Verdict "not_ok" otherwise (some failure or pending remains).
#   - Verdict "unreadable" iff `gh pr checks ... --json name,state,bucket` returns
#     empty output OR output that does not parse as JSON.
#
# Side effects:
#   - Invokes `gh pr checks "<pr_url>" --json name,state,bucket` inside $project
#     with </dev/null. ONE call per invocation.
#   - When $MERGE_GATE_SKIP_BASELINE_RED == "true", additionally invokes
#     _baseline_red_checks (which makes 2 more gh calls).
#   - No stderr output expected on the happy paths.
```

#### Example invocations

```bash
# Allowlist drops the only failing check -> ok.
MERGE_GATE_NON_BLOCKING_CHECKS='["markdown"]'
MERGE_GATE_SKIP_BASELINE_RED=false
# gh pr checks returns: [{"name":"markdown","state":"FAILURE","bucket":"fail"},
#                        {"name":"test","state":"SUCCESS","bucket":"pass"}]
_evaluate_merge_checks /repo https://github.com/o/r/pull/42
# -> "ok\tmarkdown\t"

# Real check failing -> not_ok.
# gh pr checks returns: [{"name":"typecheck","state":"FAILURE","bucket":"fail"}]
_evaluate_merge_checks /repo https://github.com/o/r/pull/42
# -> "not_ok\t\ttypecheck"

# Empty output -> unreadable.
_evaluate_merge_checks /repo https://github.com/o/r/pull/42
# -> "unreadable\t\t"
```

### 2.2 `_baseline_red_checks` (NEW)

```bash
# _baseline_red_checks(project: str, pr_url: str) -> stdout: <json-array-of-strings>
#
# Pre:
#   - `gh` and `jq` on $PATH; $project is a checkout dir.
#
# Post (always exits 0):
#   - Writes a single line to stdout that parses as a JSON array of strings.
#   - On any failure (gh fails, no base branch, non-JSON, missing field),
#     writes exactly `[]`.
#
# Side effects:
#   - Up to 2 gh calls (`gh pr view --json baseRefName`, then
#     `gh api repos/{owner}/{repo}/commits/<base>/check-runs --jq ...`).
#   - Both run inside $project with </dev/null.
```

#### Example

```bash
# gh pr view returns: {"baseRefName":"main"}
# gh api .../check-runs --jq '[ ... ]' returns: ["lychee","markdown"]
_baseline_red_checks /repo https://github.com/o/r/pull/42
# -> '["lychee","markdown"]'

# gh pr view fails:
_baseline_red_checks /repo https://github.com/o/r/pull/42
# -> '[]'
```

### 2.5 `pr_comment_new_ids` (MODIFIED)

Replace the body with the filtered variant. Same signature `(payload_json, seen_file) -> newline-separated ids on stdout`.

```bash
pr_comment_new_ids() {
    local payload_json="$1" seen_file="$2"
    local seen_ids="[]"
    if [[ -f "${seen_file}" ]]; then
        seen_ids=$(jq -c '.addressed_ids // []' "${seen_file}" 2>/dev/null || echo "[]")
    fi
    echo "${payload_json}" | jq -r \
        --argjson seen "${seen_ids}" \
        --argjson nonact "${PR_COMMENT_NON_ACTIONABLE_AUTHORS:-[]}" \
        --arg author "${PR_AUTHOR_LOGIN:-}" \
        '
        def is_non_actionable($login):
            (($author != "") and ($login == $author))
            or any($nonact[]; . as $n
                              | ($n | ascii_downcase) as $needle
                              | (($login // "") | ascii_downcase) | contains($needle));
        [ (.comments // [])[]
          | select((.id as $i | ($seen | index($i)) | not))
          | select(is_non_actionable(.author) | not)
          | .id
        ] | .[]
        ' 2>/dev/null || true
}
```

Semantics:
- Filtering is applied AFTER the seen-set difference (an id already in `addressed_ids[]` is dropped regardless of author).
- Self-filter (`$login == $author`) is disabled when `PR_AUTHOR_LOGIN` is unset/empty.
- Substring matching is case-insensitive (`ascii_downcase`).
- A `null` author is normalized to `""` and never matches a non-empty needle.

## 6. Implementation Notes

### 6.1 Patterns to follow from existing code

- **Triple-paren `gh` invocation pattern** (consistency with current code):
  ```bash
  result=$( (cd "${project}" 2>/dev/null && gh <subcommand> ... < /dev/null 2>/dev/null) || echo "")
  ```
  Every NEW gh call MUST follow this pattern (subshell `cd`, `</dev/null`, suppress stderr, `|| echo ""` fallback). Examples to mirror: L3475–3476, L3748–3749, L3760–3762.

- **JSON validation pattern** (defensive parse):
  ```bash
  echo "${json}" | jq empty 2>/dev/null || json="[]"
  ```

- **State file mutation pattern** (atomic mv-on-success):
  ```bash
  local tmp="${state_file}.tag.$$"
  if jq ... "${state_file}" > "${tmp}" 2>/dev/null; then
      mv "${tmp}" "${state_file}"
  else
      rm -f "${tmp}" 2>/dev/null || true
  fi
  ```

- **Bats stub pattern**: see `pr_comment_loop.bats` L46–67 for the `gh` stub driven by `GH_*_JSON` env vars. New tests MUST extend this pattern (not invent new harnesses).

### 6.2 Gotchas

- The deep-merge `jq -s '.[0] * .[1]'` at L740 REPLACES arrays on the right side. A user `config.json` setting `merge_gate_non_blocking_checks=[]` therefore disables the synthetic-readiness path entirely — this is the documented "opt-out" mechanism (TDD §13). T19 covers this.
- `jq --argjson` REQUIRES a valid JSON value. Hence `MERGE_GATE_NON_BLOCKING_CHECKS` and `PR_COMMENT_NON_ACTIONABLE_AUTHORS` are loaded with `jq -c` (compact form, valid JSON). The `${VAR:-[]}` default in `pr_comment_new_ids` keeps unit tests passing even when the global is unset.
- `gh pr checks --json bucket` requires `gh` ≥ 2.30. The `bucket` field defensive `// ""` fallback in the jq partition makes the code tolerate missing buckets.
- The `unset PR_AUTHOR_LOGIN` requirement (§2.6) prevents the env var from leaking across requests in the long-running daemon process. Implement via either `trap '... RETURN'` or explicit `unset` on every `return` path.
- `read -r verdict ignored_csv blocking_csv <<< "${eval_out}"` requires `IFS=$'\t'` to be set locally; the assignment line above MUST be `IFS=$'\t' read -r ...` (one-shot IFS), not a separate `IFS=$'\t'` line that would leak.
- The synthetic-readiness branch deliberately does NOT modify the post-rebase strict-CLEAN check at L3563. If post-rebase synthetic-readiness is desired, it occurs on the next merge tick (the function returns; the daemon retries).
