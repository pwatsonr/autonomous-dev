## 2. Problem Statement

Autonomous-dev today files GitHub issues for terminal pipeline failures (via
`issue_filer` in `intake/triggers/watch_tick.ts`) and, on repos that operators
have explicitly `repo enroll`ed for auto-improvement (`isEnrolled()` in
`plugins/autonomous-dev/src/ownership/commands.ts`), those issues sit in the
repo's issue tracker awaiting a human to file a `request submit` that fixes
them. This is the missing loop: the daemon can *see* the failure, but cannot
*act* on it without a human keystroke.

Concretely, in a fully-enrolled repo the following are actionable issues that
should trigger a fix request automatically:

- **Autonomous-dev-filed pipeline failure issues.** Marked with
  `autodev:pipeline-failed` (or a `autodev-failure:<fp>` HTML fingerprint —
  see `issue_filer.ts` line 52). These are already deduplicated per failure
  fingerprint, so exactly-one-request semantics are straightforward.
- **Reviewer-finding issues** filed by reviewers in the chain
  (`review-gate.ts`), typically labeled `autodev:reviewer-finding` or
  filed by the autonomous-dev bot account, describing a concrete blocking
  finding a reviewer surfaced during a review phase.
- **Human-labeled `autodev/auto-fix` issues.** A human maintainer signals
  "please try to auto-fix this" by adding a well-known label to an
  otherwise-human-authored issue. This is an explicit opt-in per issue.

Left unimplemented, the operator has to hand-copy the issue title/body into
`request submit --description ...` for every one of these, which:

1. Defeats the point of an autonomous loop for a repo the operator has
   *already* enrolled.
2. Is error-prone (typos, wrong scope, dropped context).
3. Introduces latency between a real, confirmed failure and its fix
   attempt.

However, this loop is **safety-critical**: an unconstrained implementation
would (a) create runaway loops (a failing fix creates a failure issue that
creates another fix that fails ...), (b) consume unbounded LLM cost,
(c) submit spurious fix requests for *false-negative* issues (see #617,
where the supervisor claimed a failure that never happened), and
(d) potentially auto-merge self-fixes without human review, violating the
"human as final escalation" invariant established in REQ-000056 (#620).

The current code has neither the polling loop (the watch-tick advances
existing watches but does not scan open issues) nor the runaway-guards
(no per-issue attempt cap, no global kill-switch beyond the coarse
enrollment flag, no cost cap). Both are prerequisites to safely closing
this loop.

## 3. Goals and Non-Goals

### Goals (in scope)

1. **G1 — Actionable-issue detection on enrolled repos.** For every repo
   where `isEnrolled(ownership, repoId) === true`, the daemon MUST poll
   the repo's OPEN issues on a fixed cadence and identify a bounded set of
   *actionable* issues per the §5 catalog.
2. **G2 — Idempotent auto-request submission.** For each actionable
   issue, the daemon MUST submit exactly one fix request (`type: bug` or
   `type: refactor`) with the issue title/body wired into the request's
   description and acceptance criteria. Idempotency is per-`(repoId,
   issueNumber)`; re-triggering while a request or PR is still open for
   that issue is forbidden.
3. **G3 — Normal pipeline execution.** Auto-submitted requests MUST run
   through the standard supervisor pipeline (`intake → prd → ... →
   integration → deploy`) with no bypass of any existing gate.
4. **G4 — Mandatory human merge gate.** Auto-submitted requests MUST
   surface a PR for human approval before merge; the daemon MUST NOT
   auto-merge a self-fix under any circumstance in v1. This holds even
   when the pipeline is otherwise configured for auto-merge on other
   request classes.
5. **G5 — Runaway guards.** The daemon MUST enforce, at minimum:
   (a) per-issue attempt cap, (b) max concurrent auto-fix requests
   globally and per repo, (c) cost cap in USD across all auto-fix
   requests within a rolling window, (d) exponential back-off on
   repeated failure for the same issue, (e) a global kill-switch env
   var and a per-repo enrollment flag as compound gates.
6. **G6 — False-failure resistance.** The daemon MUST NOT auto-submit a
   fix request for an issue that fingerprints to a *known false
   negative* (see REQ-000056 §5/F7 and issue #617). Actionability
   requires "confirmed / real finding" evidence; unconfirmed findings
   are skipped and logged.
7. **G7 — Default-off, flag-gated.** The whole loop MUST default OFF in
   v1 and be enabled by a single environment variable
   (`AUTONOMOUS_DEV_SELF_IMPROVE=1`) combined with per-repo
   `participate_in_auto_improvement: true`. Absence of either disables
   the loop bit-for-bit.
8. **G8 — Observability.** Every detection, every skip, every submit,
   every back-off, and every guard trip MUST emit a typed event to the
   existing events pipeline (`events.jsonl` and/or the audit sink used
   by `watch_tick`) so operators can trace loop behavior end-to-end.

### Non-Goals (explicit exclusions)

- **N1.** Auto-merging fix PRs. The human-merge gate (G4) is a hard
  invariant in v1.
- **N2.** Cross-repo triage (e.g., filing an issue in repo A because
  repo B is failing). Scope is intra-repo.
- **N3.** Rewriting the enrollment / ownership model. This PRD consumes
  the existing `mayAutoImproveScope` / `isEnrolled` gates and treats
  them as the source of truth.
- **N4.** Replacing the existing `issue_filer.ts` deduplication scheme;
  we reuse the fingerprint comment format
  `<!-- autodev-failure: <fp> -->`.
- **N5.** Building a new poller / scheduler. The loop runs inside the
  existing `watch-tick` (or a peer function invoked from the same
  tick), so operators do not have to run a second daemon.
- **N6.** Adding a UI. Portal integration is a v2 concern; v1 is CLI +
  events.jsonl + logs.
- **N7.** Auto-closing the source issue when the fix PR merges. In v1
  the daemon posts a comment linking the request; humans close the
  issue.
- **N8.** Producing new labels on the source issue beyond a
  well-defined status comment / label (`autodev:in-progress`,
  `autodev:auto-fix-attempted`) which are additive and non-destructive.

## 5. Actionable-Issue Catalog (Detection Rules)

Each row defines a distinct actionable-issue class the daemon may open a
fix request for. **All three constraints** in a row's *Detection Rule*
column MUST hold; failing any single constraint means the issue is not
actionable (either not our concern, or a *skipped* issue with a logged
reason).

| ID | Class | Detection Rule | Confirmed-Real Evidence | Default Request Type |
|----|---|---|---|---|
| A1 | Autonomous-dev-filed pipeline failure | Issue is OPEN AND author is the autonomous-dev bot (or issue body contains an `<!-- autodev-failure: <fp> -->` marker) AND issue is labeled `autodev:pipeline-failed` | Fingerprint comment exists AND at least one referenced `state.json` for the failing request shows `status: "failed"` (not merely `paused`) | `bug` |
| A2 | Reviewer finding (autonomous-dev-filed) | Issue is OPEN AND labeled `autodev:reviewer-finding` AND body contains a reviewer-finding structured block (`<!-- autodev-reviewer: <reviewer-id> -->`) | Reviewer verdict was `REQUEST_CHANGES` (not `ERROR` — ERROR falls under REQ-000056 F4, not this loop) and referenced test/artifact still exists | `bug` or `refactor` (per finding-type block) |
| A3 | Human-labeled auto-fix candidate | Issue is OPEN AND has label `autodev/auto-fix` (verbatim) AND label was applied by a human GitHub identity (not the bot) | Presence of the human-applied label IS the evidence — the human is signing off on actionability | `bug` (default) or the value of an optional `autodev/type:*` label |

**Non-actionable classes (explicit skip, must log):**

- **NA1 — Known false-negative fingerprint.** Any issue whose failure
  fingerprint (`<!-- autodev-failure: <fp> -->`) matches an entry in
  the false-negative registry (per REQ-000056 §5/F7).
- **NA2 — Already has an active auto-fix.** Any issue for which
  `getInFlightAutoFixRequest(repoId, issueNumber)` returns a request
  whose status is not terminal (`done`, `failed`, `cancelled`).
- **NA3 — Attempt cap exhausted.** Any issue where the
  `(repoId, issueNumber)` attempt counter ≥
  `AUTONOMOUS_DEV_SELF_IMPROVE_MAX_ATTEMPTS_PER_ISSUE` (default 3).
- **NA4 — In back-off window.** Any issue currently within its
  exponential back-off window after a prior failure.
- **NA5 — Rate/cost cap tripped.** Any issue detected while the global
  or per-repo caps are exhausted for the current rolling window.
- **NA6 — Kill-switch off.** `AUTONOMOUS_DEV_SELF_IMPROVE` ≠ `1` OR
  the repo is not enrolled per `isEnrolled()`.
- **NA7 — Unconfirmed / unresolved evidence.** Any A1/A2 issue whose
  confirmed-real-evidence check fails (state.json missing, fingerprint
  unresolvable, reviewer verdict was ERROR not REQUEST_CHANGES).

Every A1–A3 detection AND every NA1–NA7 skip MUST emit a typed event
(`self_improve_issue_detected`, `self_improve_issue_skipped`) with
the class ID and the specific rule row.

## 6. Ledger, Idempotency, and State

The loop introduces a small persistent ledger, stored alongside the
existing trigger store at
`~/.autonomous-dev/state/self-improve/ledger.json`. It is deliberately
separate from `triggers.json` (different concern; avoids blast radius on
schema evolution).

Schema (versioned; v1):

```json
{
  "version": 1,
  "entries": {
    "<repoId>#<issueNumber>": {
      "repoId": "owner/name",
      "issueNumber": 42,
      "issueFingerprint": "<fp or null>",
      "requestIds": ["REQ-000123", "REQ-000155"],
      "attempts": 2,
      "lastAttemptAt": "2026-07-01T12:00:00Z",
      "lastOutcome": "failed",
      "backoffUntil": "2026-07-01T18:00:00Z",
      "status": "backoff | in_flight | idle | capped"
    }
  },
  "windowCosts": {
    "<isoDate>": { "totalUsd": 1.42, "requestCount": 3 }
  }
}
```

**Idempotency invariants:**

- **I1.** Before submitting, the loop MUST check the ledger and MUST
  check any in-flight request (via the intake/lifecycle handoff manager
  or `state.json.status`) referencing the same `(repoId, issueNumber)`.
- **I2.** The ledger write MUST be atomic (temp-file + rename), matching
  the conventions in `trigger_store.ts`.
- **I3.** On daemon restart, the ledger fully reconstructs the loop's
  view of the world; no other source of truth is required.
- **I4.** A ledger entry MUST NOT be deleted on success; it is retained
  so subsequent detections short-circuit on `NA2` (existing successful
  auto-fix) or `NA3` (cap tracking survives restarts).

## 7. Guards and Kill-Switches (Runaway Prevention)

Defense-in-depth: each guard is independent, and any single trip skips
the issue with a typed event.

| ID | Guard | Default | Env Var | Notes |
|----|---|---|---|---|
| GD1 | Global kill-switch | OFF | `AUTONOMOUS_DEV_SELF_IMPROVE` (`0`/`1`) | v1 defaults off; explicit opt-in |
| GD2 | Per-repo enrollment | OFF | `participate_in_auto_improvement: true` in ownership | Reuses `isEnrolled()` |
| GD3 | Per-issue attempt cap | 3 | `AUTONOMOUS_DEV_SELF_IMPROVE_MAX_ATTEMPTS_PER_ISSUE` | Attempts persist across restarts |
| GD4 | Global concurrent cap | 2 | `AUTONOMOUS_DEV_SELF_IMPROVE_MAX_CONCURRENT_GLOBAL` | Counted from active ledger entries |
| GD5 | Per-repo concurrent cap | 1 | `AUTONOMOUS_DEV_SELF_IMPROVE_MAX_CONCURRENT_PER_REPO` | Prevents single-repo starvation |
| GD6 | Daily cost cap (USD) | 5.00 | `AUTONOMOUS_DEV_SELF_IMPROVE_MAX_COST_USD_PER_DAY` | Read from cost telemetry per request |
| GD7 | Weekly cost cap (USD) | 25.00 | `AUTONOMOUS_DEV_SELF_IMPROVE_MAX_COST_USD_PER_WEEK` | Rolling 7-day window |
| GD8 | Back-off base | 60 min | `AUTONOMOUS_DEV_SELF_IMPROVE_BACKOFF_BASE_MINUTES` | Exponential: base × 2^(attempts−1), capped at 24h |
| GD9 | False-negative registry | present | `AUTONOMOUS_DEV_SELF_IMPROVE_FN_REGISTRY_PATH` | Path to fingerprint deny-list |
| GD10 | Watch-tick opt-in | ON when GD1=1 | (implicit) | The extra work only runs when GD1 is on |
| GD11 | Max issues per tick | 5 | `AUTONOMOUS_DEV_SELF_IMPROVE_MAX_ISSUES_PER_TICK` | Prevents a burst of dozens of requests from a single sweep |
| GD12 | Human-merge invariant | ALWAYS ON | (not overridable) | Auto-fix PRs never auto-merge in v1 |

**Order of evaluation (short-circuit):** GD1 → GD2 → GD10 → GD9 → NA1
(false-negative) → GD4/GD5 (concurrency) → GD6/GD7 (cost) → NA3 (attempt
cap) → NA4 (back-off) → NA2 (in-flight) → GD11 (tick cap).

## 8. User Stories

### S-MAINT-1 — Enrolled repo maintainer: pipeline-failure issue auto-becomes a fix request

- **As an** enrolled-repo maintainer,
- **I want** every autonomous-dev-filed `autodev:pipeline-failed` OPEN issue on my repo to auto-generate a fix request,
- **so that** I don't hand-copy the failure into `request submit`.
- **Given** repo `owner/name` is enrolled AND `AUTONOMOUS_DEV_SELF_IMPROVE=1` AND there exists an OPEN issue `#42` labeled `autodev:pipeline-failed` with a valid `<!-- autodev-failure: <fp> -->` marker AND the referenced request's `state.json.status === "failed"`,
- **When** the next `watch-tick` runs and none of GD3/GD4/GD5/GD6/GD7/NA1/NA2/NA4 trip,
- **Then** the daemon submits exactly one `type: bug` fix request whose description embeds the issue title/body and whose acceptance criteria are extracted from the issue's structured markers, the ledger is updated with the new request id, a `self_improve_request_submitted` event is emitted, and a comment linking the request is posted to issue `#42`.

### S-MAINT-2 — Human-labeled `autodev/auto-fix` issue

- **As a** human maintainer,
- **I want** to opt any of my issues into auto-fix by adding the `autodev/auto-fix` label,
- **so that** I can queue small fixes without leaving GitHub.
- **Given** an OPEN issue on an enrolled repo receives the label `autodev/auto-fix` from a human actor,
- **When** the next tick runs and guards pass,
- **Then** a fix request is submitted with `type` = `bug` unless an `autodev/type:refactor` (or similar) label is also present.

### S-OP-1 — Operator: kill-switch stops the loop instantly

- **As an** operator,
- **I want** `AUTONOMOUS_DEV_SELF_IMPROVE=0` to disable the entire loop,
- **so that** I can halt runaway behavior without pushing code.
- **Given** the env var is set to `0`,
- **When** any `watch-tick` runs,
- **Then** no issue is scanned, no ledger entry is written, no request is submitted, and a single `self_improve_disabled` event is emitted at most once per tick.

### S-OP-2 — Operator: guard trips are visible

- **As an** operator,
- **I want** every guard trip to write a typed event with the guard ID,
- **so that** I can audit why an issue was skipped without reading source.
- **Given** the loop scans an issue AND GD6 (daily cost cap) is exhausted,
- **When** the loop reaches the cost-cap check,
- **Then** it skips the issue and emits `self_improve_issue_skipped` with `{ guard: "GD6", repoId, issueNumber, costUsdWindow }`.

### S-OP-3 — Operator: repeated failure backs off exponentially

- **As an** operator,
- **I want** a failing auto-fix to back off rather than hammer the issue,
- **so that** cost and noise are bounded.
- **Given** ledger entry for `(repoId, issueNumber)` has `attempts=2` and `lastOutcome="failed"`,
- **When** GD8 base = 60min,
- **Then** `backoffUntil` = `lastAttemptAt + 60min × 2^(2−1)` = +120min; the loop skips the issue as NA4 until that time; on `attempts=3` and another failure, NA3 fires and the issue is capped until manually reset.

### S-REV-1 — Reviewer / release engineer: auto-fix PRs never auto-merge

- **As a** reviewer,
- **I want** self-fix PRs to always require my approval,
- **so that** the pipeline cannot self-merge changes without human oversight.
- **Given** an auto-fix request reaches the `integration`/`deploy` phase and opens a PR,
- **When** the PR is created,
- **Then** it carries the label `autodev:self-fix`, its description references the source issue, and no auto-merge bot is triggered even if other request types are configured for auto-merge.

### S-SM-1 — Supervisor maintainer: adding a new actionable class is one detector

- **As a** supervisor / trigger maintainer,
- **I want** the actionable-class catalog to be a data-driven table,
- **so that** a new class (A4, A5, ...) requires only a new row + detector.
- **Given** the actionable-classes registry in `intake/triggers/self_improve/actionable.ts`,
- **When** a new class is appended,
- **Then** no changes to `watch_tick.ts`'s main loop are required and jest picks up new tests automatically.

### S-SAFE-1 — Safety reviewer: false-negative registry blocks bad auto-fixes

- **As the** safety reviewer,
- **I want** to add a failure fingerprint to a deny-list,
- **so that** the loop never auto-submits a request for the corresponding class of issue.
- **Given** `AUTONOMOUS_DEV_SELF_IMPROVE_FN_REGISTRY_PATH` contains the fingerprint `abc123`,
- **When** the loop encounters an issue tagged with `<!-- autodev-failure: abc123 -->`,
- **Then** the issue is skipped with `self_improve_issue_skipped` and `{ guard: "NA1", fingerprint: "abc123" }`.

### S-BOT-1 — Issue author: safe, additive touch on the issue

- **As an** issue author,
- **I want** the daemon to only add a linking comment (and optionally a status label) to my issue,
- **so that** my title/body/status are never silently changed.
- **Given** the loop submits a fix request for issue `#42`,
- **When** it acts on the issue,
- **Then** it posts one comment with the request ID and, optionally, adds the label `autodev:in-progress`; it MUST NOT edit the title, body, or close the issue.

## 9. Functional Requirements

Each requirement is testable; each maps to ≥1 user story and ≥1 rule / guard.

### FR-DET — Detection layer

- **FR-DET-01.** The daemon MUST expose a `scanIssuesForEnrolledRepos()`
  entry point invoked from `runWatchTick` (or a peer function invoked
  from the same tick) that iterates all repos where `isEnrolled()`
  returns true and enumerates OPEN issues via `gh api repos/<owner/name>/issues?state=open`
  filtered by `labels ∈ {autodev:pipeline-failed, autodev:reviewer-finding, autodev/auto-fix}`.
- **FR-DET-02.** Each candidate issue MUST be classified against the §5
  catalog exactly once per tick, producing one of `{A1, A2, A3, NA1..NA7}`.
- **FR-DET-03.** The classifier MUST NOT open more than
  `AUTONOMOUS_DEV_SELF_IMPROVE_MAX_ISSUES_PER_TICK` (default 5) fix
  requests per tick even if more issues are actionable; the surplus
  issues are re-evaluated next tick (FIFO by `updated_at`).
- **FR-DET-04.** Detection MUST NOT mutate any GitHub issue (no
  comment, no label, no close) until after the guard chain passes and
  the request submission succeeds.
- **FR-DET-05.** For A1/A2, the daemon MUST resolve the confirmed-real
  evidence per §5: read the referenced `state.json`, verify status is
  actually `failed` (not `paused`, `done`, or missing). If evidence
  cannot be resolved within 500 ms, the issue is skipped as NA7.
- **FR-DET-06.** For A3, the daemon MUST verify the `autodev/auto-fix`
  label was applied by a non-bot GitHub actor by inspecting the
  issue's `events` timeline via `gh api /repos/.../issues/.../events`.
  Bot-applied `autodev/auto-fix` labels are NOT treated as human
  opt-in.
- **FR-DET-07.** The classifier MUST NOT crash the tick on a single
  bad issue; per-issue errors are caught and downgraded to `NA7` with
  the raw error stringified in the event evidence.

### FR-SUB — Submission layer

- **FR-SUB-01.** For each A1/A2/A3 classification that survives guards,
  the daemon MUST call `request submit` (via the CLI adapter used by
  `plugins/autonomous-dev/intake/adapters/cli_adapter.ts`) with:
  - `--target-repo` = the enrolled repo path
  - `--type` = `bug` (A1/A2) or per A3 label
  - `--description` = a structured payload embedding the issue title
    verbatim, the issue body verbatim, and the derived acceptance
    criteria (§10)
  - `--source` = `self-improve`
  - `--priority` = the priority mapped from label (see FR-SUB-04)
- **FR-SUB-02.** The submission MUST be idempotent: if a submission
  fails after creating the request (e.g., ledger write fails), the
  request MUST be marked `cancelled` on next tick reconciliation to
  keep the ledger and intake DB consistent.
- **FR-SUB-03.** On successful submission, the daemon MUST write the
  ledger entry atomically **before** posting the GitHub comment, so a
  crash between the two leaves no dangling comment without a request.
- **FR-SUB-04.** Priority mapping: default priority `2`; issue labels
  `autodev/priority:P0|P1|P2|P3` MUST override to `0..3` respectively.
- **FR-SUB-05.** Every submitted request MUST have
  `state.json.self_improve.sourceIssue = { repoId, issueNumber, url,
  fingerprint }` set at creation so downstream phases can attribute
  cost and enforce the human-merge gate.
- **FR-SUB-06.** The daemon MUST post exactly one comment to the
  source issue on submission, referencing the request ID and PR URL
  (when known). The comment body MUST contain the marker
  `<!-- autodev-self-improve: REQ-NNNNNN -->` so subsequent ticks can
  detect prior linkage without re-scanning the ledger.

### FR-MERGE — Human-merge gate

- **FR-MERGE-01.** Requests with `source: "self-improve"` MUST NOT be
  eligible for any auto-merge path. This MUST be enforced in the
  `integration` / `deploy` phase branches, not only at submission time.
- **FR-MERGE-02.** PRs opened by a self-improve request MUST carry the
  label `autodev:self-fix`. Absence of this label on a PR whose branch
  matches the self-improve pattern MUST fail the pre-merge invariant
  check.
- **FR-MERGE-03.** Any implementation of auto-merge for other request
  classes MUST test `source !== "self-improve"` explicitly and MUST NOT
  rely on label-only checks alone.

### FR-GUARD — Guards

- **FR-GUARD-01.** The guard chain MUST evaluate in the order defined
  in §7; the first tripping guard short-circuits.
- **FR-GUARD-02.** GD1 (`AUTONOMOUS_DEV_SELF_IMPROVE`) MUST default to
  `0`; any value other than `1` is treated as off.
- **FR-GUARD-03.** Cost caps (GD6/GD7) MUST read from the existing
  cost-accrual field
  (`plugins/autonomous-dev/intake/adapters/*` writes `cost_accrued_usd`
  on state.json — see request state schema); the loop MUST sum
  `cost_accrued_usd` for all self-improve requests within the rolling
  window before submission.
- **FR-GUARD-04.** GD8 back-off MUST be computed as
  `min(base × 2^(attempts−1), 24h)` and stored in
  `ledger.entries[<key>].backoffUntil` as ISO-8601.
- **FR-GUARD-05.** GD11 (tick cap) MUST count only successful
  submissions in the current tick, not classifier iterations.
- **FR-GUARD-06.** Every guard trip MUST emit exactly one
  `self_improve_issue_skipped` event with `{ guard: "<GD/NA ID>", repoId,
  issueNumber, evidence }`.

### FR-LEDGER — Ledger

- **FR-LEDGER-01.** The ledger file path MUST be
  `~/.autonomous-dev/state/self-improve/ledger.json` and MUST be
  created lazily on first write with mode `0600`.
- **FR-LEDGER-02.** All ledger mutations MUST be atomic (temp file +
  `rename`).
- **FR-LEDGER-03.** The ledger MUST expose a `getInFlightAutoFixRequest`
  reader used by NA2 that returns any non-terminal request for the key.
- **FR-LEDGER-04.** The ledger MUST support a `resetLedgerEntry(repoId,
  issueNumber)` operation invoked from a new CLI verb
  `autonomous-dev self-improve reset <repoId> <issueNumber>` for
  operator recovery.

### FR-CLI — CLI surface

- **FR-CLI-01.** The `autonomous-dev triggers watch-tick` verb (or a
  new peer verb, e.g., `autonomous-dev self-improve tick`) MUST run
  the loop. The choice is up to the spec phase, but the change MUST
  NOT break existing `watch-tick` semantics.
- **FR-CLI-02.** A new `autonomous-dev self-improve status` verb MUST
  print a table of ledger entries with `{ repoId, issueNumber,
  attempts, lastOutcome, backoffUntil }`.
- **FR-CLI-03.** A new `autonomous-dev self-improve reset <repoId>
  <issueNumber>` verb MUST clear the attempt counter and back-off for
  that entry.
- **FR-CLI-04.** All new verbs MUST print a `--help` summary and exit
  non-zero on unknown sub-verb.

### FR-CONFIG — Configuration

- **FR-CFG-01.** All thresholds in §7 MUST be overridable via
  documented env vars with the `AUTONOMOUS_DEV_SELF_IMPROVE_*` prefix.
- **FR-CFG-02.** Env-var parsing MUST fail-safe on malformed values
  (fall back to default; emit `self_improve_config_invalid` event).
- **FR-CFG-03.** The false-negative registry file MUST be optional; if
  absent, no fingerprint is ever treated as false-negative and the loop
  proceeds normally (i.e., NA1 never fires).

### FR-OBS — Observability

- **FR-OBS-01.** Every classification (actionable or skipped), every
  submission, every ledger update, and every guard trip MUST emit a
  typed event via the existing audit sink used by `watch_tick`.
- **FR-OBS-02.** A single `self_improve_tick_summary` event MUST be
  appended at the end of each tick with counts of `scanned`,
  `submitted`, `skipped_by_guard: { GD1..GD11, NA1..NA7 }`, and
  `errors`.
- **FR-OBS-03.** Auto-submitted requests MUST include the source
  issue URL in the request's PRD title metadata (so downstream PRDs
  and reviews can reference it).
- **FR-OBS-04.** All new event types MUST have a JSON schema entry in
  `docs/schemas/events/`.

### FR-TEST — Test coverage

- **FR-TEST-01.** Jest unit tests MUST cover: each detection rule row
  (A1/A2/A3), each NA row (NA1–NA7), each guard (GD1–GD11), ledger

- **FR-TEST-02.** An end-to-end jest test MUST simulate: enrolled repo

- **FR-TEST-03.** An end-to-end jest test MUST simulate: enrolled repo

  self-improve PR cannot be auto-merged (FR-MERGE-01).
- **FR-TEST-05.** Tests MUST cover the false-negative registry (NA1)

## 10. Description / Acceptance-Criteria Extraction

When wiring `--description` for the submitted request, the daemon MUST
produce a payload with these sections in order, so the intake PRD phase
has structured context:

1. **Header line.** `Auto-generated from <repoId>#<issueNumber>: <issue title>`.
2. **Source URL.** `Source: <issue html_url>`.
3. **Original issue body.** Verbatim, wrapped in a fenced block for
   preservation.
4. **Derived acceptance criteria.** Extracted from the issue body
   using these rules, in order:
   - If the body contains a `## Acceptance Criteria` (case-insensitive)
     section, use its bullets verbatim.
   - Else, if the body contains a fenced ` ```test ` or fenced
     ` ```expected ` block, wrap those as criteria.
   - Else, synthesize a single criterion:
     "Reproduce the failure described above; verify the corresponding
     test/check now passes after the fix."
5. **Constraints.** A fixed footer:
   `Constraints: (a) do not auto-merge; (b) preserve existing tests;
   (c) do not modify unrelated code paths; (d) if the failure cannot be
   reproduced, mark the request paused with reason
   \`self-improve-not-reproducible\`.`

The exact string format is owned by the spec phase; this PRD constrains
only the *content* invariants above.

## 11. Non-Functional Requirements

- **NFR-PERF-01.** Per-tick overhead of the self-improve scan MUST add
  ≤5 s wall-clock on a repo with ≤50 open issues (excluding gh CLI
  network latency). Measured via a jest timing test with mocked gh.
- **NFR-PERF-02.** The classifier MUST handle up to 200 open issues per
  repo without OOM (streaming iteration, not full load).
- **NFR-RELIABILITY-01.** A failure inside the loop MUST NOT crash
  `runWatchTick`; all errors are caught and downgraded to
  `self_improve_error` events.
- **NFR-RELIABILITY-02.** Loop logic MUST be safe to invoke twice
  concurrently on the same host (file-lock the ledger, or make writes
  compare-and-swap on the file's `mtime`); at most one submission per
  `(repoId, issueNumber)` even under concurrent ticks.
- **NFR-COMPAT-01.** No existing field in `state.json`, `intake.db`, or
  `triggers.json` is renamed or removed. New fields under
  `state.json.self_improve.*` are additive.
- **NFR-COMPAT-02.** With `AUTONOMOUS_DEV_SELF_IMPROVE=0`, existing
  `watch-tick` behavior is bit-for-bit identical; verified by
  regression test.
- **NFR-SECURITY-01.** The daemon MUST NOT read or transmit any GitHub
  content beyond OPEN issues on enrolled repos. Private data outside
  the enrolled scope is never accessed.
- **NFR-SECURITY-02.** Ledger file mode MUST be `0600`; parent dir
  mode `0700`.
- **NFR-SECURITY-03.** Any GitHub API call MUST use the existing gh CLI
  auth path; no new tokens are introduced.
- **NFR-COST-01.** The rolling-window cost accounting MUST be
  deterministic to within ±$0.01 per submitted request (rounding of
  `cost_accrued_usd` values from state.json).
- **NFR-MAINT-01.** Adding a new actionable class (A4, ...) MUST
  require edits to ≤3 files: the actionable-class registry, one new
  jest test, and (optionally) one schema entry.
- **NFR-OBS-01.** All new event types MUST validate against schemas
  committed under `docs/schemas/events/`.

## 14. Risks and Mitigations

| Risk | Severity | Mitigation |
|---|---|---|
| Runaway loop: fix request fails → new failure issue → new fix request | Critical | GD3 attempt cap; GD8 back-off; ledger persistence; false-negative registry |
| Cost blow-out on a spammy failure mode | Critical | GD6/GD7 daily/weekly cost caps; GD11 per-tick cap |
| Auto-fix of a false-negative wastes compute and pollutes history | High | §5 confirmed-real-evidence rule (NA7); GD9 false-negative registry; require `state.json.status === "failed"` for A1 |
| Concurrent daemon instances double-submit | High | Ledger file-lock + compare-and-swap; NFR-RELIABILITY-02 |
| A human closes the source issue but the ledger doesn't notice → next tick re-submits | Medium | On tick start, ledger reconciler verifies each entry's issue is still OPEN; closed issues transition ledger entry to `idle` and stop counting; documented via event |
| Auto-fix PR silently auto-merges via label-only auto-merge bot | Critical | FR-MERGE-01/02/03 require explicit `source: "self-improve"` check, not label alone |
| Label taxonomy drift (`autodev/auto-fix` vs `autodev:auto-fix`) | Medium | Constants file `intake/triggers/self_improve/labels.ts`; jest fixture covers both dashed & colon-separated forms; documented as `autodev/auto-fix` (slash form) authoritative |
| A single very large issue body inflates the request description | Low | Truncate issue body to 32 KB in `--description`; emit `self_improve_body_truncated` event |
| Enrollment revoked mid-flight | Low | On tick, re-evaluate `isEnrolled()`; if false, stop new submissions but let in-flight requests finish naturally |
| Default-off shipping means feature is inert | Low | Ship in v1 default-off; flip default to on only after one observation week with zero critical guard trips |

## 15. Open Questions

1. **OQ-1 — Bot identity.** What is the exact GitHub login used by
   autonomous-dev when it files issues via `gh`? Needed for A1 author
   check and for A3 "label applied by non-bot" check.
2. **OQ-2 — Label naming.** Is the authoritative human-opt-in label
   `autodev/auto-fix` (slash) or `autodev:auto-fix` (colon)? PRD
   recommends the slash form; align with existing labels first.
3. **OQ-3 — Reviewer-finding structured block format.** What is the
   exact schema of the `<!-- autodev-reviewer: ... -->` block a reviewer
   emits when filing a finding-issue? Needed for A2 classification and
   for extracting acceptance criteria.
4. **OQ-4 — False-negative registry format.** JSON, YAML, or plain
   newline-delimited fingerprints? Recommend JSON with `{ fingerprint,
   addedBy, reason, addedAt }` per entry.
5. **OQ-5 — Cost-cap window boundary.** Is "daily" a UTC calendar day
   or a rolling 24 h window? Recommend rolling 24 h to avoid midnight
   surges.
6. **OQ-6 — Tick interval.** Does the existing `watch-tick` cadence
   (currently configured externally) meet the "≤ 2 × tick" latency
   target? If not, do we introduce a self-improve-specific cadence?
7. **OQ-7 — Portal integration.** In v2, should the portal expose a
   "queue of pending auto-fixes" view? Out of scope for v1 but flagged.
8. **OQ-8 — Interaction with REQ-000056 self-heal.** If self-heal
   auto-resolves a failure that would otherwise have surfaced as a
   `autodev:pipeline-failed` issue, is there any risk of a race where
   both loops act on the same underlying failure? Recommendation:
   self-heal runs *before* an issue is filed; by the time an issue
   exists, self-heal has already given up. Confirm the ordering in
   spec.
9. **OQ-9 — Merge-gate enforcement point.** Is the human-merge gate
   enforced in `supervisor-loop.sh` (integration phase), in a PR-open
   hook, or both? Recommend both, defense-in-depth.
10. **OQ-10 — Attempt-cap reset policy.** Should a merged human fix on
    the source issue automatically reset the attempt counter, or is
    manual `self-improve reset` required? Recommend manual for v1.

### C. Glossary

- **Enrolled repo.** A repo where
  `ownership.repos[<repoId>].participate_in_auto_improvement === true`,
  per `isEnrolled()`.
- **Actionable issue.** An OPEN issue on an enrolled repo whose class
  matches one of §5's A1/A2/A3 rules and whose confirmed-real evidence
  resolves.
- **Ledger.** The persistent record of self-improve attempts and their
  outcomes, keyed by `(repoId, issueNumber)`.
- **Fingerprint.** The `<fp>` string inside
  `<!-- autodev-failure: <fp> -->`, as produced by
  `failureFingerprint` in `issue_filer.ts`.
- **False-negative registry.** A deny-list of fingerprints for which
  the loop will never open a fix request (GD9 / NA1).
- **Self-fix PR.** A pull request opened by an auto-submitted
  `source: "self-improve"` request. Never auto-merged in v1.
- **Kill-switch.** The global env var `AUTONOMOUS_DEV_SELF_IMPROVE`;
  set to `0` (or unset), the whole loop is inert.

### D. Alignment with Repo Principles

- **"Human is the final escalation, not the first."** The loop moves
  the *initial* action left to the daemon but preserves the
  human-merge gate as the terminal invariant.
- **"Least-destructive continue rather than hard fail."** Guards prefer
  skipping to submitting under any ambiguity; NA7 downgrades unknown
  errors to a logged skip.
- **"Additive, backwards-compatible schema evolution."** All new state
  lives under `self_improve.*` sub-objects; no existing field is
  renamed or removed.
- **"Default-off, one-flag opt-in."** GD1 kill-switch + GD2 enrollment
  is the compound gate; both must be positive to activate the loop.
