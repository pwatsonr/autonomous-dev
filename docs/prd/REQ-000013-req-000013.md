## 2. Problem Statement

The autonomous-dev daemon maintains request lifecycle state in **two stores** that
must remain consistent:

1. **Local `state.json`** — the per-request file at
   `<repo>/.autonomous-dev/requests/<REQ>/state.json`, written by the
   supervisor loop on every phase boundary.
2. **The SQLite ledger** — `~/.autonomous-dev/intake.db`, the `requests` table
   (canonical store queried by the intake CLI, portal, kill/resume/feedback
   handlers, and starvation/orphan monitors).

When the daemon advances a request's `current_phase` (and `status`) in
`state.json`, the matching row in `intake.db` is **never updated**. The two
stores drift apart.

### Evidence

- During **REQ-000011**, the daemon walked all 14 phases and updated
  `state.json` correctly, yet `sqlite3 ~/.autonomous-dev/intake.db
  'SELECT current_phase, status FROM requests WHERE request_id = ?'`
  still returned `('intake','queued')`.
- **Observable side effect:** `autonomous-dev request feedback REQ-NNNNNN` is
  blocked because the ledger reports `status='queued'`, a state from which
  feedback is disallowed.

### Root Cause (confirmed by code inspection)

The phase-advancement write path in
`plugins/autonomous-dev/bin/supervisor-loop.sh`:

- `advance_phase()` (≈ lines 2232–2434) atomically updates `state.json`
  (`current_phase`, `status`, `updated_at`, events.jsonl) but issues **no**
  `UPDATE requests …` against `intake.db`.
- `intake_to_prd_if_needed()` (≈ lines 2443–2468) does the intake→prd
  transition in `state.json` only — again with **no** ledger write.

The only place the loop writes the `requests` table is the orphan-reconciliation
path (≈ line 2778), which runs on a separate cadence and does not cover normal
phase boundaries. There is no shared helper that mirrors a `state.json`
transition into `intake.db`.

### Status-vocabulary mismatch (in scope to handle)

`state.json` and `intake.db` use **different status vocabularies**, so the fix
cannot copy `status` verbatim:

| `state.json` status | `intake.db` status (CHECK constraint: `queued`, `active`, `paused`, `cancelled`, `done`, `failed`) |
|---------------------|---------|
| `queued` | `queued` |
| `running` | `active` |
| `gate` | `active` |
| `done` | `done` |
| `failed` | `failed` |
| `paused` | `paused` |

The REQ-000013 acceptance criterion confirms: after intake→prd, the ledger must
read `('prd','active')` — i.e. `running` maps to `active`.

## 3. Goals and Non-Goals

### Goals

- G1: Every daemon-driven `state.json` phase/status transition is mirrored to
  the corresponding `intake.db` `requests` row within the same operation.
- G2: The intake→prd auto-transition updates `intake.db` as well as
  `state.json`.
- G3: `status` values are correctly mapped from the state.json vocabulary to the
  intake.db CHECK-constrained vocabulary (no constraint violations).
- G4: Ledger updates are resilient — a SQLite failure is logged and does not
  silently corrupt or crash the phase advance; `state.json` remains the
  authoritative driver.
- G5: Regression coverage that simulates phase advancement and asserts the
  intake row reflects it.

### Non-Goals

- NG1: Backfilling/repairing historical drifted rows (e.g. REQ-000011) beyond
  what the existing orphan-reconciliation path already handles — unless trivial.
  Tracked separately if needed.
- NG2: Changing the two-store architecture (e.g. making `state.json` derive from
  SQLite or vice versa).
- NG3: Altering the status vocabulary or schema of either store.
- NG4: Changes to the `feedback` command's own state-machine rules beyond what is
  required for it to unblock once the ledger is correct.
- NG5: Choosing a new SQLite access technology — the existing `sqlite3` CLI
  approach (already used in the orphan path) is retained unless review directs
  otherwise.

## 5. User Stories

### Epic A — Ledger consistency on transitions

**US-1 (P0): Phase advance updates the ledger**
> As a daemon maintainer, I want each phase advancement to update the
> `intake.db` row so that the ledger never drifts from `state.json`.

- Given a request at phase `intake` with `state.json` `status=queued` and an
  `intake.db` row `(intake, queued)`,
- When the daemon advances it to `prd`,
- Then `SELECT current_phase, status FROM requests WHERE request_id = ?`
  returns `('prd','active')`.

**US-2 (P0): Status vocabulary mapped correctly**
> As a maintainer, I want state.json statuses translated to the ledger's allowed
> values so writes never violate the CHECK constraint.

- Given a transition that sets `state.json status` to `running` or `gate`,
- When the ledger row is updated,
- Then the row's `status` is `active` (not `running`/`gate`), and `done`/`failed`
  map to `done`/`failed`.

**US-3 (P0): Terminal and failure transitions mirror to ledger**
> As an operator, I want completed/failed requests to show the correct ledger
> status so downstream commands behave correctly.

- Given a request that reaches a terminal phase (status `done`) or fails
  (status `failed`),
- When `advance_phase` finalizes,
- Then the ledger row reflects `done`/`failed` and the correct `current_phase`.

**US-4 (P0): feedback unblocked after advance**
> As a submitter, I want `request feedback` to work once a request is past
> intake so I am not blocked by a stale `queued` status.

- Given a request advanced from `intake` to `prd`,
- When I run `autonomous-dev request feedback REQ-NNNNNN`,
- Then the command is not rejected on the basis of ledger `status='queued'`.

### Epic B — Resilience & observability

**US-5 (P1): Ledger write failure is logged, not silent**
> As a maintainer, I want a failed/missing SQLite update to emit a clear log
> entry rather than failing silently (the original root cause).

- Given the `sqlite3` CLI is unavailable or the UPDATE errors,
- When a transition occurs,
- Then a WARN/ERROR is logged with request_id and phase, and `state.json`
  remains authoritative (advance is not aborted mid-write).

**US-6 (P1): Missing ledger row is handled**
> As a maintainer, I want an `UPDATE` that matches zero rows to be detected and
> logged so silent no-ops are caught.

- Given the `requests` row for the request does not exist,
- When the transition update runs,
- Then the zero-row result is logged (and optionally reconciled), not treated as
  success.

## 6. Functional Requirements

Grouped by area. Each is testable.

### FR-A: Shared ledger-sync helper

- **FR-A1** A single helper function (e.g. `sync_intake_db_row`) SHALL accept a
  request_id, target phase, and state.json status, and execute an
  `UPDATE requests SET current_phase = ?, status = ?, updated_at = ? WHERE request_id = ?`
  against `${INTAKE_DB}`.
- **FR-A2** The helper SHALL map state.json status → intake.db status per the
  table in §2 (`running`→`active`, `gate`→`active`, `queued`→`queued`,
  `done`→`done`, `failed`→`failed`, `paused`→`paused`). Any unmapped value SHALL
  be logged and SHALL NOT be written (avoid CHECK violation).
- **FR-A3** The helper SHALL use the same `sqlite3` CLI invocation style and
  parameter-binding approach already used in the orphan-reconciliation path
  (≈ line 2778) for consistency and to avoid native-binding runtime failures.
- **FR-A4** `updated_at` written to the ledger SHALL equal the `ts` used for the
  matching `state.json` write within the same transition.

### FR-B: Wire the helper into all transition sites

- **FR-B1** `advance_phase()` SHALL call the helper after each successful
  `state.json` write for: (a) advance-to-next-phase, (b) terminal/`done`,
  (c) failure/`failed`, and (d) review-failure reset to author phase.
- **FR-B2** `intake_to_prd_if_needed()` SHALL call the helper after writing
  `current_phase=prd`, `status=running` to `state.json`, producing ledger
  `('prd','active')`.
- **FR-B3** The ledger update SHALL occur for both `gate` and `running`
  next-status branches in `advance_phase`.

### FR-C: Resilience & logging

- **FR-C1** A `sqlite3` failure (non-zero exit) SHALL be logged at ERROR level
  with request_id, target phase, and the mapped status; the phase advance SHALL
  still complete (state.json remains authoritative).
- **FR-C2** If `sqlite3` is not on PATH, the helper SHALL log a single ERROR and
  return without aborting the advance (mirroring orphan-path behavior).
- **FR-C3** When the `UPDATE` matches zero rows, the helper SHALL log a WARN
  identifying the request_id (detects missing ledger rows / silent no-ops).

### FR-D: Tests

- **FR-D1** At least **3 integration tests** SHALL verify the `intake.db` row
  updates on phase boundaries (minimum: intake→prd, a mid-pipeline advance to a
  `gate` review phase, and a terminal `done` transition).
- **FR-D2** A **regression test** SHALL simulate phase advancement and assert the
  intake row reflects the new `(current_phase, status)` — directly reproducing
  the REQ-000011 scenario (must fail before the fix, pass after).
- **FR-D3** A test SHALL assert the status mapping (`running`→`active`,
  `gate`→`active`) produces no CHECK-constraint violation.
- **FR-D4** A test SHALL assert resilience: a simulated `sqlite3`
  failure/zero-row update logs the expected message and does not abort the
  advance.

## 7. Non-Functional Requirements

- **NFR-1 (Correctness):** After any daemon transition, `state.json`
  `(current_phase, mapped-status)` and `intake.db` `(current_phase, status)`
  SHALL be equal for the affected request (eventual consistency within the same
  transition call).
- **NFR-2 (Performance):** Each transition adds at most one bounded `sqlite3`
  CLI invocation; added latency per phase boundary SHALL be < 200 ms on the
  reference dev machine (phase boundaries are infrequent, so this is generous).
- **NFR-3 (Atomicity/Safety):** The ledger write SHALL NOT corrupt `state.json`
  or leave a partial `.tmp`/`.restore` file on failure; existing atomic
  write-then-`mv` semantics for `state.json` are preserved.
- **NFR-4 (Backward compatibility):** No schema change to `intake.db`; the
  existing `requests` CHECK constraint on `status` SHALL continue to hold.
- **NFR-5 (Test integrity):** Existing portal test suite (**1563 tests**) SHALL
  remain 100% pass. The bats suite SHALL have no regressions.
- **NFR-6 (Observability):** All ledger-sync failures are discoverable via the
  daemon log (no silent failures — directly addresses the original bug class).

## 8. Technical Constraints

- **TC-1:** Implementation lives in
  `plugins/autonomous-dev/bin/supervisor-loop.sh`; `${INTAKE_DB}` is already
  defined (`${DAEMON_HOME}/intake.db`, line 18).
- **TC-2:** Use the `sqlite3` CLI (not `node`/`better-sqlite3`) — per existing
  in-code rationale (≈ lines 2718–2724): the native binding caused runtime
  failures; the CLI ships in the daemon environment.
- **TC-3:** `intake.db` `status` CHECK constraint allows only
  `queued|active|paused|cancelled|done|failed`. The state.json vocabulary
  (`queued|running|gate|done|failed|paused`) must be mapped, not copied.
- **TC-4:** `current_phase` in `intake.db` is free-form `TEXT` (no CHECK), so
  pipeline phase names (`prd`, `prd_review`, …) can be written directly.
- **TC-5:** SQLite parameter binding via `sqlite3` CLI requires `.parameter set`
  (sqlite ≥ 3.43), consistent with the orphan path.
- **TC-6:** The daemon owns phase transitions; this work MUST NOT change
  `current_phase`/`status` ownership semantics — only mirror the daemon's writes.

## 10. Open Questions

- **OQ-1:** Should `gate` map to `active` (assumed here, per acceptance) or to a
  distinct ledger status? Confirm with intake/DB owners. The acceptance criterion
  only pins down `running`→`active`.
- **OQ-2:** Should historical drifted rows (e.g. REQ-000011) be backfilled as
  part of this change, or left to the orphan-reconciliation path? (Currently
  NG1.)
- **OQ-3:** When the `UPDATE` matches zero rows (missing ledger row), should the
  helper attempt an `INSERT`/reconcile, or only log? (FR-C3 currently logs.)
- **OQ-4:** Should a ledger-write failure ever be surfaced beyond logs (e.g.
  metric/alert), given NFR-6 emphasizes observability?
- **OQ-5:** Is `paused` reachable through `advance_phase`, or only via the pause
  path (lines ~1674)? If the pause path also writes `state.json` without a ledger
  write, should it be included in this fix's scope?
