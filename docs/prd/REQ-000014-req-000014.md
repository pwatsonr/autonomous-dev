## 2. Problem Statement

REQ-000013 introduced a `sync_intake_db_row` helper in
`plugins/autonomous-dev/bin/supervisor-loop.sh` and wired it into the daemon's
**phase-progression** writers, so a "happy path" walk through the pipeline now
mirrors `state.json` ŌåÆ `intake.db`. Specifically (line numbers as of HEAD):

| # | Site | Function | Line |
|---|------|----------|------|
| A | intakeŌåÆprd auto-transition | `intake_to_prd_if_needed` | 2605 |
| B | phase advance (running/gate) | `advance_phase` (pass) | 2471 |
| C | terminal `done` | `advance_phase` (pass, terminal) | 2413 |
| E | review-failure reset to author phase | `advance_phase` (fail) | 2529 |
| F | retry-exhausted `failed` | `advance_phase` (fail, terminal) | 2539 |

The bug class is **not fully closed**. The daemon has at least one other
writer that updates `state.json.status` and `state.json.current_phase` without
calling `sync_intake_db_row`, re-creating the same drift symptom in a
different state transition:

### Residual gap (confirmed by code inspection)

**`escalate_to_paused` (Ōēł lines 1659ŌĆō1703):** when retries are exhausted, the
function atomically writes `state.json` with `status = "paused"` and
`current_phase_metadata.paused_reason/paused_at`, appends a
`retry_exhaustion` event, and emits an alert. It **does not** call
`sync_intake_db_row`. After this transition, `intake.db.status` continues to
show the prior value (most commonly `active`) while `state.json.status` is
`paused`. The two stores drift again ŌĆö re-introducing the original REQ-000011
class of bug for any paused request:

- `request resume REQ-NNNNNN` may be rejected because the ledger does not
  show `paused`.
- The portal/dashboards may incorrectly show the request as `active`.
- Starvation/orphan monitors may misclassify a legitimately paused request.

### Verification gap

REQ-000013's acceptance criterion was narrowly scoped to the intakeŌåÆprd
transition (`('prd','active')`). The acceptance for this follow-up must
include **the paused path** and any other transition site that mutates
`state.json.status` or `state.json.current_phase`.

### Other call sites verified clean (no fix required)

- `update_state_session` success path (Ōēł line 1895) ŌĆö sets only
  `current_phase_metadata.*`, `cost_accrued_usd`, `updated_at`. No `status`
  or `current_phase` change ŌåÆ no ledger drift.
- `update_state_session` error path (Ōēł line 1931) ŌĆö sets only retry/error
  metadata. No `status`/`current_phase` change ŌåÆ no ledger drift.
- `update_state_cost` (Ōēł line 2203) ŌĆö sets only `cost_accrued_usd`. No drift.
- Orphan reconciliation path (Ōēł line 2904) ŌĆö already issues its own
  `UPDATE requests`.

This PRD is therefore narrowly scoped: pause-path coverage plus a documented
audit of remaining writers.

## 3. Goals and Non-Goals

### Goals

- **G1:** Every daemon-driven `state.json` write that changes `status` or
  `current_phase` is mirrored to the corresponding `intake.db` row within the
  same operation.
- **G2:** `escalate_to_paused` mirrors its `status = "paused"` write to the
  ledger using the existing helper, so a paused request reads
  `('current_phase, "paused")` from `intake.db`.
- **G3:** An explicit, in-repo **audit table** documents every state.json
  writer in `supervisor-loop.sh` and whether it requires a ledger mirror,
  preventing future regression of the same class.
- **G4:** Re-verification of REQ-000013's intakeŌåÆprd acceptance criterion
  remains green (no regression).
- **G5:** Regression coverage for the pause path (the gap not covered by
  REQ-000013's tests).

### Non-Goals

- **NG1:** Re-implementing `sync_intake_db_row` ŌĆö REQ-000013's helper is
  reused as-is (status mapping, sqlite3 CLI invocation, parameter binding,
  log-only failure semantics, zero-row WARN).
- **NG2:** Adding ledger-write coverage for state.json writers that do **not**
  change `status` / `current_phase` (e.g. `update_state_cost`,
  `update_state_session` retry/cost path).
- **NG3:** Changing the orphan-reconciliation path (already writes the ledger).
- **NG4:** Adding new statuses to either vocabulary (the existing mapping ŌĆö
  including `paused ŌåÆ paused` ŌĆö already covers the requirement).
- **NG5:** Backfilling historical drifted rows beyond what the
  orphan-reconciliation cadence already does.
- **NG6:** User-driven CLI/portal actions (`request pause/resume/cancel/kill`)
  ŌĆö these are routed through a different path (intake-router-client) and
  tracked separately; this PRD covers daemon-driven transitions only.

## 6. Functional Requirements

Each FR is independently testable.

### FR-A: Pause path mirrors to ledger

- **FR-A1** `escalate_to_paused` SHALL call `sync_intake_db_row` immediately
  after the successful `mv` of the paused state.json, passing
  `(request_id, current_phase, "paused", ts)`, where `current_phase` is the
  phase value read from state.json at the time of the pause and `ts` is the
  same ISO-8601 timestamp written to `state.json.updated_at`.
- **FR-A2** The status mapping `paused ŌåÆ paused` (already supported by
  `map_state_status_to_intake`) SHALL be used; no mapping change is
  introduced.
- **FR-A3** A `sqlite3` failure or zero-row update SHALL be logged via the
  existing helper semantics (ERROR or WARN respectively) and SHALL NOT abort
  the pause; `state.json` remains authoritative.

### FR-B: Audit and exemption documentation

- **FR-B1** A maintainer-facing audit table SHALL be added to
  `docs/triage/` (or the equivalent owner-designated location) enumerating
  every `mv "${tmp}" "${state_file}"` site in `supervisor-loop.sh` (currently
  16 sites at lines 483, 1148, 1276, 1314, 1669, 1895, 1931, 1953, 2203,
  2390, 2431, 2492, 2521, 2584, 2805) with one of:
    - `MIRROR` (calls `sync_intake_db_row` ŌĆö name the call site),
    - `EXEMPT` (does not change `status`/`current_phase` ŌĆö name the fields
      written),
    - or `OUT_OF_SCOPE` (handled elsewhere, e.g. orphan path).
- **FR-B2** The audit SHALL be cross-referenced from a comment block above
  `sync_intake_db_row` (`supervisor-loop.sh:2250`), so a future reader
  finds it from the helper definition.

### FR-C: Tests

- **FR-C1** A regression test SHALL simulate retry exhaustion driving
  `escalate_to_paused` and SHALL assert `sqlite3 ${INTAKE_DB}` returns
  `(<phase>, 'paused')` for the affected request.
- **FR-C2** A test SHALL re-assert REQ-000013's intakeŌåÆprd acceptance
  criterion (`('prd','active')`) so this follow-up does not regress the
  predecessor.
- **FR-C3** A negative test SHALL assert that `update_state_session` /
  `update_state_cost` (non-status writers) do **not** call
  `sync_intake_db_row` (verifies the "EXEMPT" classification, e.g. by
  asserting no `sync_intake_db_row:` log line is emitted on a pure cost
  update).
- **FR-C4** A test SHALL assert resilience: a simulated `sqlite3` failure
  during the pause-path mirror produces the expected ERROR log line and
  does not abort the pause (`state.json` still ends `paused`).

### FR-D: Reuse, not reinvention

- **FR-D1** No new SQLite invocation pattern, no new status-mapping function,
  and no new logging convention SHALL be introduced. All ledger writes
  added by this change go through the existing
  `sync_intake_db_row` helper (`supervisor-loop.sh:2250ŌĆō2323`).

## 7. Non-Functional Requirements

- **NFR-1 (Correctness):** Post-transition, for any daemon-driven write to
  `state.json` that changes `status` or `current_phase`, the
  `(current_phase, mapped-status)` pair SHALL be equal between `state.json`
  and `intake.db` within the same call.
- **NFR-2 (Performance):** Each new mirror call adds at most one bounded
  `sqlite3` CLI invocation; added latency on the pause path SHALL be
  < 200 ms on the reference dev machine. Pause events are rare (only on
  retry exhaustion) ŌĆö the cost is negligible.
- **NFR-3 (Atomicity/Safety):** No corruption of `state.json` or leftover
  `.tmp`/`.restore` files on ledger-write failure; the existing
  write-then-`mv` semantics are preserved.
- **NFR-4 (Backward compatibility):** No schema change. The `status` CHECK
  constraint on `intake.db` continues to hold (`paused` is already allowed).
- **NFR-5 (Test integrity):** Existing portal test suite SHALL remain 100%
  pass. The bats suite SHALL have no regressions. The new tests SHALL be
  isolated (no shared writable state with other suites).
- **NFR-6 (Observability):** Every ledger-sync failure is discoverable via
  the daemon log; the pause-path mirror SHALL emit one of:
  `sync_intake_db_row: ledger updated ŌĆ”`, `sync_intake_db_row: 0 rows
  updated ŌĆ”`, or `sync_intake_db_row: sqlite3 UPDATE failed ŌĆ”`.

## 8. Technical Constraints

- **TC-1:** Implementation lives in
  `plugins/autonomous-dev/bin/supervisor-loop.sh`. `${INTAKE_DB}` is already
  defined (line 18).
- **TC-2:** `sync_intake_db_row` and `map_state_status_to_intake` are already
  present (lines 2237ŌĆō2323). Reuse without modification.
- **TC-3:** `intake.db` `status` CHECK constraint allows
  `queued|active|paused|cancelled|done|failed`. `paused` is already valid.
- **TC-4:** `escalate_to_paused` is called from `handle_phase_failure` (via
  retry-budget enforcement). The current phase value SHALL be read from the
  same state.json as the pause write to avoid race conditions.
- **TC-5:** The daemon owns phase/status transitions; this work strictly
  mirrors a daemon-driven `state.json` write to the ledger ŌĆö it does not
  change ownership or transition semantics.
- **TC-6:** No native SQLite bindings (`better-sqlite3`) are introduced.
  CLI-only, consistent with the orphan path and REQ-000013.

## 10. Open Questions

- **OQ-1:** Should the `escalate_to_paused` ledger mirror use the value of
  `current_phase` **before** the pause (the phase that exhausted retries)
  or a sentinel like `paused`? The intake.db CHECK constraint allows any
  TEXT for `current_phase`, so either is feasible ŌĆö recommend keeping the
  current phase (FR-A1) so dashboards still surface where the request paused.
- **OQ-2:** When the audit (FR-B1) is added, should "EXEMPT" writers receive
  a `# LEDGER-EXEMPT: status/current_phase not modified` comment inline at
  each `mv` site, in addition to the centralized table? (Recommendation: yes,
  for grep-ability ŌĆö but optional per reviewer preference.)
- **OQ-3:** Are there user-driven CLI/portal handlers (out of scope per
  NG6) that also mutate `state.json.status` (e.g. `cmd_pause`, `cmd_resume`,
  `cmd_cancel`)? If so, they should be tracked under a separate request.
  This PRD explicitly does not assume their behavior.
- **OQ-4:** Should the pause-path mirror also clear/update fields like
  `intake.db.paused_reason` if such a column exists in the schema?
  (Schema check pending; current helper writes only `current_phase`,
  `status`, `updated_at` ŌĆö recommended to leave as-is unless the schema
  has a reason field.)
- **OQ-5:** Should the re-verification test (FR-C2) be added in this PRD's
  scope or marked as a duplicate of REQ-000013's existing test? (If the
  REQ-000013 test already covers `('prd','active')`, FR-C2 collapses into a
  CI assertion rather than new code.)

## 11. Risk Assessment

| ID | Risk | Severity | Notes |
|----|------|----------|-------|
| R1 | Adding a `sqlite3` call inside `escalate_to_paused` increases latency on the failure path | Low | Pause is rare; <200 ms cost; NFR-2 |
| R2 | A future writer is added without ledger sync and silently regresses the bug | Medium | Mitigated by FR-B1 audit + FR-B2 in-source pointer + FR-C3 negative test |
| R3 | The pause-path mirror writes `current_phase` that differs from the value read by a concurrent reader | Low | Same atomicity window as the state.json write; `sync_intake_db_row` runs after the `mv`, same as other call sites |
| R4 | Bats/portal regressions when touching `supervisor-loop.sh` | Medium | NFR-5; run full suites |
| R5 | No schema change, no data migration | Low | TC-3, NFR-4 ŌĆö ledger schema untouched |
| R6 | Scope creep into user-driven CLI handlers (NG6) | Medium | Strictly out of scope here; tracked separately |

> **High-risk flags:** None introduce breaking API/schema changes or data
> migrations. R6 is the most likely scope-creep vector and should be
> rejected unless a follow-up PRD is opened.
