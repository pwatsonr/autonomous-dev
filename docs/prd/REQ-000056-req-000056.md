## 2. Problem Statement

The autonomous-dev supervisor (`supervisor-loop.sh`) currently treats only
**terminal phase failures** as first-class events: when `phase-result-*.json`
arrives with `status: "fail"` or `status: "error"`, the loop runs
`advance_phase()` / `check_retry_exhaustion()` / `escalate_to_paused()`. But
the pipeline experiences a *wider* class of in-flight degraded conditions
that today either (a) silently retry until exhaustion, (b) hang on a review
gate, or (c) escalate to PAUSED and wait for a human — even when an
automatic, safer remediation is obvious. Concretely (per #620 and its
linked failure-mode issues):

- **#615 — review-gate loops.** When the multi-reviewer chain (`run_review_gate_phase`)
  produces a `REQUEST_CHANGES` outcome on every iteration, the request
  bounces between author phase and review phase until `max_retries` is
  spent. Today there is no detection of "this is the same reviewer
  blocking on the same finding" — every loop is treated as an independent
  retry.
- **#616 — soft-timeout / phase-timeout extension.** The soft-timeout
  ceiling (`AUTONOMOUS_DEV_MAX_SOFT_TIMEOUT_REENTRIES`) was added in
  REQ-000051 / ADR-004, but once the ceiling is reached the supervisor
  immediately promotes to hard timeout even if the working tree shows
  ongoing progress. There is no policy that says "phase is making progress,
  extend the budget once before promoting."
- **#617 — verification false-negatives.** Phase-result `status: "fail"`
  is taken at face value: the supervisor does not re-read the produced
  test-result artifact to confirm the failure was real before failing the
  gate. #617 was intermittent precisely because the supervisor trusted a
  transient false-negative.
- **#618 — reviewer ERROR / unparseable output.** `run_review_gate_phase`
  synthesizes an `error` phase-result on CLI nonzero, bad JSON, or write
  failure, and `advance_phase()` then treats `error` like `fail`. There is
  no retry-once-then-exclude-if-non-blocking policy; a single flaky
  reviewer can fail the whole gate.
- **Suspiciously-empty / suspiciously-fast results.** A session that
  returns in <N seconds with an empty/sentinel `phase-result-*.json` is
  currently accepted as a pass; there is no sanity-check on result
  plausibility.
- **Novel / unknown failure.** If the supervisor encounters a state it
  doesn't recognize (e.g. malformed `state.json`, missing artifact path,
  ledger drift), the safest behavior is to pause for human review with
  full diagnostic context — but today it can hard-fail or loop.

**Principle (from #620):** every reachable failure mode must map to
(a) being *detected and recorded* always, and (b) an *automatic
remediation or safe continue*, with a human as the **final** escalation,
not the first.

The current code partially implements this for terminal failures
(`escalate_to_paused`, `emit_alert`, `record_soft_timeout`,
`emit_soft_timeout_promotion_alert`), but coverage of *in-flight*
degraded conditions is incomplete and inconsistent.

## 3. Goals and Non-Goals

### Goals (in scope)

1. **G1 — Comprehensive detection.** Every in-flight degraded condition
   enumerated in §5 is detected and recorded to
   `events.jsonl` with a typed event name, even if the immediate outcome
   is a safe continue. Detection is mandatory and unconditional; it does
   not depend on remediation succeeding.
2. **G2 — Per-mode automatic remediation.** Each failure mode in §5 maps
   to exactly one default remediation policy (table in §6). Remediation
   runs without human input.
3. **G3 — Human as final escalation only.** If remediation fails or the
   failure is novel, the supervisor pauses the request (`status: paused`)
   with a `paused_reason` that captures the failure class, full context
   pointers (events.jsonl offset, session log path, artifact paths), and
   an `issue_filed_url` when applicable — but never hard-fails silently.
4. **G4 — Reviewer-chain robustness.** `run_review_gate_phase` retries a
   single ERROR/unparseable reviewer once, then excludes that reviewer
   from the chain for the current phase if the reviewer is configured as
   non-blocking, recording an `excluded_reviewer` event.
5. **G5 — Self-verification before failing a gate on a test result.**
   When a phase-result claims `fail` and references a test-result
   artifact, the supervisor re-reads that artifact and confirms at least
   one test actually failed before propagating `fail`. If the artifact
   contradicts the claim, the supervisor flips the result to `pass` and
   emits a `verification_false_negative_corrected` event.
6. **G6 — Soft-timeout extension policy.** When the soft-timeout ceiling
   is reached **and** `snapshot_working_tree` shows progress since the
   last snapshot, the supervisor extends the budget exactly once before
   promoting to hard timeout, emitting a `phase_budget_extended` event.
7. **G7 — bats / unit coverage per failure mode.** Each detection
   pathway and each remediation pathway has at least one bats test (for
   bash) or a unit test (for the TypeScript surface in
   `review-gate.ts` / `review-gate-cli.ts`) asserting both detection and
   remediation behavior.
8. **G8 — Observability.** Every remediation emits an `emit_alert` of the
   appropriate type, and a single per-request "self-heal summary" is
   appended to `events.jsonl` at terminal transitions.

### Non-Goals (explicit exclusions)

- **N1.** Re-architecting the reviewer chain (only adding retry/exclude
  policy at the dispatch layer).
- **N2.** Changing the on-disk schema of `state.json` or `intake.db`
  beyond additive fields under `current_phase_metadata.self_heal.*`.
- **N3.** Designing a new ML-based novelty detector. "Novel failure" is
  defined heuristically as "no matching handler in the §6 mapping table."
- **N4.** Auto-creating GitHub issues without a configured token. Issue
  filing for novel failures is **opportunistic**: if the GitHub CLI / token
  is unavailable, the supervisor records the would-be issue body in
  `events.jsonl` and pauses for human filing.
- **N5.** Cross-request remediation (e.g., "this reviewer is flaky across
  the fleet"). Scope is per-request, in-run.
- **N6.** Replacing `escalate_to_paused`; this PRD layers on top of it.

## 5. Failure Modes (Detection Catalog)

Each row is a distinct in-flight degraded condition the supervisor must
detect. Detection emits a typed event to `events.jsonl` **even if
remediation succeeds**. Event payload fields are illustrative; the exact
schema is owned by the spec phase.

| ID | Failure Mode | Detection Trigger | Event Type | Source Issue |
|----|---|---|---|---|
| F1 | Review-gate loop (same reviewer, same finding) | `run_review_gate_phase` returned `REQUEST_CHANGES` from the **same reviewer** with a **substring-stable `.reason`** ≥3 consecutive iterations | `review_gate_loop_detected` | #615 |
| F2 | Repeated reviewer timeout | Reviewer-chain CLI exits with timeout sentinel (or per-reviewer phase-result `status: "timeout"`) ≥2 times within one phase | `reviewer_timeout_repeated` | #618 / #620 |
| F3 | Repeated phase timeout WITH progress | `check_phase_timeout` triggers ≥1 hard timeout **and** `snapshot_working_tree` diff between phase start and current snapshot is non-empty | `phase_timeout_with_progress` | #616 |
| F4 | Reviewer ERROR / unparseable output | `run_review_gate_phase` synthesizes an `error` result for `REVIEW_GATE_CLI_NONZERO`, `REVIEW_GATE_BAD_JSON`, or `REVIEW_GATE_RESULT_WRITE_FAILED` | `reviewer_error_detected` | #618 |
| F5 | Suspiciously-empty result | `phase-result-*.json` exists with `status: "pass"` **and** zero artifacts **and** session wall-clock < `AUTONOMOUS_DEV_MIN_PHASE_DURATION_SECONDS` (default: 5) **and** phase ∈ {prd, tdd, plan, spec, code} | `suspicious_empty_result` | #620 |
| F6 | Suspiciously-fast result | Session wall-clock < `AUTONOMOUS_DEV_MIN_PHASE_DURATION_SECONDS` for an author phase that previously took ≥10× that duration in last successful run | `suspicious_fast_result` | #620 |
| F7 | Verification false-negative | `phase-result-*.json` claims `fail` with `status_reason` referencing a test-result artifact, but re-reading the artifact shows zero failed tests | `verification_false_negative_detected` | #617 |
| F8 | Novel / unknown failure | Phase-result `status` ∉ {pass, fail, error, timeout}; OR `state.json` is unparseable; OR a required artifact path is missing | `novel_failure_detected` | #620 |
| F9 | State ledger drift (defensive) | `state.json.status` ≠ `intake.db.status` at phase boundary | `state_ledger_drift_detected` | #620 (defense-in-depth re REQ-000013/14) |

Every detection writes to `events.jsonl` with: `request_id`, `phase`,
`detected_at`, `failure_mode`, `evidence` (typed sub-object), and
`remediation_planned` (one of the policies in §6).

## 6. Remediation Policy Mapping

Every failure mode in §5 maps to exactly one default remediation. Policies
are codified as named functions in `supervisor-loop.sh` (or a sourced
`lib/self-heal.sh`); novel modes default to `R_PAUSE_FOR_REVIEW`.

| ID | Failure Mode | Remediation | Notes |
|----|---|---|---|
| F1 | Review-gate loop | `R_ESCALATE_REVIEWER_TIMEOUT_OR_FALL_BACK_TO_SINGLE_REVIEWER` — disable the chain for the remainder of this request's review phase, fall back to the single hardcoded reviewer | Recorded as `review_chain_disabled_for_request` |
| F2 | Repeated reviewer timeout | `R_ESCALATE_REVIEWER_TIMEOUT` — extend reviewer timeout once (configurable multiplier, default 2×), then fall back to single reviewer if still timing out | Emits `reviewer_timeout_extended` |
| F3 | Phase timeout WITH progress | `R_EXTEND_PHASE_BUDGET` — extend the phase budget by `AUTONOMOUS_DEV_BUDGET_EXTENSION_FACTOR` (default 1.5×) exactly once per phase per request; on second hit, promote to hard timeout via existing path | Emits `phase_budget_extended` (builds on #616 soft-timeout) |
| F4 | Reviewer ERROR / unparseable | `R_RETRY_ONCE_THEN_EXCLUDE_IF_NON_BLOCKING` — retry the failing reviewer once; on second failure, exclude that reviewer if `reviewer.blocking == false`, else fail the gate | Emits `reviewer_excluded` or `reviewer_failed_blocking` |
| F5 / F6 | Suspicious result | `R_REQUEUE_AUTHOR_PHASE_ONCE` — discard the result, decrement `retry_count` rollback to avoid double-counting, re-spawn the session once with a `suspicious_previous_result` hint | Emits `phase_requeued_suspicious` |
| F7 | Verification false-negative | `R_SELF_VERIFY` — re-read the test-result artifact; if zero failures, override the phase-result to `pass` and emit `verification_false_negative_corrected`; if artifact missing, re-run the test command before propagating `fail` | Does **not** consume retry budget |
| F8 | Novel failure | `R_CAPTURE_AND_PAUSE` — write a full diagnostic bundle (state.json snapshot, last 200 lines of session log, last 50 events.jsonl entries) to `requests/<id>/diagnostic-bundle-<ts>.json`; attempt opportunistic issue filing; transition to `paused` with `paused_reason: "novel_failure"` | Never hard-fail |
| F9 | State ledger drift | `R_RECONCILE_LEDGER` — call `sync_intake_db_row` (existing REQ-000013/14 helper) and emit `ledger_reconciled`. If reconciliation fails, pause | Defense-in-depth; no behavior change to ledger writers |

**Invariant:** for every failure mode F, the supervisor performs
`detect(F) → record_event(F) → attempt_remediation(F) → record_outcome(F)`.
If `attempt_remediation` raises or returns non-zero, the supervisor falls
back to `R_CAPTURE_AND_PAUSE`. **At no point** may the supervisor
hard-fail the request without a `paused` checkpoint that a human can
resume from.

## 7. User Stories

Stories are written from the operator (S-OP), supervisor maintainer
(S-SM), reviewer-chain author (S-RC), and submitter (S-US) perspectives.

### S-OP-1 — Operator: transient reviewer flake should not pause

- **As an** operator,
- **I want** a single reviewer that errors or returns unparseable JSON to be retried once and then excluded if non-blocking,
- **so that** a flaky reviewer does not pause the whole request.
- **Given** a request is at a `*_review` phase using the reviewer chain,
  **When** exactly one non-blocking reviewer returns `REVIEW_GATE_BAD_JSON` on the first attempt and the same on a single retry,
  **Then** the supervisor records `reviewer_error_detected` and `reviewer_excluded` events, the chain proceeds with the remaining reviewers, and the request does not enter `paused`.

### S-OP-2 — Operator: verified false-negative does not consume retry budget

- **As an** operator,
- **I want** the supervisor to confirm a claimed test failure by reading the test-result artifact before failing the gate,
- **so that** a transient false-negative (per #617) does not advance `retry_count`.
- **Given** a phase-result claims `status: "fail"` with `status_reason` referencing `test-results.json`,
  **When** the supervisor re-reads `test-results.json` and finds zero failed tests,
  **Then** the supervisor overrides the phase-result to `pass`, emits `verification_false_negative_corrected`, and `retry_count` is **not** incremented.

### S-OP-3 — Operator: timeout-with-progress extends the budget

- **As an** operator,
- **I want** a phase that times out **with** observable working-tree progress to get a one-time budget extension,
- **so that** a productive long-running phase isn't killed.
- **Given** `check_phase_timeout` triggers for phase P and `snapshot_working_tree` shows a non-empty diff vs. phase-start,
  **When** the budget extension has not yet been used for (request, phase),
  **Then** the supervisor extends the timeout by `AUTONOMOUS_DEV_BUDGET_EXTENSION_FACTOR` once, emits `phase_budget_extended`, and the phase continues; on a second timeout for the same (request, phase), the existing hard-timeout path runs.

### S-SM-1 — Maintainer: adding a failure mode is one mapping-table entry

- **As a** supervisor maintainer,
- **I want** failure modes to be added by appending one row to a single mapping table and writing the detector + remediation function,
- **so that** new modes don't require touching the main supervisor loop.
- **Given** the self-heal dispatch table in `lib/self-heal.sh` (or sourced equivalent),
  **When** I add a new `(detector_fn, event_type, remediation_fn)` row,
  **Then** no edits to `supervisor-loop.sh`'s main loop body are required, and `bats` discovers the new test file automatically.

### S-RC-1 — Reviewer-chain owner: bad reviewer JSON is observable

- **As the** reviewer-chain owner,
- **I want** every reviewer ERROR / unparseable result to emit a typed event with the failing reviewer name and the raw stdout snippet (truncated),
- **so that** I can diagnose flaky reviewers from `events.jsonl` alone.
- **Given** a reviewer returns non-JSON,
  **When** `run_review_gate_phase` synthesizes the `error` result,
  **Then** an event of type `reviewer_error_detected` is appended with `{reviewer_name, exit_code, stdout_head}` and the synthesized-result path.

### S-US-1 — Submitter: novel failures pause safely, never hard-fail

- **As a** submitter,
- **I want** an unrecognized failure to pause my request with a diagnostic bundle,
- **so that** I can resume it after the daemon learns the failure mode, rather than losing all state.
- **Given** the supervisor encounters a failure mode not present in the §6 mapping,
  **When** it triggers `R_CAPTURE_AND_PAUSE`,
  **Then** `state.json.status` becomes `paused`, a `diagnostic-bundle-<ts>.json` is written, a `novel_failure_detected` event is appended, and `request resume <id>` is accepted.

## 8. Functional Requirements

Requirements are testable; each maps to ≥1 user story and ≥1 failure mode.

### FR-DETECT — Detection layer

- **FR-DETECT-01.** The supervisor MUST source a `lib/self-heal.sh`
  module (new file) that exports the dispatch table referenced in §6.
- **FR-DETECT-02.** For each failure mode F1–F9, the supervisor MUST
  call the detector function before, during, or after the relevant phase
  transition as specified in §5.
- **FR-DETECT-03.** Each detector function MUST append exactly one event
  to `events.jsonl` per detection occurrence, with fields:
  `{type, request_id, phase, detected_at, failure_mode, evidence,
  remediation_planned}`.
- **FR-DETECT-04.** Detection MUST NOT mutate `state.json.status` or
  `state.json.current_phase`; only the remediation layer mutates those.
- **FR-DETECT-05.** Detection MUST be unconditional with respect to the
  remediation outcome (i.e., detection events are written even if
  remediation later fails).
- **FR-DETECT-06.** F1 (review-gate loop) MUST track per-request,
  per-reviewer iteration counts in
  `state.json.current_phase_metadata.self_heal.review_loop[<reviewer>]`
  and trigger when the same reviewer returned `REQUEST_CHANGES` with a
  substring-stable `.reason` ≥3 consecutive iterations.
- **FR-DETECT-07.** F5 / F6 (suspicious result) MUST compare session
  wall-clock against the configurable threshold
  `AUTONOMOUS_DEV_MIN_PHASE_DURATION_SECONDS` (default 5).

### FR-REMEDIATE — Remediation layer

- **FR-REM-01.** Each remediation function MUST be idempotent within a
  single phase (calling it twice for the same (request, phase, mode)
  triggers exactly one mutation).
- **FR-REM-02.** R_EXTEND_PHASE_BUDGET MUST extend the timeout exactly
  once per (request, phase) pair, tracked in
  `current_phase_metadata.self_heal.budget_extended_at`.
- **FR-REM-03.** R_RETRY_ONCE_THEN_EXCLUDE_IF_NON_BLOCKING MUST retry a
  failed reviewer exactly once, then check `reviewer.blocking`; if
  `blocking == true`, fall through to `fail` (existing behavior); if
  `blocking == false`, append `reviewer_excluded` and continue with the
  remaining reviewers.
- **FR-REM-04.** R_SELF_VERIFY MUST re-read the test-result artifact
  referenced in `status_reason`. If the artifact contains zero failed
  tests, the phase-result MUST be rewritten to `status: "pass"` with
  `self_verified: true`, and `retry_count` MUST NOT be incremented. If
  the artifact is missing, the supervisor MUST re-run the documented
  test command (per request `target_repo` + project conventions) before
  propagating `fail`.
- **FR-REM-05.** R_REQUEUE_AUTHOR_PHASE_ONCE MUST allow at most one
  requeue per (request, phase) pair (tracked in
  `current_phase_metadata.self_heal.requeued`), and MUST NOT increment
  `retry_count`.
- **FR-REM-06.** R_CAPTURE_AND_PAUSE MUST write
  `requests/<id>/diagnostic-bundle-<ts>.json` containing
  `{state_json, last_200_session_log_lines, last_50_events,
   detected_failure_mode, supervisor_version}` and then call the
  existing `escalate_to_paused` with `paused_reason:
  "novel_failure"`.
- **FR-REM-07.** R_RECONCILE_LEDGER MUST call the existing
  `sync_intake_db_row` helper and emit `ledger_reconciled` on success.
- **FR-REM-08.** Every remediation path MUST emit an `emit_alert` with
  a type matching the remediation policy name.

### FR-CONFIG — Configuration

- **FR-CFG-01.** All thresholds MUST be overridable via environment
  variables with the prefix `AUTONOMOUS_DEV_SELF_HEAL_*` and a default
  documented in `lib/self-heal.sh`.
- **FR-CFG-02.** Self-healing MUST be disable-able as a whole via
  `AUTONOMOUS_DEV_SELF_HEAL=0` (default `1`); when disabled, the
  supervisor behaves as it does today.

### FR-OBSERVE — Observability

- **FR-OBS-01.** At every terminal transition (`done`, `paused`,
  `failed`), the supervisor MUST append a `self_heal_summary` event
  with counts of each detection type and each remediation outcome for
  that request.
- **FR-OBS-02.** `current_phase_metadata.self_heal` MUST be an
  additive sub-object; no existing field is renamed or removed.

### FR-TEST — Test coverage

- **FR-TEST-01.** There MUST be at least one bats test per failure
  mode F1–F9 that asserts detection writes the correct event.
- **FR-TEST-02.** There MUST be at least one bats test per remediation
  policy that asserts the correct mutation and the correct event(s).
- **FR-TEST-03.** There MUST be an integration bats test that simulates
  an end-to-end "transient reviewer flake → retry → exclude →
  request_completes" flow and an end-to-end "novel failure → pause →
  resume" flow.
- **FR-TEST-04.** Unit tests for any TypeScript surface (review-gate.ts
  retry logic) MUST cover the retry-then-exclude policy.

## 9. Non-Functional Requirements

- **NFR-PERF-01.** Detection overhead MUST add ≤50 ms per phase
  transition on the median host (measured against the existing
  `advance_phase` median).
- **NFR-PERF-02.** Diagnostic bundle write (R_CAPTURE_AND_PAUSE) MUST
  complete in ≤500 ms for a request with a 200-line session log.
- **NFR-RELIABILITY-01.** A failure inside a self-heal detector or
  remediator MUST NOT crash the supervisor loop; all errors are caught
  and downgraded to `R_CAPTURE_AND_PAUSE`.
- **NFR-RELIABILITY-02.** Self-healing logic MUST be safe to run twice
  on the same input (idempotent per FR-REM-01).
- **NFR-COMPAT-01.** Existing `events.jsonl` consumers (portal,
  dashboards) MUST continue to parse the stream; new event types are
  additive.
- **NFR-COMPAT-02.** `AUTONOMOUS_DEV_SELF_HEAL=0` MUST restore the
  current behavior bit-for-bit for the failure paths it bypasses.
- **NFR-SECURITY-01.** Diagnostic bundles MUST NOT leak environment
  variable values; only an allow-list of `AUTONOMOUS_DEV_*` non-secret
  vars is captured.
- **NFR-OBSERVABILITY-01.** Every event written by self-healing MUST
  validate against an added JSON schema in `docs/schemas/events/`.
- **NFR-MAINTAINABILITY-01.** Adding a new failure mode MUST require
  edits to ≤3 files: `lib/self-heal.sh` (one new row + two functions),
  one new bats test file, and (optionally) one schema entry.

- **TC-01.** Primary implementation surface is
  `plugins/autonomous-dev/bin/supervisor-loop.sh`. A new sourced module
  `plugins/autonomous-dev/bin/lib/self-heal.sh` is permitted.
- **TC-02.** Existing functions to extend (not replace):
  `run_review_gate_phase`, `advance_phase`, `check_phase_timeout`,
  `record_soft_timeout`, `emit_soft_timeout_promotion_alert`,
  `escalate_to_paused`, `emit_alert`, `sync_intake_db_row`.
- **TC-03.** Must coexist with the soft-timeout machinery introduced
  in REQ-000051 / ADR-004; `R_EXTEND_PHASE_BUDGET` LAYERS on top of
  the soft-timeout ceiling, it does not replace it.
- **TC-04.** Must coexist with the ledger-sync helpers from REQ-000013
  / REQ-000014; `R_RECONCILE_LEDGER` calls `sync_intake_db_row`.
- **TC-05.** No new third-party runtime dependencies. Bun + jq + bats
  remain the only required tools (matching repo conventions).
- **TC-06.** All state mutations MUST go through atomic temp-file +
  rename (`mv`) as used elsewhere in `supervisor-loop.sh`.
- **TC-07.** The dispatch table MUST be data-driven (an associative
  array or a JSON literal sourced at startup) so that `bats` tests can
  exercise individual rows without booting the full loop.

## 12. Risks and Mitigations

| Risk | Severity | Mitigation |
|---|---|---|
| Self-healing masks a genuinely broken reviewer (over-exclusion) | High | Excluded reviewer events are emitted with `reviewer_name` so the chain author can audit; exclusion is per-request, not persisted |
| R_SELF_VERIFY flips a real failure to pass because the test-result artifact is stale | High | Verify artifact `mtime` is newer than the phase start; if stale, do NOT override; emit `self_verify_skipped_stale_artifact` |
| Budget extension hides a runaway phase | Medium | Extension is single-shot per (request, phase); second timeout uses existing hard path |
| Novel-failure pause floods on a new bug | Medium | Diagnostic bundles are written once per pause; deduplicate via `paused_reason` + first-occurrence event timestamp |
| Schema additions to `state.json` break older daemons during rolling restart | Low | New fields are under `current_phase_metadata.self_heal.*`; older code ignores unknown fields |
| **High-risk: this PRD changes phase-failure semantics** (existing operators may rely on current "fast fail" behavior) | High | `AUTONOMOUS_DEV_SELF_HEAL=0` kill-switch restores prior behavior; ship as opt-in in v1, default-on in v2 after one observation week |

## 13. Open Questions

1. **OQ-1 — Reviewer "blocking" attribute source.** Where does
   `reviewer.blocking` come from? Is it declared in the reviewer-chain
   YAML/JSON config, or inferred from reviewer name? *Needed for
   FR-REM-03.*
2. **OQ-2 — Test-result artifact discovery for R_SELF_VERIFY.** Is the
   artifact path always in `phase-result.status_reason`, or is there a
   convention (e.g., `requests/<id>/test-results.json`)? *Needed for
   FR-REM-04.*
3. **OQ-3 — Suspicious-fast baseline.** F6 compares against "previous
   successful run's wall-clock"; is per-phase median acceptable, or do
   we need per-(phase, request-type)? *Needed for FR-DETECT-07.*
4. **OQ-4 — Novel-failure issue filing.** Should opportunistic issue
   filing target a specific repo (autonomous-dev), or the request's
   `target_repo`? Default proposal: autonomous-dev.
5. **OQ-5 — Default for `AUTONOMOUS_DEV_SELF_HEAL`.** Ship default-on or
   default-off in v1? PRD recommendation: default-**on** with kill-switch,
   since #620 is explicit that "human is the final escalation."
6. **OQ-6 — Interaction with circuit-breaker alerts.** Does
   `emit_alert` rate-limit on the same alert type? If so, self-heal
   events may be suppressed under high churn. Needs confirmation.
7. **OQ-7 — Diagnostic bundle size cap.** What is the upper bound for
   a single bundle (e.g., ≤1 MB)? Sessions can be larger than 200 lines.

### C. Glossary

- **Author phase**: non-review phase (prd, tdd, plan, spec, code).
- **Review phase**: `*_review` phase routed through the multi-reviewer
  chain when enabled.
- **Soft timeout**: a phase-timeout where progress is detected; counted
  separately under `current_phase_metadata.soft_timeout_count`.
- **Hard timeout**: a phase-timeout where progress is NOT detected, OR a
  soft timeout that exceeds the ceiling.
- **Reviewer chain**: the ordered list of reviewers dispatched by
  `bin/review-gate.ts`.
- **Self-heal dispatch table**: the `(failure_mode, detector_fn,
  event_type, remediation_fn)` mapping defined in §6.

### D. Out-of-Band Considerations

- The CLAUDE.md / standards in this repo emphasize "least-destructive
  continue rather than hard fail." This PRD encodes that principle as
  the universal fallback (R_CAPTURE_AND_PAUSE) for any path not
  otherwise handled.
- Telemetry consumers downstream (portal, dashboards) should subscribe
  to the new `self_heal_summary` event for at-a-glance health rollups.
