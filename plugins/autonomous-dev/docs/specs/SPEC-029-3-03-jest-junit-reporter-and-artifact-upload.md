# SPEC-029-3-03: `jest-junit` Dependency + Reporter Configuration + CI Artifact Upload

- **Depends on**: SPEC-029-3-02 (the `test` job's `--reporters=jest-junit` flag references the dep this spec installs; SPEC-029-3-02 ships first per parent plan task order, so this spec's first CI run is when the reporter activates)

## Description

Install `jest-junit` as a dev dependency in `plugins/autonomous-dev/`, configure jest to use it as a reporter (alongside the default reporter), wire an `actions/upload-artifact@v4` step in `.github/workflows/ci.yml` to publish the resulting `jest-junit.xml`, and add the `reports/` output directory to gitignore. After this spec ships, the GitHub Actions UI's test-results tab renders per-suite pass/fail without operators needing to scroll the raw log; the XML is also retained as a downloadable artifact for 30 days.

This is a P2 enhancement per PRD-016 FR-1662 — the harness migration ships green without it — but is bundled with the rest of the CI gate for two reasons: (1) the reporter is wired into SPEC-029-3-02's jest invocation already, so installing the dep is the next logical step; (2) per-suite visibility in the UI dramatically lowers the cost of triaging CI failures, which is a recurring activity for this team.

## Implementation Details

### Step 1: Install `jest-junit`

From `plugins/autonomous-dev/`:

```
$ npm install --save-dev jest-junit
```

Acceptance:

- `package.json`'s `devDependencies` now contains `"jest-junit": "^16.x"` (current major as of 2026-05; verify the resolved version at install time and pin to the current major).
- `package-lock.json` updates with `jest-junit` and any transitive deps.
- The package is MIT licensed and has no native bindings (TDD-029 §8.1 requirement). Verify by:

  ```
  $ npm view jest-junit license
  MIT

  $ npm view jest-junit dependencies
  ```

  If the dependencies list shows native-binding packages (e.g., `node-gyp`-built modules), surface as a PR comment and pause; the install is rejected.

### Step 2: Configure the reporter in `jest.config.cjs`

Open `plugins/autonomous-dev/jest.config.cjs`. Add (or merge into) the `reporters` array:

```js
module.exports = {
  // ... existing config (testEnvironment, transform, etc.) unchanged
  reporters: [
    'default',
    [
      'jest-junit',
      {
        outputDirectory: './reports',
        outputName: 'jest-junit.xml',
      },
    ],
  ],
};
```

Rules:

- The `default` reporter MUST be retained as the first array entry. Without it, jest's terminal output (PASS/FAIL summary) disappears, breaking local development.
- The `jest-junit` reporter entry is a `[name, options]` tuple per jest's reporter format.
- `outputDirectory: './reports'` is relative to the jest working directory (`plugins/autonomous-dev/` per SPEC-029-3-02).
- `outputName: 'jest-junit.xml'` matches the env vars set in SPEC-029-3-02's CI step.
- If the project prefers env-var-driven configuration (and the existing `jest.config.cjs` does not have a `reporters` field), the SPEC-029-3-02 env vars `JEST_JUNIT_OUTPUT_DIR` and `JEST_JUNIT_OUTPUT_NAME` already configure the reporter at the CLI level; the `jest.config.cjs` change is then optional. Recommended: write the config explicitly so local `npx jest` runs also produce the XML, not just CI runs.

### Step 3: Add the artifact-upload step to `.github/workflows/ci.yml`

After the `Run jest (PRD-016 FR-1661)` step (added by SPEC-029-3-02) in the `test` job, append:

```yaml
- name: Upload jest JUnit report
  if: always()
  uses: actions/upload-artifact@v4
  with:
    name: jest-junit-${{ matrix.os }}-node${{ matrix.node-version }}
    path: plugins/autonomous-dev/reports/jest-junit.xml
    retention-days: 30
```

Rules:

- `if: always()` ensures the artifact uploads even if the jest step failed. A failed run's XML is the most diagnostic artifact.
- The `name` uses `matrix.os` and `matrix.node-version` so multi-leg matrices produce non-colliding artifact names. If the project's `test` job is single-leg (no matrix), simplify to `name: jest-junit`.
- `path` is relative to the workflow's working directory (typically the repo root); the path explicitly points at `plugins/autonomous-dev/reports/jest-junit.xml`.
- `retention-days: 30` matches existing CI artifact policy (PLAN-029-3 R-305 documents the rationale; reduce to 14 if storage cap is hit).
- The action MUST be `actions/upload-artifact@v4` (not v3 or earlier); v3 was deprecated by GitHub.

### Step 4: Add `reports/` to `.gitignore`

Add the line `plugins/autonomous-dev/reports/` to the repo's gitignore. Choose one location:

- `plugins/autonomous-dev/.gitignore` — preferred, scoped to the plugin.
- Repo-root `.gitignore` — acceptable if the plugin doesn't have its own gitignore.

Verify:

```
$ git check-ignore plugins/autonomous-dev/reports/jest-junit.xml
plugins/autonomous-dev/reports/jest-junit.xml
```

The path printing means the file is correctly ignored. If git does not recognize the ignore, the rule is malformed.

### Step 5: Local end-to-end verification

From `plugins/autonomous-dev/`:

```
$ rm -rf reports/   # start clean
$ npx jest --runInBand --ci --reporters=default --reporters=jest-junit
$ ls reports/jest-junit.xml
$ head -20 reports/jest-junit.xml
```

Expected:

- `reports/jest-junit.xml` exists.
- The XML's first lines look like `<?xml version="1.0" encoding="UTF-8"?>` followed by `<testsuites name="jest tests" ...>` and `<testsuite ...>` elements per suite.
- The `tests` and `failures` attributes on the `<testsuites>` root tag match the jest summary's totals (verify by sanity-eyeball, not exact match).

If the XML is malformed or absent, the reporter wiring is broken — debug before commit.

```
$ git status plugins/autonomous-dev/reports/
```

Expected: empty (the gitignore is working).

### What NOT to do

- Do NOT pin `jest-junit` to an exact patch version (`16.0.0`) when a caret range (`^16.0.0`) suffices. Caret ranges receive bug-fix updates without lockfile churn.
- Do NOT skip the `if: always()` on the upload step. A failed jest run's XML is the most useful artifact; gating uploads on success defeats the purpose.
- Do NOT commit the `reports/` directory or any of its contents. The gitignore is the safeguard; if a stray `git add reports/` happens, abort the commit and re-do.
- Do NOT remove the `default` reporter from the `reporters` array. Local jest runs need terminal output.
- Do NOT use `actions/upload-artifact@v3` or earlier. v3 is deprecated and will be removed.
- Do NOT rename the artifact's filename to something other than `jest-junit.xml`. Future tooling (and this team's muscle memory) expects the canonical name.
- Do NOT bundle this spec's changes with SPEC-029-3-02's `--ci` flag wiring. The two specs share a workflow file but have distinct contracts; per-spec commits keep review focused.
- Do NOT add a `cd plugins/autonomous-dev &&` prefix to any new step. The existing `working-directory:` directive on the jest step covers it; the artifact upload step needs full repo-root paths.

## Notes

- The reporter wiring is mechanical but the verification is not. The local end-to-end check (Step 5) is the only way to confirm the XML's structure is correct; absent the check, a reviewer cannot tell whether the wiring "works" or just "doesn't crash."
- The `if: always()` on the upload step is a frequent oversight. Without it, a failing jest step terminates the job before the upload step runs; the failed run's XML never reaches the artifacts tab — exactly when it would be most useful.
- The `matrix.os` / `matrix.node-version` interpolation in the artifact name handles multi-leg matrices. If the existing `test` job is single-leg, the simpler `name: jest-junit` works; either form is acceptable per the parent plan. The matrix form is forward-compatible.
- `jest-junit`'s widely-used status (millions of weekly downloads as of 2026 per parent plan R-306) and MIT license make the dep choice low-risk. The spec still calls for a license check at install time because the team's policy requires it.
- After this spec ships, opening any GitHub Actions workflow run page surfaces a `jest-junit-<matrix-leg>.xml` artifact at the bottom of the page. The artifact is also rendered as a per-suite test-results table in the UI's test-results tab (a recently-added GitHub feature).
- A reviewer's checklist for this PR: (1) run `npm ci` from `plugins/autonomous-dev/` and confirm jest-junit installs cleanly, (2) run `npx jest --runInBand --ci` and verify `reports/jest-junit.xml` appears, (3) read `git status` and confirm `reports/` is ignored, (4) read the CI workflow diff and confirm the upload step has `if: always()` and matches the template.
- If the artifact upload action's policy changes in a future GitHub Actions release (e.g., default retention shortens), the spec's `retention-days: 30` is the explicit pin that prevents drift.
- The XML file's stable structure (jest-junit emits standard JUnit XML) makes it a reliable input for future automation: artifact diffing across PRs, flake-history dashboards, etc. None of those are in scope here, but the deliverable enables them.
