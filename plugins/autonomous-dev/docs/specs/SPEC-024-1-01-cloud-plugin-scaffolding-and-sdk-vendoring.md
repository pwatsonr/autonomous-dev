## Description
Lay the on-disk shape and dependency graph for the four cloud-deployment sibling plugins introduced by TDD-024 §5: `autonomous-dev-deploy-gcp`, `-aws`, `-azure`, and `-k8s`. This spec is purely declarative and structural — no backend logic, helper agents, conformance tests, or READMEs (those are covered by SPEC-024-1-02 through SPEC-024-1-05). The output is four fully-loadable but no-op plugin shells that pass `claude plugin validate`, declare `extends: ["autonomous-dev"]`, advertise their `deployment_backend` metadata block, and have their cloud SDK dependencies pinned in `package.json` with committed lockfiles.

The deliberate split keeps the v2-schema manifest contract (PLAN-022-1) and dependency graph reviewable in isolation from the runtime backend code. Sibling specs may import from these scaffolds but MUST NOT modify the manifests or `package.json`.

### `tests/plugins/cloud-plugin-manifests.test.ts`

A single Jest suite that loops over the four plugin directories and validates each manifest. The test re-uses the v2 schema validator from PLAN-022-1 (`validatePluginManifestV2`).

```ts
import { validatePluginManifestV2 } from '../../src/plugins/manifest-v2-validator';
import { readFileSync } from 'node:fs';
import { join } from 'node:path';

const PLUGINS = [
  'autonomous-dev-deploy-gcp',
  'autonomous-dev-deploy-aws',
  'autonomous-dev-deploy-azure',
  'autonomous-dev-deploy-k8s',
] as const;

describe.each(PLUGINS)('plugin manifest: %s', (name) => {
  const manifest = JSON.parse(
    readFileSync(join('plugins', name, '.claude-plugin', 'plugin.json'), 'utf8'),
  );
  test('matches v2 schema', () => {
    expect(validatePluginManifestV2(manifest).valid).toBe(true);
  });
  test('extends autonomous-dev', () => {
    expect(manifest.extends).toEqual(['autonomous-dev']);
  });
  test('declares deployment_backend with credential-proxy', () => {
    expect(manifest.deployment_backend?.credential_provider).toBe('credential-proxy');
    expect(manifest.deployment_backend?.regions_supported).toBeInstanceOf(Array);
    expect(manifest.deployment_backend?.regions_supported.length).toBeGreaterThan(0);
    expect(manifest.deployment_backend?.services_supported).toBeInstanceOf(Array);
    expect(manifest.deployment_backend?.services_supported.length).toBeGreaterThan(0);
  });
});
```

### Lockfile commit policy

`npm install --package-lock-only` is run in each plugin directory, and the resulting `package-lock.json` is committed. CI does NOT run `npm ci` from the repo root for these plugins (they are sibling workspaces); instead, the install smoke (SPEC-024-1-05) confirms `npm ci` succeeds in each plugin directory.

## Notes

- This spec deliberately ships four no-op plugins. Plugin discovery will see them; the deploy phase will not yet enumerate them as backends because the `BackendRegistry` registration happens at runtime once `backend.ts` exists (delivered by SPEC-024-1-02 and SPEC-024-1-03).
- SDK major versions were chosen against the latest stable release as of TDD-024's authoring date. Dependabot will open PRs for minor/patch only; major-version bumps require a follow-up plan per PLAN-024-1's risk register.
- `regions_supported` lists are illustrative subsets covering the most common regions per cloud. They are not exhaustive — operators can use any region the SDK supports; this list drives the `deploy backends describe` output and CLI auto-completion only.
- The `credential_provider: "credential-proxy"` value is a string literal that the runtime registry (PLAN-024-2) will key off of. No other provider is supported in v1.
- Lockfiles are committed (per repo convention). When SDKs receive minor updates, Dependabot opens a PR that updates the lockfile and the install smoke (SPEC-024-1-05) re-runs.
- The `tests/plugins/cloud-plugin-manifests.test.ts` file lives at the repo root `tests/` directory, NOT inside any plugin, so it can be discovered by the existing root Jest config without per-plugin test infra.
