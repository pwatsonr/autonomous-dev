| B-14 | med | autopilot (universal-dev) | Autopilot loop kicked off via `autopilot-loop.sh` on a freshly-initialised state (`status: running, current_phase: prd, iteration: 0`) produced **zero artifacts and zero cost** and its `.claude/` dir was gone afterwards. Found 2026-05-13 in the Path-C build-out test. The `claude -p "/universal-dev:autopilot-resume"` session likely failed/exited because `autopilot-resume` expects to *continue* an in-progress run, not start one from `iteration: 0`. Fresh runs should go through `/universal-dev:autopilot "<direction>"` (the entry point) which initialises state THEN drives iteration 1, not through `autopilot-resume` on a pre-init'd file. | Investigate the failure mode (capture the `claude -p` session's stderr; maybe `autopilot-loop.sh` needs to detect an uninitialised state and bootstrap via the entry-point skill instead). Or: in future test scripts, kick off with `claude -p "/universal-dev:autopilot \"<direction>\" --max-iterations N"` directly rather than autopilot-loop.sh against a hand-written state. |
