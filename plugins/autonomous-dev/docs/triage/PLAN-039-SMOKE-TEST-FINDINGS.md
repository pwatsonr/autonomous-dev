| **B-09** | **HIGH** | daemon | **`resolve_agent` and `LEGACY_PHASES` are out of sync — a legacy-sequence request would die mid-pipeline.** `LEGACY_PHASES` (in `bin/lib/phase-legacy.sh`) = `intake prd prd_review tdd tdd_review plan plan_review spec spec_review code code_review test test_review validate`. `resolve_agent` (in `bin/supervisor-loop.sh`, added in PR-2) handles `prd … spec_review code code_review security_review deploy` — it has **no mapping for `test`, `test_review`, or `validate`**, and `security_review`/`deploy` aren't in the legacy sequence. So a request following the legacy sequence (which is *every* request, since `state_json_writer` writes `phase_overrides: []` → legacy fallback — see B-04) reaches `test` → `resolve_agent` returns empty → `dispatch_phase_session` returns 3 → main-loop error branch → retries → eventually `failed`. The pipeline literally cannot complete. | Reconcile the two. Either: (a) update `resolve_agent` to cover `test → test-executor`, `test_review → quality-reviewer`, `validate → quality-reviewer` (matching LEGACY_PHASES) and drop/keep `security_review`/`deploy` as appropriate; or (b) the canonical sequence should come from `intake/types/phase-override.ts` (the TS matrix is "the canonical reference" per phase-legacy.sh's header) and `state_json_writer` should populate `phase_overrides[]` from it instead of leaving `[]` — then both `next_phase_for_state` and `resolve_agent` work off the same explicit sequence. TDD-038 §7.1's transition table also uses `security_review`/`deploy` — it's likewise out of sync with LEGACY_PHASES. Pick one source of truth. |
