# Manual Verification — PLAN-011-2 Claude App slash commands

This checklist exercises the slash-command UX end-to-end inside Claude
Code. Automated tests cover the bridge in isolation
(`tests/unit/claude_command_bridge.test.ts`) and the static
discoverability of the .md files
(`tests/integration/claude_commands.test.ts`); this list covers
plugin reload, autocomplete, and human-readable error formatting that
do not lend themselves to automation.

## Prerequisites

- The plugin is installed in your `~/.claude/plugins/` (or symlinked from
  this repo).
- `npm install && npm run build` has been run inside `plugins/autonomous-dev/`
  so `dist/intake/adapters/claude_command_bridge.js` exists.
- A Claude Code session is open at any directory.

## 1. Plugin reload picks up the new commands

- [ ] Open Claude Code in a fresh session.
- [ ] Run `/plugins reload autonomous-dev` (or restart Claude Code).
- [ ] No error banner appears in the session log.

## 2. Autocomplete shows the 10 commands

In the message input box, type `/` then `auto` and confirm the suggestion
list contains exactly the 10 entries below (in any order):

- [ ] `/autonomous-dev-submit`
- [ ] `/autonomous-dev-status`
- [ ] `/autonomous-dev-list`
- [ ] `/autonomous-dev-cancel`
- [ ] `/autonomous-dev-pause`
- [ ] `/autonomous-dev-resume`
- [ ] `/autonomous-dev-priority`
- [ ] `/autonomous-dev-logs`
- [ ] `/autonomous-dev-feedback`
- [ ] `/autonomous-dev-kill`

No other `/autonomous-dev-*` entries should appear.

## 3. Per-command help text is sensible

For each of the 10 commands, hovering or typing the bare command should
show the description from its frontmatter. Spot-check three:

- [ ] `/autonomous-dev-submit` → "Submit a new autonomous-dev request (returns REQ-NNNNNN identifier)"
- [ ] `/autonomous-dev-list` → "List recent autonomous-dev requests (default: active only)"
- [ ] `/autonomous-dev-priority` → "Change priority of an autonomous-dev request (high|normal|low)"

## 4. Bridge produces JSON when invoked directly

From inside `plugins/autonomous-dev/`:

- [ ] `node dist/intake/adapters/claude_command_bridge.js status REQ-000001`
      prints exactly one line of JSON to stdout. The line parses as
      `{ "ok": true, "data": ... }` or `{ "ok": false, "errorCode": ... }`.
- [ ] No output appears on stderr.
- [ ] The exit code matches the table in `docs/bridge_contract.md`
      (success → 0, validation error → 1, system error → 2).

## 5. Exit codes propagate through the bash proxy

From inside `plugins/autonomous-dev/`:

- [ ] `bash commands/_shared/bridge_proxy.sh; bridge_proxy_invoke status REQ-000001` returns the
      same exit code as the direct invocation in step 4.
- [ ] `bash -c 'source commands/_shared/bridge_proxy.sh && bridge_proxy_invoke frobnicate'`
      exits 1 (UNKNOWN_SUBCOMMAND).
- [ ] `bash -c 'source commands/_shared/bridge_proxy.sh && bridge_proxy_invoke status'`
      exits 1 (INVALID_ARGUMENT — missing request_id).

## 6. Error UX inside Claude Code

Run a known-bad invocation and confirm the error reaches the user as
formatted text (not raw JSON):

- [ ] `/autonomous-dev-status` (no arg) → user-visible error mentions
      `request_id` and recommends `REQ-NNNNNN`.
- [ ] `/autonomous-dev-priority REQ-000001 urgent` → user-visible error
      lists allowed values (`high, normal, low`).

## 7. Module/bridge resilience

- [ ] Temporarily move `dist/intake/adapters/claude_command_bridge.js`
      aside, then run `/autonomous-dev-status REQ-000001`. Operator-visible
      error mentions `npm install && npm run build` resolution. Restore
      the file.
- [ ] Temporarily set `AUTONOMOUS_DEV_EXPECTED_VERSION=99.99.99` in the
      shell that launched Claude Code, restart, and run any command.
      Error envelope contains `VERSION_MISMATCH`. Unset the env var when
      done.

## 8. No coloured output unless explicitly enabled

- [ ] Confirm `process.env.NO_COLOR=1` is honoured by the formatter
      (the bridge itself emits plain JSON, but the bash proxy's
      diagnostic output should respect NO_COLOR).
